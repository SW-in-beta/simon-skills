# Spec: ML Recommender (사용자 행동 기반 상품 추천 시스템)

**Status**: Approved
**Created**: 2026-03-07
**Last Updated**: 2026-03-07 (Phase 1-B INVEST 보강)

## 0. Vision Summary

**무엇을**: 사용자 행동 데이터(조회, 클릭, 구매)를 기반으로 Collaborative Filtering 추천 모델을 학습하고, REST API로 개인화 상품 추천을 제공하는 시스템
**왜**: 사용자 경험 개선 및 전환율 향상을 위한 데이터 기반 상품 추천 자동화
**누구를 위해**: (1) 추천을 받는 최종 사용자, (2) 추천 성능을 모니터링하는 관리자
**규모 가정**: 일일 활성 사용자 수만 명 수준, 상품 수 수천~수만 개, 이벤트 초당 1000건 이상 처리

## 1. User Stories & Scenarios

각 스토리는 우선순위(P1, P2, P3)가 있으며, 독립적으로 테스트/배포 가능해야 한다.
P1만으로 MVP가 가능하다: 상품 등록 -> 행동 수집 -> 모델 학습 -> 추천 제공 -> 성능 모니터링.

### US-1: 사용자 행동 데이터 수집 (P1)

사용자의 상품 조회, 클릭, 구매 등의 행동 데이터를 수집하여 추천 모델의 입력 데이터로 활용한다. 추천 시스템의 근간이 되는 데이터 파이프라인이므로 최우선으로 구현한다.

**Why this priority**: 행동 데이터 없이는 collaborative filtering 모델 학습이 불가능하므로 전체 시스템의 전제 조건
**Independent Test**: 행동 이벤트 API 호출 후 데이터가 정상적으로 저장/조회되는지 확인
**INVEST 검증**:
- I: 상품 카탈로그(US-3)와 독립적으로 구현 가능 (사용자 ID, 상품 ID는 외부 식별자)
- N: 이벤트 수집 방식(동기/비동기)은 열려 있음
- V: 추천 모델 학습의 전제 조건으로 명확한 가치 제공
- E: S-M 규모 (API 엔드포인트 + 이벤트 저장 로직)
- S: 파일 5-10개 수준
- T: Given/When/Then 시나리오로 검증 가능

**Acceptance Scenarios**:
1. **Given** 사용자가 상품을 조회하면, **When** 클라이언트가 행동 이벤트 API를 호출하면, **Then** 사용자 ID, 상품 ID, 행동 유형(view/click/purchase), 타임스탬프가 저장된다
2. **Given** 유효하지 않은 사용자 ID 또는 상품 ID로 이벤트를 전송하면, **When** API가 요청을 수신하면, **Then** 400 에러와 구체적 에러 메시지를 반환한다
3. **Given** 동일 사용자가 같은 상품에 대해 짧은 시간(5초) 내 중복 이벤트를 보내면, **When** API가 요청을 수신하면, **Then** 중복 이벤트는 1건으로 집계된다

---

### US-2: 상품 추천 API (P1)

특정 사용자에 대해 collaborative filtering 모델 기반 상품 추천 목록을 REST API로 제공한다. 시스템의 핵심 가치를 전달하는 기능이다.

**Why this priority**: 추천 결과 제공이 시스템의 핵심 목적이므로 MVP 필수 기능
**Independent Test**: 행동 데이터가 존재하는 사용자에 대해 추천 API 호출 시 정렬된 상품 목록 반환 확인
**INVEST 검증**:
- I: 모델 서빙 인터페이스에만 의존, 학습 파이프라인과 분리 가능
- N: 추천 결과 제공이 목적이며 API 형식은 열려 있음
- V: 개인화 추천이라는 핵심 비즈니스 가치 제공
- E: M 규모 (API + 모델 서빙 + 폴백 로직)
- S: 파일 8-12개 수준
- T: Given/When/Then 시나리오로 검증 가능

**Acceptance Scenarios**:
1. **Given** 충분한 행동 데이터가 있는 사용자가 있을 때, **When** GET /api/v1/recommendations/{user_id} 를 호출하면, **Then** 추천 점수 순으로 정렬된 상품 ID 목록(기본 10개)과 각 추천 점수를 반환한다
2. **Given** 행동 데이터가 없는 신규 사용자일 때, **When** 추천 API를 호출하면, **Then** 인기 상품 기반 폴백 추천 목록을 반환한다 (cold start 대응)
3. **Given** 추천 API 호출 시, **When** count 파라미터로 반환 개수를 지정하면, **Then** 지정한 개수만큼 추천 결과를 반환한다 (최소 1, 최대 50)
4. **Given** 존재하지 않는 사용자 ID로 호출하면, **When** API가 요청을 수신하면, **Then** 404 에러를 반환한다

---

### US-3: 상품 카탈로그 관리 API (P1)

추천 대상이 되는 상품 정보를 등록/조회/수정/삭제할 수 있는 API를 제공한다.

**Why this priority**: 추천 대상 상품이 없으면 추천 자체가 불가능하므로 MVP 필수 기능
**Independent Test**: 상품 CRUD 후 목록 조회로 반영 여부 확인
**INVEST 검증**:
- I: 다른 스토리와 완전히 독립적으로 구현/테스트 가능
- N: CRUD 기능 목적이 명확, 구현 방식은 열려 있음
- V: 추천 대상 상품 관리의 기본 가치 제공
- E: S 규모 (CRUD API)
- S: 파일 5-8개 수준
- T: Given/When/Then 시나리오로 검증 가능

**Acceptance Scenarios**:
1. **Given** 관리자가 상품 정보(이름, 카테고리, 설명, 가격)를 제출하면, **When** POST /api/v1/products 를 호출하면, **Then** 상품이 등록되고 고유 ID가 반환된다
2. **Given** 등록된 상품이 있을 때, **When** GET /api/v1/products 를 호출하면, **Then** 페이지네이션된 상품 목록을 반환한다
3. **Given** 등록된 상품이 있을 때, **When** PUT /api/v1/products/{id} 로 수정하면, **Then** 상품 정보가 업데이트된다
4. **Given** 이미 추천에 사용 중인 상품을 삭제하면, **When** DELETE 요청을 보내면, **Then** 상품이 비활성화(soft delete)되고 이후 추천에서 제외된다

---

### US-4: Collaborative Filtering 모델 학습 (P1)

수집된 사용자 행동 데이터를 기반으로 collaborative filtering 추천 모델을 학습한다.

**Why this priority**: 모델 학습 없이는 추천 결과 생성이 불가능하므로 MVP 필수 기능
**Independent Test**: 샘플 행동 데이터로 모델 학습 후 추천 점수 생성 여부 확인
**INVEST 검증**:
- I: 행동 데이터 인터페이스에만 의존, 학습 로직은 독립적
- N: CF 알고리즘 선택(ALS, SVD 등)은 열려 있음
- V: 개인화 추천 생성이라는 핵심 가치 제공
- E: M 규모 (학습 파이프라인 + 모델 저장 + 평가)
- S: 파일 8-12개 수준
- T: 학습 완료, 모델 저장, 평가 지표 생성으로 검증 가능

**Acceptance Scenarios**:
1. **Given** 최소 100건 이상의 행동 데이터가 존재할 때, **When** 모델 학습을 실행하면, **Then** 학습이 완료되고 모델 아티팩트가 저장된다
2. **Given** 학습이 완료되면, **When** 모델 메타데이터를 조회하면, **Then** 학습 시간, 데이터 건수, 모델 버전, 평가 지표(RMSE, precision@k)가 포함된다
3. **Given** 학습 중 오류가 발생하면, **When** 에러가 감지되면, **Then** 기존 모델을 유지하고 에러 로그를 기록한다

---

### US-5: 추천 성능 지표 대시보드 (P1)

관리자가 웹 대시보드에서 추천 시스템의 성능 지표를 실시간으로 모니터링할 수 있다.

**Why this priority**: 사용자가 명시적으로 관리자 대시보드 모니터링을 요구했으므로 MVP 필수 기능
**Independent Test**: 대시보드 접속 후 핵심 지표 차트가 렌더링되는지 확인
**INVEST 검증**:
- I: 백엔드 지표 API에만 의존, 프론트엔드 독립 구현 가능
- N: 시각화 방식(차트 유형 등)은 열려 있음
- V: 추천 성능 가시성이라는 명확한 가치 제공
- E: M 규모 (대시보드 UI + 차트 렌더링 + 지표 API)
- S: 파일 10-15개 수준
- T: 대시보드 렌더링, 지표 표시, 기간 필터로 검증 가능

**Acceptance Scenarios**:
1. **Given** 관리자가 대시보드에 접속하면, **When** 메인 페이지가 로드되면, **Then** CTR(클릭률), 전환율, 추천 커버리지, 추천 API 응답 시간 지표를 표시한다
2. **Given** 대시보드에서 기간 필터를 변경하면, **When** 날짜 범위를 선택하면, **Then** 해당 기간의 지표가 차트로 업데이트된다
3. **Given** 추천 모델이 여러 버전 존재할 때, **When** 모델 버전별 성능 비교를 선택하면, **Then** 버전 간 주요 지표를 비교 차트로 표시한다

---

### US-6: 관리자 인증 (P1)

대시보드 접근을 인증된 관리자로 제한한다.

**Why this priority**: 대시보드 보안은 MVP에서도 필수, 비인증 접근 차단 필요
**Independent Test**: 인증 없이 대시보드 접근 시 로그인 페이지로 리다이렉트 확인
**INVEST 검증**:
- I: 대시보드 UI와 독립적으로 인증 레이어 구현 가능
- N: 인증 방식(JWT, session 등)은 열려 있음
- V: 보안이라는 명확한 가치 제공
- E: S 규모 (로그인 API + 토큰 관리)
- S: 파일 5-8개 수준
- T: 인증/비인증 접근으로 명확히 검증 가능

**Acceptance Scenarios**:
1. **Given** 인증되지 않은 사용자가 대시보드 URL에 접근하면, **When** 페이지 로드를 시도하면, **Then** 로그인 페이지로 리다이렉트된다
2. **Given** 유효한 관리자 계정으로 로그인하면, **When** 이메일과 비밀번호를 제출하면, **Then** JWT 토큰이 발급되고 대시보드에 접근할 수 있다
3. **Given** 잘못된 자격 증명으로 로그인을 시도하면, **When** 인증 실패 시, **Then** 401 에러와 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지를 반환한다

---

### US-7: 모델 학습 스케줄링 및 수동 트리거 (P2)

관리자가 대시보드에서 모델 재학습을 수동으로 트리거하거나, 자동 학습 스케줄을 설정할 수 있다.

**Why this priority**: 자동 재학습은 운영 편의성을 높이지만 MVP 없이도 수동 학습으로 동작 가능
**Independent Test**: 대시보드에서 재학습 버튼 클릭 후 학습 시작/완료 상태 확인
**INVEST 검증**:
- I: 모델 학습(US-4)의 트리거 인터페이스만 사용, 학습 로직과 분리
- N: 스케줄링 방식(cron, queue 등)은 열려 있음
- V: 운영 편의성이라는 명확한 가치
- E: M 규모 (스케줄러 + UI + 상태 관리)
- S: 파일 6-10개 수준
- T: 수동 트리거, 스케줄 설정, 중복 방지로 검증 가능

**Acceptance Scenarios**:
1. **Given** 관리자가 대시보드에서 "모델 재학습" 버튼을 클릭하면, **When** 학습 요청이 전송되면, **Then** 학습 작업이 시작되고 상태(대기중/학습중/완료/실패)가 실시간 표시된다
2. **Given** 관리자가 자동 학습 스케줄을 "매일 03:00"으로 설정하면, **When** 설정을 저장하면, **Then** 해당 시간에 자동으로 모델 학습이 트리거된다
3. **Given** 학습이 이미 진행 중일 때, **When** 재학습 버튼을 클릭하면, **Then** "이미 학습이 진행 중입니다"라는 메시지를 표시한다

---

### US-8: A/B 테스트 관리 (P2)

관리자가 여러 추천 모델 버전 간 A/B 테스트를 설정하고 결과를 비교할 수 있다.

**Why this priority**: 모델 개선을 위한 실험 프레임워크는 중요하지만 초기 운영에는 단일 모델로 충분
**Independent Test**: A/B 테스트 생성 후 트래픽 분배가 설정대로 동작하는지 확인
**INVEST 검증**:
- I: 추천 API(US-2)의 모델 선택 레이어로 독립 구현 가능
- N: 트래픽 분배 방식은 열려 있음
- V: 데이터 기반 모델 개선이라는 가치 제공
- E: M-L 규모 (실험 관리 + 트래픽 분배 + 결과 분석)
- S: 파일 10-15개 수준
- T: 트래픽 비율, 결과 비교로 검증 가능

**Acceptance Scenarios**:
1. **Given** 관리자가 A/B 테스트를 생성하면, **When** 두 모델 버전과 트래픽 비율(예: 80:20)을 설정하면, **Then** 추천 API가 설정된 비율에 따라 다른 모델의 결과를 반환한다
2. **Given** A/B 테스트가 진행 중일 때, **When** 대시보드에서 테스트 결과를 조회하면, **Then** 각 모델 버전의 CTR, 전환율을 비교 표시한다
3. **Given** A/B 테스트 결과가 통계적으로 유의미하면, **When** 관리자가 승자 모델을 선택하면, **Then** 해당 모델이 프로덕션 모델로 승격된다

---

### US-9: 추천 제외/부스트 규칙 관리 (P2)

관리자가 특정 상품을 추천에서 제외하거나 우선순위를 높이는 비즈니스 규칙을 설정할 수 있다.

**Why this priority**: 비즈니스 로직 오버라이드는 운영 유연성을 위해 필요하지만 초기 버전에서는 순수 모델 결과로 충분
**Independent Test**: 제외 규칙 설정 후 해당 상품이 추천 결과에서 빠지는지 확인
**INVEST 검증**:
- I: 추천 API(US-2)의 후처리 레이어로 독립 구현 가능
- N: 규칙 표현 방식은 열려 있음
- V: 비즈니스 유연성이라는 명확한 가치
- E: S-M 규모 (규칙 CRUD + 필터링 로직)
- S: 파일 6-10개 수준
- T: 제외/부스트 적용 여부로 명확히 검증 가능

**Acceptance Scenarios**:
1. **Given** 관리자가 특정 상품을 추천 제외 목록에 추가하면, **When** 해당 사용자에게 추천 API를 호출하면, **Then** 해당 상품은 추천 결과에 포함되지 않는다
2. **Given** 관리자가 특정 상품에 부스트 점수를 설정하면, **When** 추천 API를 호출하면, **Then** 해당 상품의 추천 점수에 부스트가 반영된다
3. **Given** 제외/부스트 규칙에 유효 기간이 설정되어 있을 때, **When** 기간이 만료되면, **Then** 해당 규칙이 자동으로 비활성화된다

---

### US-10: 추천 결과 설명 (P3)

사용자에게 "왜 이 상품이 추천되었는지" 설명을 제공한다.

**Why this priority**: 추천 투명성은 사용자 신뢰를 높이지만 핵심 기능은 아님
**Independent Test**: 추천 API에 explain=true 파라미터 추가 후 설명 필드 포함 여부 확인
**INVEST 검증**:
- I: 추천 API(US-2)에 옵셔널 기능으로 독립 추가 가능
- N: 설명 생성 방식은 열려 있음
- V: 추천 투명성이라는 가치 제공
- E: S 규모 (설명 생성 로직 추가)
- S: 파일 3-5개 수준
- T: explain 파라미터 유무로 명확히 검증 가능

**Acceptance Scenarios**:
1. **Given** 추천 API 호출 시 explain=true 파라미터를 포함하면, **When** 응답을 받으면, **Then** 각 추천 상품에 대해 "이 상품을 좋아한 유사한 사용자들이 구매했습니다" 등의 설명이 포함된다
2. **Given** cold start 폴백 추천인 경우, **When** explain=true로 호출하면, **Then** "인기 상품 기반 추천입니다"라는 설명이 반환된다

---

### US-11: 실시간 이벤트 스트림 대시보드 (P3)

관리자가 대시보드에서 실시간으로 유입되는 사용자 행동 이벤트 스트림을 확인할 수 있다.

**Why this priority**: 운영 가시성을 높이지만 핵심 추천 기능에는 영향 없음
**Independent Test**: 대시보드에서 실시간 이벤트 피드가 자동 업데이트되는지 확인
**INVEST 검증**:
- I: 이벤트 수집(US-1)과 대시보드(US-5)에 독립적으로 추가 가능
- N: 스트림 표시 방식은 열려 있음
- V: 운영 가시성이라는 가치 제공
- E: S-M 규모 (WebSocket/SSE + UI 컴포넌트)
- S: 파일 5-8개 수준
- T: 이벤트 발생 후 대시보드 업데이트로 검증 가능

**Acceptance Scenarios**:
1. **Given** 관리자가 실시간 이벤트 탭에 접속하면, **When** 새로운 행동 이벤트가 발생하면, **Then** 3초 이내에 이벤트가 목록에 추가된다
2. **Given** 이벤트 스트림에서 이벤트 유형(view/click/purchase) 필터를 적용하면, **When** 필터를 선택하면, **Then** 해당 유형의 이벤트만 표시된다

## 2. Functional Requirements

- **FR-001**: 시스템은 반드시 사용자 행동 이벤트(view, click, purchase)를 REST API로 수집해야 한다
- **FR-002**: 시스템은 반드시 Collaborative Filtering 알고리즘을 기반으로 개인화된 상품 추천을 생성해야 한다
- **FR-003**: 시스템은 반드시 REST API를 통해 추천 결과를 JSON 형식으로 제공해야 한다
- **FR-004**: 시스템은 반드시 행동 데이터가 없는 사용자(cold start)에 대해 인기 상품 기반 폴백 추천을 제공해야 한다
- **FR-005**: 관리자 대시보드는 반드시 CTR, 전환율, 추천 커버리지, API 응답 시간 등 핵심 성능 지표를 시각화해야 한다
- **FR-006**: 대시보드 접근은 반드시 인증된 관리자만 허용해야 한다
- **FR-007**: 모델 학습은 반드시 기존 서비스 중단 없이 수행되어야 한다 (학습 중에도 기존 모델로 추천 제공)
- **FR-008**: 추천 API는 반드시 200ms 이내에 응답해야 한다
- **FR-009**: 시스템은 반드시 모델 버전을 관리하고 롤백할 수 있어야 한다

## 3. Edge Cases

- **EC-001**: 완전히 새로운 시스템(행동 데이터 0건)에서 추천 API 호출 시 -- 전체 인기 상품 기반 폴백 제공
- **EC-002**: 상품이 삭제(비활성화)된 직후 해당 상품이 추천 목록에 포함되어 있는 경우 -- 추천 결과 생성 시 비활성 상품 필터링
- **EC-003**: 대량의 행동 이벤트가 동시에 유입되는 경우(스파이크 트래픽) -- 이벤트 수집 API의 처리량 제한(rate limiting) 및 비동기 처리
- **EC-004**: 모델 학습 중 서버 장애 발생 -- 기존 모델 유지, 학습 재시도 메커니즘
- **EC-005**: 사용자가 조회한 상품만 있고 구매 이력이 전혀 없는 경우 -- 조회 데이터 기반 약한 신호로 추천 생성
- **EC-006**: 추천 가능 상품이 요청 개수보다 적은 경우 -- 가용한 만큼만 반환하고 응답에 총 가용 수 포함

## 4. Key Entities

- **User (사용자)**: 추천을 받는 대상. 고유 식별자, 생성일 등 기본 정보
- **Product (상품)**: 추천 대상 상품. 이름, 카테고리, 설명, 가격, 활성 상태
- **UserEvent (사용자 행동 이벤트)**: 사용자와 상품 간의 상호작용 기록. 행동 유형(view/click/purchase), 타임스탬프
- **RecommendationModel (추천 모델)**: 학습된 모델의 메타데이터. 버전, 학습 시간, 데이터 건수, 평가 지표, 상태(active/inactive)
- **Recommendation (추천 결과)**: 특정 사용자에 대한 추천 상품 목록. 추천 점수, 추천 이유
- **Admin (관리자)**: 대시보드 접근 권한을 가진 사용자. 이메일, 비밀번호(해시), 역할
- **ABTest (A/B 테스트)**: 모델 간 비교 실험. 모델 버전들, 트래픽 비율, 시작/종료일, 상태
- **BusinessRule (비즈니스 규칙)**: 추천 제외/부스트 규칙. 대상 상품, 규칙 유형, 유효 기간

**엔티티 관계**:
- User 1:N UserEvent (사용자가 여러 행동 이벤트를 생성)
- Product 1:N UserEvent (상품에 대해 여러 행동 이벤트 발생)
- RecommendationModel 1:N Recommendation (하나의 모델이 여러 추천 결과 생성)
- User 1:N Recommendation (사용자가 여러 추천을 받음)
- Product 1:N Recommendation (상품이 여러 추천에 포함)
- ABTest N:M RecommendationModel (A/B 테스트가 여러 모델을 비교)
- BusinessRule N:1 Product (규칙이 특정 상품에 적용)

## 5. Success Criteria

- **SC-001**: 추천 API의 p95 응답 시간이 200ms 이내여야 한다
- **SC-002**: Cold start 사용자에게도 추천 결과를 제공해야 한다 (폴백 포함 100% 응답률)
- **SC-003**: 관리자 대시보드에서 최소 4개 핵심 지표(CTR, 전환율, 커버리지, 응답 시간)를 확인할 수 있어야 한다
- **SC-004**: 모델 재학습 시 서비스 중단이 0초여야 한다 (기존 모델로 계속 서빙)
- **SC-005**: 사용자 행동 이벤트 수집 API가 초당 1000건 이상의 이벤트를 처리할 수 있어야 한다
- **SC-006**: 추천 커버리지(전체 상품 중 추천에 포함되는 상품 비율)가 70% 이상이어야 한다
