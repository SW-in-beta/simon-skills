# 사용자 스토리 (User Stories)

## 에픽 구성

| 에픽 ID | 에픽 이름 | 설명 |
|---------|-----------|------|
| E1 | 데이터 수집 및 처리 | 사용자 행동 데이터 수집/전처리 파이프라인 |
| E2 | 추천 모델 | Collaborative Filtering 모델 개발/학습 |

## E1: 데이터 수집 및 처리

### US-1.1: 사용자 행동 이벤트 수집

> **역할**: 백엔드 개발자로서
> **원함**: 사용자의 상품 관련 행동(조회, 클릭, 장바구니, 구매, 평점)을 실시간으로 수집할 수 있기를
> **이유**: 추천 모델 학습에 필요한 상호작용 데이터를 축적하기 위해

**수락 기준**:
- [ ] POST /api/v1/events 엔드포인트로 행동 이벤트를 수신할 수 있다
- [ ] 이벤트에는 user_id, item_id, event_type, timestamp가 필수로 포함된다
- [ ] event_type은 view, click, add_to_cart, purchase, rating 중 하나여야 한다
- [ ] 유효하지 않은 이벤트는 400 에러와 함께 상세 사유를 반환한다
- [ ] 수신된 이벤트는 메시지 큐(Kafka)에 발행된다
- [ ] 이벤트 수신 응답 시간은 p99 < 50ms를 충족한다

**스토리 포인트**: 5

---

### US-1.2: 이벤트 데이터 적재

> **역할**: 데이터 엔지니어로서
> **원함**: 수집된 행동 이벤트를 데이터 레이크에 안정적으로 적재할 수 있기를
> **이유**: 모델 학습용 데이터셋을 구성하기 위해

**수락 기준**:
- [ ] Kafka에서 소비한 이벤트를 S3/GCS에 Parquet 형식으로 저장한다
- [ ] 시간 단위로 파티셔닝된다 (year/month/day/hour)
- [ ] 중복 이벤트는 제거된다 (exactly-once semantics)
- [ ] 데이터 적재 지연은 최대 5분 이내이다
- [ ] 적재 실패 시 Dead Letter Queue에 보관되어 재처리 가능하다

**스토리 포인트**: 5

---

### US-1.3: 데이터 전처리 파이프라인

> **역할**: ML 엔지니어로서
> **원함**: 원본 이벤트 데이터를 모델 학습에 사용할 수 있는 형태로 전처리할 수 있기를
> **이유**: 깨끗한 User-Item 상호작용 매트릭스를 생성하기 위해

**수락 기준**:
- [ ] 봇 트래픽이 필터링된다 (User-Agent 기반 + 행동 패턴 기반)
- [ ] 이상치(1분 내 100건 이상 이벤트 발생 사용자 등)가 제거된다
- [ ] 행동 이벤트가 가중치 기반 암묵적 피드백으로 변환된다
- [ ] User-Item 희소 행렬이 생성되어 피처 스토어에 저장된다
- [ ] 전처리 파이프라인은 Airflow DAG로 일 1회 실행된다
- [ ] 전처리 완료 후 데이터 품질 리포트가 생성된다

**스토리 포인트**: 8

---

### US-1.4: Cold Start 사용자/상품 식별

> **역할**: ML 엔지니어로서
> **원함**: 상호작용이 부족한 사용자와 상품을 자동으로 식별할 수 있기를
> **이유**: Cold Start 상황에서 적절한 폴백 전략을 적용하기 위해

**수락 기준**:
- [ ] 상호작용 5건 미만인 사용자를 "cold" 사용자로 분류한다
- [ ] 상호작용 10건 미만인 상품을 "cold" 상품으로 분류한다
- [ ] Cold/Warm/Hot 분류 결과가 피처 스토어에 저장된다
- [ ] 분류 기준(임계값)은 설정 파일로 조정 가능하다

**스토리 포인트**: 3

## E2: 추천 모델

### US-2.1: ALS 모델 학습 파이프라인

> **역할**: ML 엔지니어로서
> **원함**: Implicit ALS 알고리즘으로 추천 모델을 정기적으로 학습할 수 있기를
> **이유**: 최신 사용자 행동을 반영한 추천을 제공하기 위해

**수락 기준**:
- [ ] implicit 라이브러리의 ALS 알고리즘으로 모델을 학습한다
- [ ] 하이퍼파라미터(factors, regularization, iterations, alpha)는 설정 파일로 관리된다
- [ ] 학습은 일 1회 Airflow DAG로 자동 실행된다
- [ ] 학습된 모델은 MLflow에 버전 관리되어 등록된다
- [ ] 학습 소요 시간, 데이터 크기, 하이퍼파라미터가 로그에 기록된다
- [ ] 학습 실패 시 Slack 알림이 발송된다

**스토리 포인트**: 8

---

### US-2.2: 오프라인 모델 평가

> **역할**: ML 엔지니어로서
> **원함**: 학습된 모델의 추천 품질을 오프라인 지표로 평가할 수 있기를
> **이유**: 프로덕션 배포 전 모델 성능을 검증하기 위해

**수락 기준**:
- [ ] Train/Test 분할은 시간 기반으로 수행된다 (마지막 N일을 테스트셋)
- [ ] Precision@10, Recall@10, NDCG@10, MAP, Hit Rate, Coverage를 계산한다
- [ ] 평가 결과가 MLflow 메트릭으로 기록된다
- [ ] 이전 모델 대비 성능 비교 리포트가 자동 생성된다
- [ ] 성능이 기존 모델 대비 하락하면 자동 배포를 차단한다

**스토리 포인트**: 5

---

### US-2.3: 추천 결과 사전 계산

> **역할**: ML 엔지니어로서
> **원함**: 모든 활성 사용자에 대한 추천 결과를 사전에 계산하여 캐시에 저장할 수 있기를
> **이유**: API 응답 시간을 최소화하기 위해

**수락 기준**:
- [ ] 활성 사용자(최근 30일 내 1건 이상 활동) 전원에 대해 Top-50 추천을 계산한다
- [ ] 결과는 Redis에 user_id를 키로 저장된다
- [ ] 캐시 TTL은 설정 가능하다 (기본값: 6시간)
- [ ] 배치 계산은 모델 학습 완료 후 자동으로 트리거된다
- [ ] 계산 완료까지 소요 시간은 100만 사용자 기준 30분 이내이다

**스토리 포인트**: 5

---

### US-2.4: 유사 상품 계산

> **역할**: ML 엔지니어로서
> **원함**: 각 상품에 대한 유사 상품 목록을 계산할 수 있기를
> **이유**: 상품 상세 페이지에서 "비슷한 상품" 추천을 제공하기 위해

**수락 기준**:
- [ ] Item Factor 벡터 간 코사인 유사도로 유사 상품을 계산한다
- [ ] 상품당 Top-20 유사 상품과 유사도 점수를 저장한다
- [ ] 결과는 Redis에 item_id를 키로 저장된다
- [ ] 모델 학습 완료 후 자동으로 재계산된다

**스토리 포인트**: 3

---

### US-2.5: Cold Start 폴백 추천

> **역할**: 시스템으로서
> **원함**: Cold Start 사용자에게 인기 상품 기반 추천을 제공할 수 있기를
> **이유**: 신규 사용자도 빈 추천 목록 없이 서비스를 이용할 수 있도록

**수락 기준**:
- [ ] 최근 7일간 상호작용 수 기준 인기 상품 Top-100을 계산한다
- [ ] 카테고리별 인기 상품도 별도로 계산한다
- [ ] Cold 사용자의 추천 요청 시 인기 상품 목록이 반환된다
- [ ] 응답에 fallback_used: true 플래그가 포함된다
- [ ] 인기 상품 목록은 일 1회 갱신된다

**스토리 포인트**: 3

## E3: 추천 API

### US-3.1: 개인화 추천 조회 API

> **역할**: 클라이언트 개발자로서
> **원함**: 특정 사용자에 대한 개인화 추천 상품 목록을 API로 조회할 수 있기를
> **이유**: 서비스 프론트엔드에서 맞춤 추천을 표시하기 위해

**수락 기준**:
- [ ] GET /api/v1/recommendations/{user_id} 엔드포인트가 동작한다
- [ ] limit 파라미터로 반환 상품 수를 지정할 수 있다 (기본값 20, 최대 100)
- [ ] category 파라미터로 특정 카테고리만 필터링할 수 있다
- [ ] exclude_purchased=true 시 이미 구매한 상품이 제외된다
- [ ] 각 추천 상품에 score, rank, reason이 포함된다
- [ ] 응답에 model_version, generated_at이 포함된다
- [ ] 응답 시간이 p50 < 50ms, p99 < 200ms를 충족한다
- [ ] 존재하지 않는 user_id에 대해 404를 반환한다

**스토리 포인트**: 5

---

### US-3.2: 유사 상품 조회 API

> **역할**: 클라이언트 개발자로서
> **원함**: 특정 상품과 유사한 상품 목록을 API로 조회할 수 있기를
> **이유**: 상품 상세 페이지에서 관련 상품을 표시하기 위해

**수락 기준**:
- [ ] GET /api/v1/items/{item_id}/similar 엔드포인트가 동작한다
- [ ] limit 파라미터로 반환 상품 수를 지정할 수 있다 (기본값 10)
- [ ] 각 유사 상품에 similarity_score가 포함된다
- [ ] 존재하지 않는 item_id에 대해 404를 반환한다

**스토리 포인트**: 3

---

### US-3.3: 배치 추천 조회 API

> **역할**: 백엔드 개발자로서
> **원함**: 여러 사용자의 추천을 한 번에 조회할 수 있기를
> **이유**: 이메일/푸시 발송 등 배치 처리 시 효율적으로 추천을 가져오기 위해

**수락 기준**:
- [ ] POST /api/v1/recommendations/batch 엔드포인트가 동작한다
- [ ] 최대 100명의 user_id를 한 번에 요청할 수 있다
- [ ] 100명 초과 요청 시 400 에러를 반환한다
- [ ] 각 사용자별 추천 결과가 배열로 반환된다
- [ ] 일부 사용자를 찾을 수 없어도 나머지 결과는 정상 반환된다

**스토리 포인트**: 3

---

### US-3.4: API 인증 및 Rate Limiting

> **역할**: API 소비자로서
> **원함**: API Key 기반으로 인증하고, Rate Limit 정보를 확인할 수 있기를
> **이유**: 안전하게 API를 사용하고 호출 제한을 관리하기 위해

**수락 기준**:
- [ ] 모든 API 호출에 X-API-Key 헤더가 필요하다
- [ ] 유효하지 않은 API Key로 요청 시 401을 반환한다
- [ ] 응답 헤더에 X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset이 포함된다
- [ ] Rate Limit 초과 시 429 Too Many Requests를 반환한다
- [ ] API Key별 기본 Rate Limit은 1,000 req/min이다

**스토리 포인트**: 5

---

### US-3.5: 추천 피드백 수집 API

> **역할**: 클라이언트 개발자로서
> **원함**: 추천 결과에 대한 사용자 반응(클릭, 무시, 구매)을 피드백할 수 있기를
> **이유**: 추천 성능의 온라인 지표를 계산하기 위해

**수락 기준**:
- [ ] POST /api/v1/recommendations/{user_id}/feedback 엔드포인트가 동작한다
- [ ] feedback_type은 clicked, dismissed, purchased 중 하나여야 한다
- [ ] 피드백 데이터는 비동기로 저장된다 (Kafka 발행)
- [ ] 응답 시간은 p99 < 30ms를 충족한다

**스토리 포인트**: 3

## E4: 관리자 대시보드

### US-4.1: 메인 대시보드

> **역할**: 관리자로서
> **원함**: 추천 시스템의 핵심 지표와 상태를 한눈에 파악할 수 있기를
> **이유**: 시스템이 정상적으로 동작하고 있는지 빠르게 확인하기 위해

**수락 기준**:
- [ ] 실시간 추천 API 호출 수 추이 그래프가 표시된다
- [ ] 오늘의 CTR, 전환율, 추천 커버리지가 카드 형태로 표시된다
- [ ] 현재 활성 모델 버전과 마지막 학습 시간이 표시된다
- [ ] API 평균 응답시간과 에러율이 표시된다
- [ ] 30초/1분/5분 주기로 자동 새로고침된다
- [ ] 기간 필터(1일/7일/30일/커스텀)가 동작한다

**스토리 포인트**: 8

---

### US-4.2: 오프라인 모델 성능 모니터링

> **역할**: ML 엔지니어로서
> **원함**: 모델의 오프라인 평가 지표 추이를 시각적으로 확인할 수 있기를
> **이유**: 모델 성능 변화를 추적하고 이상을 조기에 감지하기 위해

**수락 기준**:
- [ ] Precision@K, Recall@K, NDCG@K, MAP 추이가 시계열 그래프로 표시된다
- [ ] 모델 버전별 지표를 비교할 수 있는 테이블이 있다
- [ ] 그래프에서 특정 시점을 클릭하면 해당 모델의 상세 정보를 확인할 수 있다
- [ ] K 값(10, 20, 50)을 선택할 수 있다

**스토리 포인트**: 5

---

### US-4.3: 온라인 성능 모니터링

> **역할**: PO로서
> **원함**: 추천의 비즈니스 성과(CTR, 전환율, 매출)를 실시간으로 모니터링할 수 있기를
> **이유**: 추천이 비즈니스 목표에 기여하고 있는지 판단하기 위해

**수락 기준**:
- [ ] 추천 CTR(클릭 수 / 노출 수) 추이 그래프가 표시된다
- [ ] 추천 전환율(구매 수 / 클릭 수) 추이 그래프가 표시된다
- [ ] 추천 경유 매출 금액이 표시된다
- [ ] 추천 vs 비추천 트래픽의 전환율 비교가 가능하다
- [ ] A/B 테스트 실험군별 지표 비교가 가능하다

**스토리 포인트**: 8

---

### US-4.4: 사용자/상품 분석

> **역할**: 관리자로서
> **원함**: 사용자 세그먼트 분포와 특정 사용자의 추천 결과를 조회할 수 있기를
> **이유**: 추천 품질을 정성적으로 검증하고 이상 사례를 확인하기 위해

**수락 기준**:
- [ ] Cold/Warm/Hot 사용자 비율이 파이 차트로 표시된다
- [ ] 사용자 ID를 검색하여 해당 사용자의 추천 결과를 미리볼 수 있다
- [ ] 사용자의 최근 행동 히스토리와 추천 결과를 나란히 볼 수 있다
- [ ] 추천 빈도 상위 상품 Top-20이 테이블로 표시된다
- [ ] 카테고리별 추천 성능(CTR, 전환율)이 비교 차트로 표시된다

**스토리 포인트**: 8

---

### US-4.5: 알림 설정

> **역할**: 관리자로서
> **원함**: 추천 성능 지표에 임계치를 설정하고, 초과 시 알림을 받을 수 있기를
> **이유**: 성능 이상을 즉시 감지하고 대응하기 위해

**수락 기준**:
- [ ] 지표별 Warning/Critical 임계치를 설정할 수 있다
- [ ] 알림 채널(Slack, 이메일)을 선택할 수 있다
- [ ] 알림 지속 조건(N분 이상 지속)을 설정할 수 있다
- [ ] 발생한 알림 히스토리를 조회할 수 있다
- [ ] 알림을 Acknowledge(확인) 처리할 수 있다

**스토리 포인트**: 5

---

### US-4.6: 대시보드 접근 제어

> **역할**: 시스템 관리자로서
> **원함**: 대시보드 접근 권한을 역할 기반으로 관리할 수 있기를
> **이유**: 민감한 데이터와 설정을 보호하기 위해

**수락 기준**:
- [ ] Admin(전체 기능 접근), Viewer(조회만 가능) 역할이 존재한다
- [ ] SSO/OAuth 기반 로그인이 지원된다
- [ ] Admin만 알림 설정, 모델 배포 승인 등을 수행할 수 있다
- [ ] 로그인/로그아웃 이력이 감사 로그에 기록된다

**스토리 포인트**: 5

## E5: 시스템 운영

### US-5.1: 모델 배포 자동화

> **역할**: ML 엔지니어로서
> **원함**: 검증 통과한 모델을 안전하게 프로덕션에 배포할 수 있기를
> **이유**: 수동 개입 없이 최신 모델을 서빙하기 위해

**수락 기준**:
- [ ] 오프라인 평가 통과 시 자동으로 Shadow 배포가 시작된다
- [ ] Shadow 배포 중 온라인 지표를 수집하여 비교한다
- [ ] Admin이 대시보드에서 배포를 승인/거부할 수 있다
- [ ] 배포 후 롤백이 1분 이내에 가능하다
- [ ] 배포 히스토리가 기록된다

**스토리 포인트**: 8

---

### US-5.2: 시스템 헬스체크 및 모니터링

> **역할**: SRE로서
> **원함**: 추천 시스템의 인프라 상태를 실시간으로 모니터링할 수 있기를
> **이유**: 시스템 장애를 조기에 감지하고 빠르게 대응하기 위해

**수락 기준**:
- [ ] /health 엔드포인트가 서비스 상태를 반환한다
- [ ] Prometheus 메트릭이 수집된다 (요청 수, 응답시간, 에러율)
- [ ] Grafana 대시보드에서 시스템 리소스 사용량을 확인할 수 있다
- [ ] Redis, PostgreSQL, Kafka 연결 상태가 모니터링된다

**스토리 포인트**: 5
