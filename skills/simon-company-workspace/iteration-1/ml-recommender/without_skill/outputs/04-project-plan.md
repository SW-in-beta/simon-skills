## 리스크 관리

| 리스크 | 영향도 | 발생 확률 | 완화 전략 |
|--------|--------|-----------|-----------|
| 충분한 사용자 행동 데이터 부족 | 높음 | 중간 | 기존 로그 데이터 활용, 합성 데이터 생성, 최소 데이터 요구량 조기 파악 |
| Cold Start 문제로 추천 품질 저하 | 높음 | 높음 | 인기 상품 폴백, Content-Based 하이브리드 방식 병행 |
| API 응답 시간 SLA 미달 | 높음 | 중간 | 사전 계산 + Redis 캐시 전략, 조기 성능 테스트 |
| 모델 학습 시간 지연 | 중간 | 중간 | 증분 학습 도입, 분산 학습(Spark) 활용 |
| 개인정보보호 규정 위반 | 높음 | 낮음 | 보안팀 사전 검토, 데이터 익명화/가명화, 동의 기반 수집 |
| 핵심 인력 이탈 | 높음 | 낮음 | 기술 문서화, 코드 리뷰 문화, 지식 공유 세션 |
| 외부 시스템(Kafka, Redis) 장애 | 중간 | 중간 | Circuit Breaker 패턴, Graceful Degradation, 폴백 로직 |

## 성공 기준 (Definition of Done)

### MVP 기준 (Phase 2 완료 시)

| 항목 | 기준 |
|------|------|
| 추천 API | 개인화 추천 API p99 < 200ms, 1,000 RPS 처리 가능 |
| 모델 성능 | 오프라인 NDCG@10 >= 0.15 |
| 대시보드 | 핵심 지표(CTR, 모델 상태, API 성능) 실시간 모니터링 가능 |
| 데이터 파이프라인 | 이벤트 수집 → 모델 학습 → 결과 서빙 자동화 |
| 가용성 | 99.9% 가동률 |

### GA 기준 (Phase 4 완료 시)

| 항목 | 기준 |
|------|------|
| 추천 품질 | 온라인 CTR 비추천 대비 20% 이상 향상 |
| 시스템 안정성 | 부하 테스트 통과, 장애 복구 5분 이내 |
| 운영 효율 | 모델 자동 배포, 알림 시스템 동작 |
| 보안 | 보안 감사 통과 |
| 테스트 커버리지 | 백엔드 80% 이상, 프론트엔드 70% 이상 |

- 각 스프린트에서 기술 부채 해결에 전체 용량의 20%를 할당한다
- 기술 부채 항목은 백로그에 등록하고 우선순위를 부여한다
- Phase 3 (Sprint 7)에 집중적인 리팩토링/안정화 시간을 배정한다
- 코드 리뷰 시 기술 부채 발생 여부를 필수 확인 항목으로 포함한다
