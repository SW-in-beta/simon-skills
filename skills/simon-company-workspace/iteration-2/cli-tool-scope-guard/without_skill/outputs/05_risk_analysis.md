# 리스크 분석서

## 1. 기술적 리스크

### R1: 인코딩 자동 감지 오탐
- **발생 확률**: 중간
- **영향도**: 높음 (출력 데이터가 깨짐)
- **설명**: chardet 기반 감지는 짧은 텍스트, 혼합 인코딩, 바이너리 유사 데이터에서 오탐 가능
- **완화 전략**:
  - `--encoding` 수동 지정 옵션을 항상 제공
  - 감지 신뢰도(confidence) 임계값 설정 (예: 0.7 미만이면 경고 출력)
  - `--verbose` 모드에서 감지된 인코딩과 신뢰도 표시
  - 테스트 픽스처에 다양한 인코딩 파일 포함

### R2: 비표준 CSV 파일 처리 실패
- **발생 확률**: 중간
- **영향도**: 중간 (특정 파일에서 파싱 실패)
- **설명**: 실제 CSV 파일은 RFC 4180을 엄격히 따르지 않는 경우가 많음 (행별 컬럼 수 불일치, BOM 포함, 혼합 줄바꿈 등)
- **완화 전략**:
  - `csv.Reader`의 `FieldsPerRecord = -1` 설정으로 유연한 파싱
  - BOM 자동 제거 처리
  - 에러 발생 행 번호와 내용을 포함한 상세 에러 메시지
  - `--strict` / `--lenient` 모드 옵션

### R3: stdin 인코딩 감지 시 데이터 손실
- **발생 확률**: 낮음
- **영향도**: 높음 (입력 데이터 일부 누락)
- **설명**: stdin은 seek 불가하므로 인코딩 감지용으로 읽은 데이터를 복원해야 함
- **완화 전략**:
  - `io.TeeReader` + 버퍼 → `io.MultiReader`로 데이터 복원
  - 충분한 단위 테스트로 경계 조건 검증 (버퍼 경계에서 멀티바이트 문자 분할 등)

### R4: 대용량 파일에서 JSON 배열 출력 시 메모리 이슈
- **발생 확률**: 낮음
- **영향도**: 중간
- **설명**: JSON 배열 모드에서 실수로 전체 데이터를 메모리에 버퍼링할 가능성
- **완화 전략**:
  - 행별 즉시 flush하는 스트리밍 JSON 배열 출력 구현
  - 벤치마크 테스트로 메모리 사용량 검증
  - 메모리 프로파일링 (`pprof`)으로 누수 확인

## 2. 일정 리스크

### R5: 인코딩 관련 엣지 케이스 과다
- **발생 확률**: 중간
- **영향도**: 중간 (일정 지연 1~2일)
- **설명**: 인코딩 감지/변환 엣지 케이스가 예상보다 많을 수 있음
- **완화 전략**:
  - 지원 인코딩 범위를 MVP에서는 UTF-8, EUC-KR, Shift-JIS로 제한
  - 추가 인코딩은 후속 버전에서 지원
  - 인코딩 모듈을 독립적으로 테스트 가능하게 설계

### R6: 테스트 픽스처 준비 지연
- **발생 확률**: 낮음
- **영향도**: 낮음
- **설명**: 다양한 인코딩/형식의 테스트 CSV 파일 준비에 시간 소요
- **완화 전략**:
  - 테스트 코드 내에서 프로그래밍 방식으로 테스트 데이터 생성
  - 인코딩 변환은 Go 표준 라이브러리로 테스트 데이터 직접 생성 가능

## 3. 프로젝트 리스크

### R7: 요구사항 변경 (범위 확장)
- **발생 확률**: 중간
- **영향도**: 높음 (일정 초과)
- **설명**: 개발 중 "이 기능도 추가해달라" 요청이 발생할 수 있음
- **완화 전략**:
  - 범위 정의서를 기준으로 범위 변경 시 마일스톤 재조정
  - Out of Scope 항목을 명확히 문서화하여 기대치 관리
  - 추가 요청은 GitHub Issue로 등록하고 v1.1 이후 검토

## 5. 전체 리스크 수준 평가

**종합 판단: 낮음~중간**

본 프로젝트는 Go 표준 라이브러리와 성숙한 오픈소스 라이브러리를 기반으로 하며, 핵심 로직의 복잡도가 높지 않다. 가장 큰 리스크는 인코딩 감지 정확도와 범위 확장이며, 둘 다 명확한 완화 전략이 존재한다.
