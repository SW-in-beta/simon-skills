# Scope Guard 평가

## 프로젝트 요청
CSV 파일을 읽어서 JSON으로 변환하는 CLI 도구. 인코딩 자동 감지, 컬럼 필터링, 대용량 파일 스트리밍 지원.

## 평가 기준

| 기준 | simon-company 기준 | 이 프로젝트 | 판정 |
|------|-------------------|------------|------|
| 필요 팀 수 | 3개 이상 (예: BE+FE+DevOps) | 2개 (Backend + QA) | 리다이렉트 |
| 인프라 필요 | CI/CD, 배포, 모니터링 필요 | 로컬 실행으로 충분 | 리다이렉트 |
| 디자인 필요 | UI/UX 설계 필요 | CLI 위주, UI 없음 | 리다이렉트 |
| 예상 Feature 수 | 5개 이상 | 3-4개 (CSV 파싱, 인코딩 감지, 컬럼 필터링, 스트리밍 출력) | 리다이렉트 |

## 판정 결과: 리다이렉트 권장

**4개 기준 모두 "리다이렉트"로 판정됨.**

## 근거

1. **팀 규모 과잉**: CLI 도구는 Backend(구현) + QA(테스트) 2개 팀이면 충분하다. Design, Frontend, DBA, DevOps, ML 팀이 모두 불필요하다. 다중 팀 조율 오버헤드만 발생한다.

2. **인프라 불필요**: 로컬에서 실행하는 CLI 도구이므로 CI/CD 파이프라인, Docker 컨테이너화, 모니터링 시스템이 필요 없다.

3. **UI 없음**: 터미널 CLI 인터페이스만 필요하므로 Design 팀의 와이어프레임, 디자인 토큰, 컴포넌트 트리 설계가 불필요하다.

4. **Feature 수 부족**: 핵심 기능이 3-4개로 simon-company의 대규모 프로젝트 기준(5개 이상)에 미달한다.

## 리다이렉트 제안

이 프로젝트는 **simon-bot-pm** 또는 **simon-bot** 사용을 권장합니다.

- **simon-bot-pm**: PM이 요구사항을 정리하고 Backend 개발자가 구현하는 소규모 파이프라인으로 충분
- **simon-bot**: 단일 개발자가 직접 구현하기에 적합한 규모

### 이유
simon-company는 8단계 파이프라인, Triple Review Protocol, 팀간 Contract 정의, Sprint 계획 등 대규모 프로젝트를 위한 중량급 프로세스를 포함합니다. CLI 도구 하나에 이 프로세스를 적용하면 조율 비용이 실제 구현 비용보다 훨씬 커집니다.

SKILL.md 문서의 예시에서도 "CLI 데이터 처리 도구" → PM, CTO, Backend, QA → **simon-bot-pm으로 리다이렉트 제안**으로 명시되어 있습니다.

## 사용자 결정

사용자가 그래도 simon-company로 진행을 원하므로, 축소된 팀 편성으로 Phase 0을 완료합니다.
