# ML 모델 설계서

## 1. 모델 개요

### 1-1. 문제 정의
사용자의 암시적(implicit) 피드백 데이터를 기반으로 각 사용자에게 가장 관심 있을 법한 상품 Top-K를 예측하는 문제.

### 1-2. 문제 유형
- **Top-K 추천 (Ranking Problem)**
- **암시적 피드백 (Implicit Feedback)**: 조회, 클릭, 구매 등 행동 데이터 (명시적 평점이 아닌)
- **협업 필터링 (Collaborative Filtering)**: 사용자-아이템 상호작용 패턴만 활용

---

## 2. 데이터 전처리

### 2-1. 입력 데이터
```
소스: user_item_interactions (DW)
기간: 최근 90일
최소 요건:
  - 사용자: 최소 5회 이상 상호작용
  - 아이템: 최소 3명 이상 사용자와 상호작용
```

### 2-2. 상호작용 점수 계산
각 (user, item) 쌍에 대해 가중 상호작용 점수를 계산한다.

```
interaction_score = Σ (event_weight × event_count × time_decay)
```

| 이벤트 | 가중치 |
|--------|--------|
| page_view | 1.0 |
| click | 1.5 |
| add_to_cart | 3.0 |
| purchase | 5.0 |
| wishlist | 2.5 |

### 2-3. 시간 감쇠 (Time Decay)
최신 행동에 더 높은 가중치를 부여한다.

```
time_decay = exp(-λ × days_since_event)
λ = 0.01 (반감기 약 69일)
```

### 2-4. 데이터 분할
| 데이터셋 | 비율 | 용도 |
|----------|------|------|
| 학습 (Train) | 80% | 모델 학습 |
| 검증 (Validation) | 10% | 하이퍼파라미터 튜닝 |
| 테스트 (Test) | 10% | 최종 평가 |

**분할 방식:** 시간 기반 분할 (Time-based Split)
- 학습: 최근 90일 ~ 10일 전
- 검증: 최근 10일 ~ 5일 전
- 테스트: 최근 5일

## 3. 모델 상세

### 3-1. 주 모델: ALS (Alternating Least Squares)

#### 알고리즘 개요
사용자-아이템 상호작용 행렬 R을 두 개의 저차원 행렬 U(사용자 잠재 인자)와
I(아이템 잠재 인자)로 분해한다.

```
R ≈ U × I^T

R: (m × n) 사용자-아이템 상호작용 행렬
U: (m × k) 사용자 잠재 인자 행렬
I: (n × k) 아이템 잠재 인자 행렬
k: 잠재 인자 수
```

#### 암시적 피드백 처리 (Hu, Koren, Volinsky 2008)
```
선호도: p_ui = 1 if r_ui > 0, else 0
신뢰도: c_ui = 1 + α × r_ui

목적함수: min Σ c_ui × (p_ui - u_u^T × i_i)^2 + λ(||U||^2 + ||I||^2)
```

#### 하이퍼파라미터
| 파라미터 | 설명 | 기본값 | 탐색 범위 | 탐색 방법 |
|----------|------|--------|-----------|-----------|
| n_factors | 잠재 인자 수 | 100 | [50, 100, 150, 200, 300] | Grid Search |
| n_iterations | 학습 반복 수 | 20 | [10, 15, 20, 30, 50] | Grid Search |
| regularization | 정규화 계수 (λ) | 0.1 | [0.01, 0.05, 0.1, 0.5, 1.0] | Grid Search |
| alpha | 신뢰도 계수 (α) | 40 | [10, 20, 40, 60, 100] | Grid Search |

#### 구현 라이브러리
- **1순위:** `implicit` (Python, GPU 지원, 대규모 데이터 최적화)
- **2순위:** Spark MLlib ALS (분산 처리, 초대규모 데이터)

---

### 3-2. 보조 모델: Item-Based CF

#### 알고리즘 개요
아이템 간 유사도를 계산하고, 사용자가 과거에 상호작용한 아이템과
유사한 아이템을 추천한다.

```
score(u, i) = Σ sim(i, j) × r_uj  (j ∈ 사용자 u가 상호작용한 아이템)
sim(i, j) = cos(v_i, v_j)  (아이템 벡터 간 코사인 유사도)
```

#### 용도
- 유사 상품 추천 (`/recommendations/items/{item_id}/similar`)
- ALS 모델 성능 비교를 위한 베이스라인

---

### 3-3. 폴백 모델: Popularity-Based

#### 알고리즘
```
popularity_score = Σ (event_weight × event_count × time_decay) / log(1 + total_users_interacted)
```

#### 용도
- 콜드 스타트 사용자 (상호작용 < 5)
- CF 모델 결과 부족 시 보충
- 전체 인기 상품 + 카테고리별 인기 상품

---

## 4. 추천 생성 파이프라인

### 4-1. 오프라인 배치 파이프라인 (Daily)

```
┌───────────────────────────────────────────────────────────┐
│ Step 1: 데이터 추출                                        │
│ - DW에서 최근 90일 user_item_interactions 추출              │
│ - 최소 상호작용 기준 미달 사용자/아이템 필터링                  │
│ - 예상 소요: 15분                                          │
├───────────────────────────────────────────────────────────┤
│ Step 2: 전처리                                             │
│ - 상호작용 점수 계산 (가중치 + 시간 감쇠)                     │
│ - 희소 행렬 (CSR) 변환                                      │
│ - 예상 소요: 10분                                          │
├───────────────────────────────────────────────────────────┤
│ Step 3: 모델 학습                                          │
│ - ALS 모델 학습 (GPU 활용)                                  │
│ - 하이퍼파라미터: 이전 최적값 사용 (주기적 재탐색)             │
│ - 예상 소요: 30~60분 (데이터 규모에 따라)                    │
├───────────────────────────────────────────────────────────┤
│ Step 4: 오프라인 평가                                       │
│ - 테스트 데이터셋 대상 Precision@K, NDCG@K 등 계산           │
│ - 이전 모델 대비 성능 비교                                   │
│ - 성능 하락 시 자동 롤백 (이전 모델 유지)                     │
│ - 예상 소요: 15분                                          │
├───────────────────────────────────────────────────────────┤
│ Step 5: 추천 결과 사전 계산                                  │
│ - 전체 활성 사용자에 대해 Top-50 추천 상품 계산               │
│ - 전체 활성 아이템에 대해 유사 아이템 Top-30 계산              │
│ - 인기 상품 Top-100 계산 (전체 + 카테고리별)                  │
│ - 예상 소요: 30분                                          │
├───────────────────────────────────────────────────────────┤
│ Step 6: 결과 적재                                          │
│ - Redis에 추천 결과 적재 (TTL: 25시간, 다음 배치 전 만료 방지) │
│ - Model Registry에 모델 아티팩트 저장                        │
│ - 모델 메타데이터 DB 업데이트                                │
│ - 예상 소요: 15분                                          │
├───────────────────────────────────────────────────────────┤
│ 총 예상 소요: 약 2시간                                      │
└───────────────────────────────────────────────────────────┘
```

### 4-2. 온라인 서빙 흐름

```
[API 요청: GET /recommendations/{user_id}]
         │
         ▼
[A/B 테스트 라우터]
  - 실험 대상 사용자인가?
  - 어떤 variant에 할당?
         │
         ▼
[캐시 조회: Redis reco:user:{user_id}]
    │                    │
    ▼ (HIT)              ▼ (MISS)
[결과 반환]         [폴백 체인]
                        │
                        ▼
               [세그먼트 인기 상품]
                    │         │
                    ▼ (HIT)   ▼ (MISS)
               [결과 반환]  [전체 인기 상품]
                                  │
                                  ▼
                           [결과 반환]
```

## 5. 평가 체계

### 5-1. 오프라인 평가 지표

| 지표 | 수식 | 목표값 | 설명 |
|------|------|--------|------|
| Precision@K | (관련 아이템 ∩ 추천 Top-K) / K | >= 0.15 | 추천의 정확도 |
| Recall@K | (관련 아이템 ∩ 추천 Top-K) / 전체 관련 아이템 | >= 0.10 | 추천의 재현율 |
| NDCG@K | DCG@K / IDCG@K | >= 0.20 | 순위 품질 |
| MAP | 평균 AP | >= 0.12 | 전체 랭킹 품질 |
| 커버리지 | 추천된 고유 아이템 / 전체 아이템 | >= 80% | 추천 다양성 |
| Hit Rate@K | 추천 Top-K에 관련 아이템이 1개 이상 있는 사용자 비율 | >= 0.50 | 추천 유용성 |

### 5-2. 온라인 평가 지표

| 지표 | 측정 방법 | 목표값 |
|------|-----------|--------|
| CTR | 추천 클릭 수 / 추천 노출 수 | 기존 대비 +20% |
| 전환율 | 추천 구매 수 / 추천 클릭 수 | 기존 대비 +15% |
| 사용자당 매출 | 추천 경유 매출 / 활성 사용자 수 | 기존 대비 +10% |
| 세션당 클릭 수 | 추천 클릭 수 / 세션 수 | 증가 |
| 이탈률 변화 | 추천 노출 후 이탈 비율 | 감소 |

### 5-3. 모델 배포 기준
새 모델이 서빙에 투입되려면 다음 조건을 모두 만족해야 한다.

```
[자동 배포 조건]
1. 오프라인 지표가 모두 목표값 이상
2. 이전 모델 대비 주요 지표(NDCG@10) 5% 이상 하락하지 않음
3. 학습 파이프라인 에러 없음

[수동 검토 필요]
- 이전 모델 대비 주요 지표 5% 이상 하락 시
- 커버리지 80% 미달 시
- 학습 데이터 규모가 이전 대비 20% 이상 변동 시
```

## 6. 콜드 스타트 전략

### 6-1. 신규 사용자 (User Cold Start)

| 단계 | 조건 | 전략 |
|------|------|------|
| Tier 0 | 상호작용 0회 | 전체 인기 상품 |
| Tier 1 | 상호작용 1~2회 | 카테고리 기반 인기 상품 |
| Tier 2 | 상호작용 3~4회 | 유사 사용자 그룹 추천 |
| Tier 3 | 상호작용 5회+ | 개인화 CF 추천 |

### 6-2. 신규 아이템 (Item Cold Start)

| 전략 | 설명 |
|------|------|
| 카테고리 배치 | 같은 카테고리 아이템 목록에 랜덤 삽입 |
| 탐색 슬롯 | 추천 결과 중 1~2개 슬롯을 신규 아이템용으로 할당 |
| 부스팅 | 등록 후 7일간 인기도 점수에 부스팅 적용 |

## 7. 모델 모니터링

### 7-1. 드리프트 감지
| 지표 | 감시 대상 | 알림 기준 |
|------|-----------|-----------|
| 데이터 드리프트 | 상호작용 분포 변화 (PSI) | PSI > 0.2 |
| 성능 드리프트 | 온라인 CTR 변화 | 주간 평균 대비 20% 하락 |
| 커버리지 드리프트 | 추천 가능 사용자 비율 | 이전 대비 10% 하락 |

### 7-2. 재학습 트리거
- **정기:** 매일 새벽 3시 (기본)
- **임시:** 데이터 드리프트 감지 시 수동 트리거
- **긴급:** 온라인 성능 급격 하락 시 이전 모델로 즉시 롤백 + 원인 분석
