# 리스크 분석 및 완화 전략

## 1. 기술 리스크

### R-001: 콜드 스타트 문제
| 항목 | 내용 |
|------|------|
| 설명 | 신규 사용자/아이템에 대해 협업 필터링 추천이 불가능 |
| 영향도 | 높음 |
| 발생 가능성 | 높음 (신규 사용자 유입은 필연적) |
| 영향 범위 | 추천 품질, 사용자 경험, 전환율 |

**완화 전략:**
1. 다단계 폴백 체인 구현 (개인화 → 세그먼트 → 인기 상품 → 큐레이션)
2. 신규 사용자 온보딩 시 관심 카테고리 선택 유도
3. 최소 상호작용 기준(5회)에 도달하면 즉시 개인화 추천으로 전환
4. 콜드 스타트 비율을 대시보드에서 모니터링
5. 향후 콘텐츠 기반 필터링 하이브리드 모델로 보완

**모니터링 지표:** 폴백 추천 비율, 신규 사용자 추천 CTR

---

### R-002: 데이터 부족 또는 편향
| 항목 | 내용 |
|------|------|
| 설명 | 학습 데이터가 부족하거나 특정 상품/카테고리에 편향 |
| 영향도 | 높음 |
| 발생 가능성 | 중간 |
| 영향 범위 | 모델 성능, 추천 다양성 |

**완화 전략:**
1. 서비스 초기 최소 데이터 기준 설정 (사용자 1만명, 상호작용 50만건)
2. 인기도 편향 제거: 인기 아이템에 대한 다운샘플링
3. 다양성 지표(Gini, Entropy) 모니터링
4. 탐색(exploration) 슬롯을 통해 롱테일 아이템 노출
5. 가중치 조정을 통한 이벤트별 편향 보정

**모니터링 지표:** 추천 커버리지, Gini 계수, 카테고리 분포

---

### R-003: API 응답 시간 초과
| 항목 | 내용 |
|------|------|
| 설명 | P99 200ms 요구사항을 충족하지 못하는 경우 |
| 영향도 | 중간 |
| 발생 가능성 | 중간 |
| 영향 범위 | 사용자 경험, 서비스 SLA |

**완화 전략:**
1. 추천 결과 사전 계산 + Redis 캐싱 (주 전략)
2. 캐시 미스 시 즉시 폴백 (인기 상품, 캐시에 항상 보유)
3. 타임아웃 설정: 캐시 조회 50ms, 전체 요청 150ms
4. Redis Cluster 구성으로 읽기 분산
5. 부하 테스트를 통한 병목 사전 발견

**모니터링 지표:** P50/P99 응답 시간, 캐시 히트율, Redis 지연

---

### R-004: 모델 성능 미달
| 항목 | 내용 |
|------|------|
| 설명 | 오프라인/온라인 평가에서 목표 지표를 달성하지 못하는 경우 |
| 영향도 | 높음 |
| 발생 가능성 | 중간 |
| 영향 범위 | 추천 품질, 비즈니스 KPI |

**완화 전략:**
1. 복수 모델 후보 평가 (ALS, Item CF, User CF)
2. 오프라인 평가 기준 미달 시 자동 롤백
3. 온라인 A/B 테스트로 실제 영향 검증 후 배포
4. 하이퍼파라미터 튜닝 체계화 (Grid Search → Bayesian Optimization)
5. 점진적 목표 상향 (초기에는 인기 상품 대비 개선만 확인)

**모니터링 지표:** Precision@K, NDCG@K, 온라인 CTR, 전환율

---

### R-005: 데이터 파이프라인 장애
| 항목 | 내용 |
|------|------|
| 설명 | Kafka, Spark, Airflow 등 파이프라인 구성 요소 장애 |
| 영향도 | 중간 |
| 발생 가능성 | 중간 |
| 영향 범위 | 데이터 적시성, 모델 학습 |

**완화 전략:**
1. 관리형 서비스 사용 (MSK, MWAA 등)으로 운영 부담 최소화
2. Kafka: 리플리케이션 팩터 3, ISR 설정
3. Airflow: DAG 재시도 설정 (최대 3회, 지수 백오프)
4. S3 원본 이벤트 보관으로 재처리 가능
5. 학습 실패 시 이전 모델로 자동 폴백 (추천 서빙은 계속)

**모니터링 지표:** Kafka consumer lag, Airflow DAG 성공률, 데이터 갱신 지연

## 2. 프로젝트 리스크

### R-006: 일정 지연
| 항목 | 내용 |
|------|------|
| 설명 | 예상보다 개발 기간이 길어지는 경우 |
| 영향도 | 중간 |
| 발생 가능성 | 중간 |
| 영향 범위 | 출시 일정, 비즈니스 기회 |

**완화 전략:**
1. MVP 우선 접근: P0 기능만으로 초기 출시 가능하도록 설계
2. 스프린트 리뷰에서 조기에 지연 감지
3. 기능 축소 옵션: 대시보드 일부 기능을 후순위로 이동 가능
4. 병렬 작업 극대화 (데이터/ML/API 팀 독립 진행)

**MVP 최소 범위:**
- 개인 추천 API (폴백 포함)
- 기본 대시보드 (CTR, 전환율만)
- 일일 모델 학습 배치
- → Sprint 1~5 완료로 MVP 출시 가능 (약 10주)

## 3. 보안/규정 리스크

### R-008: 개인정보 처리
| 항목 | 내용 |
|------|------|
| 설명 | 사용자 행동 데이터 수집 및 활용 시 개인정보 보호 규정 준수 |
| 영향도 | 높음 |
| 발생 가능성 | 낮음 |
| 영향 범위 | 법적 리스크, 서비스 신뢰도 |

**완화 전략:**
1. user_id는 내부 식별자만 사용 (이름, 이메일 등 PII 미수집)
2. 데이터 보관 기간 제한 (90일)
3. 개인정보 처리 방침에 추천 목적 데이터 활용 명시
4. 사용자 데이터 삭제 요청 처리 프로세스 구축
5. 법무팀 사전 검토
