# 사용자 스토리 (User Stories)

## 페르소나 정의

### P1: 일반 사용자 (End User)
- 쇼핑몰/서비스를 이용하는 최종 사용자
- 자신에게 맞는 상품을 빠르게 찾고 싶어한다

### P2: 운영 관리자 (Admin)
- 추천 시스템을 관리하고 성능을 모니터링하는 내부 운영자
- 추천 품질을 지속적으로 개선하고 싶어한다

### P3: ML 엔지니어 (ML Engineer)
- 추천 모델을 개발하고 실험하는 기술 담당자
- 모델 성능을 빠르게 비교하고 배포하고 싶어한다

### P4: 서비스 개발자 (Developer)
- 추천 API를 연동하는 클라이언트 서비스 개발자
- 안정적이고 사용하기 쉬운 API를 원한다

---

## Epic 1: 개인화 추천 (Personalized Recommendations)

### US-101: 개인 맞춤 상품 추천 조회
> **사용자(P1)로서**, 내 행동 이력에 기반한 맞춤 상품 추천을 받고 싶다,
> **왜냐하면** 내가 관심 있을 만한 상품을 빠르게 발견하고 싶기 때문이다.

**수락 기준:**
- [ ] 로그인한 사용자에게 최소 10개의 추천 상품이 표시된다
- [ ] 추천 상품은 사용자의 과거 행동(조회, 구매, 클릭)을 반영한다
- [ ] 이미 구매한 상품은 기본적으로 추천 목록에서 제외된다
- [ ] 추천 목록은 1초 이내에 로딩된다

**스토리 포인트:** 8

---

### US-102: 유사 상품 추천
> **사용자(P1)로서**, 현재 보고 있는 상품과 유사한 다른 상품을 추천받고 싶다,
> **왜냐하면** 비교 검토를 통해 더 나은 구매 결정을 내리고 싶기 때문이다.

**수락 기준:**
- [ ] 상품 상세 페이지에서 "비슷한 상품" 섹션이 표시된다
- [ ] 유사 상품은 현재 상품과 행동 패턴이 유사한 상품이다
- [ ] 최소 5개, 최대 20개의 유사 상품을 표시한다
- [ ] 품절 상품은 제외하거나 별도 표시한다

**스토리 포인트:** 5

---

### US-103: 신규 사용자 추천
> **신규 사용자(P1)로서**, 행동 이력이 없더라도 인기 있는 상품을 추천받고 싶다,
> **왜냐하면** 서비스를 처음 사용하더라도 좋은 상품을 발견하고 싶기 때문이다.

**수락 기준:**
- [ ] 행동 이력이 없는 사용자에게 인기 상품 기반 추천이 제공된다
- [ ] 카테고리별 인기 상품이 다양하게 포함된다
- [ ] 사용자가 몇 가지 행동을 하면 점진적으로 개인화된 추천으로 전환된다
- [ ] 폴백 추천임을 추적할 수 있다 (내부 로깅)

**스토리 포인트:** 5

---

### US-104: 카테고리 필터 추천
> **사용자(P1)로서**, 특정 카테고리 내에서 맞춤 추천을 받고 싶다,
> **왜냐하면** 현재 관심 있는 카테고리에서 빠르게 상품을 찾고 싶기 때문이다.

**수락 기준:**
- [ ] 카테고리 파라미터를 지정하면 해당 카테고리 내에서만 추천한다
- [ ] 카테고리 필터 적용 후에도 최소 5개 추천이 제공된다
- [ ] 필터된 추천도 개인화가 반영된다

**스토리 포인트:** 3

## Epic 2: REST API

### US-201: 추천 API 연동
> **서비스 개발자(P4)로서**, 간단한 REST API 호출로 추천 결과를 가져오고 싶다,
> **왜냐하면** 최소한의 연동 작업으로 추천 기능을 서비스에 통합하고 싶기 때문이다.

**수락 기준:**
- [ ] `GET /api/v1/recommendations/{user_id}` 엔드포인트가 동작한다
- [ ] API 문서(OpenAPI/Swagger)가 자동 생성되어 접근 가능하다
- [ ] 요청/응답 예시가 문서에 포함되어 있다
- [ ] SDK 없이 HTTP 클라이언트만으로 연동 가능하다

**스토리 포인트:** 8

---

### US-202: API 인증 및 보안
> **서비스 개발자(P4)로서**, API 호출 시 안전한 인증 방식을 사용하고 싶다,
> **왜냐하면** 비인가 접근으로부터 추천 데이터를 보호해야 하기 때문이다.

**수락 기준:**
- [ ] API Key 기반 인증이 지원된다
- [ ] 유효하지 않은 API Key로 호출 시 401 응답을 반환한다
- [ ] 클라이언트별 속도 제한이 적용된다 (초당 100 요청)
- [ ] 속도 제한 초과 시 429 응답과 Retry-After 헤더를 반환한다

**스토리 포인트:** 5

---

### US-203: 배치 추천 요청
> **서비스 개발자(P4)로서**, 여러 사용자의 추천을 한 번에 요청하고 싶다,
> **왜냐하면** 대량 푸시 알림이나 이메일 캠페인에 추천을 활용하고 싶기 때문이다.

**수락 기준:**
- [ ] `POST /api/v1/recommendations/batch` 엔드포인트가 동작한다
- [ ] 한 번에 최대 100명의 사용자 추천을 요청할 수 있다
- [ ] 개별 사용자별 실패가 전체 요청에 영향을 주지 않는다 (partial success)
- [ ] 응답 시간은 사용자 수에 비례하되, 100명 기준 5초 이내

**스토리 포인트:** 5

---

### US-204: 추천 피드백 전송
> **서비스 개발자(P4)로서**, 사용자의 추천 상호작용(클릭, 무시, 구매)을 피드백으로 전송하고 싶다,
> **왜냐하면** 추천 성능 측정과 모델 개선에 활용되어야 하기 때문이다.

**수락 기준:**
- [ ] `POST /api/v1/recommendations/{recommendation_id}/feedback` 엔드포인트가 동작한다
- [ ] click, dismiss, purchase 등의 액션 타입을 지원한다
- [ ] `recommendation_id`를 통해 어떤 추천에 대한 피드백인지 추적 가능하다
- [ ] 피드백 데이터가 대시보드 지표에 반영된다

**스토리 포인트:** 3

## Epic 3: 관리자 대시보드

### US-301: 추천 성능 요약 보기
> **운영 관리자(P2)로서**, 대시보드 홈에서 핵심 추천 성능 지표를 한눈에 보고 싶다,
> **왜냐하면** 추천 시스템의 전반적인 건강 상태를 빠르게 파악해야 하기 때문이다.

**수락 기준:**
- [ ] 대시보드 홈에 CTR, 전환율, 일 추천 요청 수가 카드 형태로 표시된다
- [ ] 각 지표의 전일/전주 대비 변화율이 표시된다
- [ ] 지표 클릭 시 상세 분석 페이지로 이동한다
- [ ] 데이터가 5분 이내에 갱신된다

**스토리 포인트:** 5

---

### US-302: 추천 성능 추이 분석
> **운영 관리자(P2)로서**, 추천 성능의 시간별 추이를 그래프로 확인하고 싶다,
> **왜냐하면** 성능 변화 트렌드를 파악하고 이상 징후를 조기에 감지해야 하기 때문이다.

**수락 기준:**
- [ ] 일별/주별/월별 기간 필터로 CTR, 전환율 추이를 조회할 수 있다
- [ ] 그래프에서 특정 날짜를 클릭하면 상세 데이터를 볼 수 있다
- [ ] 카테고리별 성능 비교가 가능하다
- [ ] 커스텀 기간 설정이 가능하다 (최대 90일)

**스토리 포인트:** 8

---

### US-303: 모델 버전 관리
> **ML 엔지니어(P3)로서**, 대시보드에서 모델 버전 목록과 각 버전의 성능을 비교하고 싶다,
> **왜냐하면** 최적의 모델을 빠르게 선택하고 배포해야 하기 때문이다.

**수락 기준:**
- [ ] 모델 버전 목록이 학습 일시, 오프라인 평가 지표와 함께 표시된다
- [ ] 두 버전을 선택하여 지표를 나란히 비교할 수 있다
- [ ] 현재 서빙 중인 모델이 표시된다
- [ ] 이전 버전으로 롤백할 수 있다

**스토리 포인트:** 8

---

### US-304: A/B 테스트 실험 관리
> **ML 엔지니어(P3)로서**, 대시보드에서 A/B 테스트를 생성하고 결과를 확인하고 싶다,
> **왜냐하면** 새로운 모델이 기존 모델보다 실제로 나은지 통계적으로 검증해야 하기 때문이다.

**수락 기준:**
- [ ] 새로운 실험을 생성할 수 있다 (이름, 변형, 트래픽 비율, 기간)
- [ ] 실행 중인 실험의 실시간 성능 비교를 볼 수 있다
- [ ] 통계적 유의성(p-value)이 표시된다
- [ ] 실험을 종료하고 승자 변형을 전체 적용할 수 있다

**스토리 포인트:** 13

---

### US-305: 특정 사용자 추천 미리보기
> **운영 관리자(P2)로서**, 특정 사용자 ID를 입력하면 해당 사용자의 추천 결과를 미리 확인하고 싶다,
> **왜냐하면** CS 문의 대응 및 추천 품질 직접 검증이 필요하기 때문이다.

**수락 기준:**
- [ ] 사용자 ID 입력 시 해당 사용자의 추천 결과가 표시된다
- [ ] 각 추천 상품의 점수(score)와 순위가 표시된다
- [ ] 해당 사용자의 최근 행동 이력도 함께 조회된다
- [ ] 추천 전략 (개인화/폴백) 정보가 표시된다

**스토리 포인트:** 5

---

### US-306: 시스템 상태 모니터링
> **운영 관리자(P2)로서**, API 응답 시간, 에러율, 학습 파이프라인 상태를 모니터링하고 싶다,
> **왜냐하면** 시스템 장애를 조기에 감지하고 대응해야 하기 때문이다.

**수락 기준:**
- [ ] API P50/P99 응답 시간이 실시간 그래프로 표시된다
- [ ] 에러율 추이가 표시되고, 임계값 초과 시 시각적 알림이 표시된다
- [ ] 모델 학습 파이프라인 상태(성공/실패/진행중)가 표시된다
- [ ] 마지막 학습 시간과 다음 학습 예정 시간이 표시된다

**스토리 포인트:** 5

---

## Epic 4: 데이터 수집

### US-401: 사용자 행동 이벤트 수집
> **시스템으로서**, 사용자의 모든 관련 행동 이벤트를 실시간으로 수집하고 저장하고 싶다,
> **왜냐하면** 추천 모델 학습과 성능 측정의 근거 데이터가 필요하기 때문이다.

**수락 기준:**
- [ ] page_view, click, add_to_cart, purchase, rating, wishlist 이벤트를 수집한다
- [ ] 이벤트 유실률이 0.1% 이하이다
- [ ] 중복 이벤트가 제거된다 (idempotent)
- [ ] 이벤트 발생 후 5분 이내에 분석 가능하다

**스토리 포인트:** 13

---

### US-402: 이벤트 데이터 품질 보장
> **데이터 엔지니어로서**, 수집된 이벤트의 스키마 유효성을 검증하고 비정상 데이터를 격리하고 싶다,
> **왜냐하면** 모델 학습에 깨끗한 데이터만 사용해야 하기 때문이다.

**수락 기준:**
- [ ] 이벤트 스키마 유효성 검증이 수집 시점에 수행된다
- [ ] 유효하지 않은 이벤트는 DLQ(Dead Letter Queue)에 적재된다
- [ ] DLQ 적재 건수가 대시보드에서 모니터링된다
- [ ] 일별 데이터 품질 리포트가 생성된다

**스토리 포인트:** 5
