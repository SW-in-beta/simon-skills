# 기술 스택 의사결정 기록 (ADR)

## ADR-001: 추천 API 서버 언어/프레임워크

### 상태
결정됨

### 맥락
추천 API 서버는 P99 200ms 이하 응답, 초당 5,000 요청 처리가 요구된다.
주로 Redis 캐시에서 사전 계산된 결과를 조회하여 반환하는 I/O 바운드 작업이다.

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| Go (Gin/Echo) | 높은 성능, 낮은 메모리, 동시성 우수 | ML 생태계 부족 |
| Python (FastAPI) | ML 팀과 언어 통일, 빠른 개발 | 상대적 낮은 성능 |
| Rust (Actix) | 최고 성능 | 높은 러닝 커브, 채용 어려움 |

### 결정
**Go (Echo)** 선택

### 근거
- 주 작업이 Redis 캐시 조회 → 반환이므로, Go의 성능 이점이 크다
- P99 200ms 요구사항을 안정적으로 충족할 수 있다
- 팀 내 Go 개발 경험이 있다 (기존 서비스 참고)
- ML 로직은 별도 파이프라인에서 처리하므로 API 서버에서 Python 필요 없음

## ADR-002: ML 프레임워크

### 상태
결정됨

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| implicit (Python) | 암시적 피드백 최적화, GPU 지원, 간결 | 대규모 분산 미지원 |
| Spark MLlib | 분산 처리, 초대규모 데이터 | 모델 종류 제한, 복잡한 설정 |
| surprise (Python) | 다양한 CF 알고리즘, 사용 편의 | 대규모 데이터 미지원, GPU 미지원 |
| PyTorch (자체 구현) | 유연성, 커스텀 모델 | 개발 비용 높음 |

### 결정
**implicit** (1순위) + **Spark MLlib** (스케일 아웃 필요 시)

### 근거
- ALS with implicit feedback에 최적화된 라이브러리
- GPU 가속 지원으로 학습 속도 우수
- 1,000만 규모 상호작용까지 단일 머신에서 처리 가능
- 데이터 규모가 더 커지면 Spark MLlib로 전환 가능

## ADR-003: 캐시 솔루션

### 상태
결정됨

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| Redis Cluster | 고성능, 다양한 자료구조, 관리형 서비스 | 메모리 비용 |
| Memcached | 단순, 빠름 | 자료구조 제한, 클러스터링 제한 |
| DynamoDB DAX | AWS 통합 | 벤더 락인, 비용 |

### 결정
**Redis Cluster** (관리형: AWS ElastiCache 또는 GCP Memorystore)

### 근거
- JSON 직렬화된 추천 결과 저장에 적합
- TTL 관리 용이
- 관리형 서비스로 운영 부담 최소화
- A/B 테스트 사용자 할당 저장에도 활용 가능

## ADR-004: 메시지 큐

### 상태
결정됨

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| Apache Kafka | 높은 처리량, 내구성, 리플레이 | 운영 복잡도 |
| AWS Kinesis | 관리형, AWS 통합 | 벤더 락인, 샤드 관리 |
| RabbitMQ | 유연한 라우팅, 설정 간단 | 대규모 처리량 제한 |

### 결정
**Apache Kafka** (관리형: AWS MSK 또는 Confluent Cloud)

### 근거
- 초당 10,000+ 이벤트 처리 요구사항 충족
- 이벤트 리플레이(재처리) 가능 — 데이터 파이프라인 오류 복구에 유리
- 기존 인프라에 Kafka 경험 있음
- 관리형 서비스로 운영 부담 감소

## ADR-005: 데이터 웨어하우스

### 상태
결정됨

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| BigQuery | 서버리스, 대용량 분석 최적 | GCP 종속 |
| Redshift | AWS 통합, Spectrum(S3 직접 조회) | 클러스터 관리 |
| Snowflake | 멀티 클라우드, 분리된 컴퓨팅/스토리지 | 비용 |

### 결정
**기존 인프라에 따라 결정** (BigQuery 또는 Redshift)

### 근거
- 기존 회사 인프라의 클라우드 환경을 따른다
- 핵심은 SQL 기반 집계 쿼리 성능이며, 모든 선택지가 충족
- 대시보드 메트릭 조회 및 모델 학습 데이터 추출에 사용

## ADR-006: 관리자 대시보드 프레임워크

### 상태
결정됨

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| Next.js + React | SSR/SSG, 풍부한 생태계, 타입 안전 | 빌드 설정 복잡 |
| Vue.js (Nuxt) | 낮은 러닝 커브, 좋은 문서 | React 대비 생태계 작음 |
| Grafana + 커스텀 패널 | 모니터링 특화, 빠른 구축 | 커스텀 UI 제한 |

### 결정
**Next.js + React + TypeScript**

### 근거
- 관리자 대시보드는 모니터링뿐 아니라 모델 관리, A/B 테스트 관리 등 CRUD 기능도 포함
- Grafana로는 관리 기능 구현이 제한적
- 팀의 React 경험 활용
- Recharts 또는 Apache ECharts로 데이터 시각화 구현

### 차트 라이브러리
| 선택지 | 장점 | 단점 |
|--------|------|------|
| Recharts | React 네이티브, 선언적 API | 대규모 데이터 성능 |
| Apache ECharts | 고성능, 풍부한 차트 종류 | React 래퍼 필요 |
| D3.js | 완전 커스텀 가능 | 높은 개발 비용 |

**결정:** Recharts (간단한 차트) + ECharts (복잡한 시각화)

## ADR-007: ML 파이프라인 오케스트레이션

### 상태
결정됨

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| Apache Airflow | 표준, 풍부한 오퍼레이터, 스케줄링 | 설정 복잡, 무거움 |
| Prefect | 모던 API, 쉬운 설정 | 상대적 작은 생태계 |
| Dagster | 타입 안전, 테스트 용이 | 러닝 커브 |
| Kubernetes CronJob | 단순, 인프라 통합 | 의존성 관리 어려움 |

### 결정
**Apache Airflow** (관리형: AWS MWAA 또는 Cloud Composer)

### 근거
- 데이터/ML 파이프라인 표준 도구
- DAG 기반 의존성 관리, 재시도, 알림 기능
- 기존 데이터 팀의 경험 활용
- 관리형 서비스로 운영 부담 감소

## ADR-008: 모델 레지스트리

### 상태
결정됨

### 선택지
| 선택지 | 장점 | 단점 |
|--------|------|------|
| MLflow | 오픈소스, 모델 버전 관리, 실험 추적 | 호스팅 필요 |
| Weights & Biases | 풍부한 UI, 실험 추적 | 비용, 외부 서비스 |
| 자체 구현 (S3 + DB) | 단순, 커스텀 가능 | 개발/유지보수 비용 |

### 결정
**MLflow**

### 근거
- 모델 버전 관리, 실험 추적, 모델 서빙 모두 지원
- 오픈소스로 비용 부담 없음
- Airflow와 연동 용이
- REST API 제공으로 대시보드 연동 가능
