# 팀 구성 및 역할 정의

## 1. 팀 구조

```
                    ┌─────────────────┐
                    │  Product Owner   │
                    │   (1명)          │
                    └────────┬────────┘
                             │
                    ┌────────┴────────┐
                    │   Tech Lead      │
                    │   (1명, ML)      │
                    └────────┬────────┘
                             │
          ┌──────────────────┼──────────────────┐
          │                  │                  │
   ┌──────┴──────┐   ┌──────┴──────┐   ┌──────┴──────┐
   │  ML Squad    │   │ Backend     │   │ Frontend    │
   │  (2명)       │   │ Squad (2명) │   │ Squad (1명) │
   └─────────────┘   └─────────────┘   └─────────────┘
```

## 2. 팀 구성원 상세

### Product Owner (1명)
| 항목 | 내용 |
|------|------|
| 주요 역할 | 비즈니스 요구사항 정의, 백로그 관리, 우선순위 결정 |
| 필요 역량 | 추천 시스템 도메인 이해, 데이터 분석 능력, 이해관계자 소통 |
| 참여도 | 50% (겸임 가능) |
| 산출물 | PRD, 사용자 스토리, 수락 기준, KPI 대시보드 |

### Tech Lead / ML Engineer (1명)
| 항목 | 내용 |
|------|------|
| 주요 역할 | 기술 아키텍처 설계, 모델 선정 및 파이프라인 설계, 코드 리뷰 |
| 필요 역량 | 추천 시스템 경험 3년+, Python/Spark, MLOps 경험 |
| 참여도 | 100% |
| 산출물 | 아키텍처 문서, 모델 평가 보고서, 기술 의사결정 기록 |

### ML Engineer (1명)
| 항목 | 내용 |
|------|------|
| 주요 역할 | 추천 모델 개발, 학습 파이프라인 구현, 모델 평가 |
| 필요 역량 | Python, scikit-learn/PyTorch, 추천 알고리즘 이해, 특성 엔지니어링 |
| 참여도 | 100% |
| 산출물 | 모델 코드, 학습 파이프라인, 오프라인 평가 결과 |

### Data Engineer (1명)
| 항목 | 내용 |
|------|------|
| 주요 역할 | 데이터 수집 파이프라인, ETL, 데이터 저장소 구축 |
| 필요 역량 | Kafka/Kinesis, Spark, Airflow, SQL, 데이터 모델링 |
| 참여도 | 100% (Phase 2~3), 50% (Phase 4~6) |
| 산출물 | ETL 파이프라인, 데이터 스키마, 데이터 품질 모니터링 |

### Backend Engineer (2명)
| 항목 | 내용 |
|------|------|
| 주요 역할 | 추천 API 개발, 서빙 인프라 구축, A/B 테스트 프레임워크 |
| 필요 역량 | Go 또는 Python(FastAPI), Redis, gRPC/REST, 대용량 트래픽 경험 |
| 참여도 | 100% |
| 산출물 | API 서버, 캐싱 계층, 서빙 인프라, API 문서 |

### Frontend Engineer (1명)
| 항목 | 내용 |
|------|------|
| 주요 역할 | 관리자 대시보드 개발 |
| 필요 역량 | React/Next.js, 데이터 시각화(D3.js/Chart.js), REST API 연동 |
| 참여도 | 100% (Phase 5), 30% (그 외) |
| 산출물 | 관리자 대시보드 UI, 차트/그래프 컴포넌트 |

### QA Engineer (1명, 겸임)
| 항목 | 내용 |
|------|------|
| 주요 역할 | 테스트 계획 수립, E2E 테스트, 성능 테스트 |
| 필요 역량 | API 테스트, 부하 테스트, 테스트 자동화 |
| 참여도 | 50% (겸임) |
| 산출물 | 테스트 계획서, 테스트 케이스, 성능 테스트 보고서 |

## 3. 팀 규모 요약

| 구분 | 인원 | 비고 |
|------|------|------|
| Product Owner | 1명 | 겸임 (50%) |
| ML Engineer (Tech Lead 포함) | 2명 | 전담 |
| Data Engineer | 1명 | 전담 → 반전담 |
| Backend Engineer | 2명 | 전담 |
| Frontend Engineer | 1명 | 전담 (Phase 5) |
| QA Engineer | 1명 | 겸임 (50%) |
| **총계** | **8명** | **전환 FTE 약 6.3명** |

## 4. RACI 매트릭스

| 활동 | PO | Tech Lead | ML Eng | Data Eng | BE Eng | FE Eng | QA |
|------|----|-----------|--------|----------|--------|--------|-----|
| 요구사항 정의 | A | C | I | I | I | I | I |
| 아키텍처 설계 | I | A/R | C | C | C | C | I |
| 데이터 파이프라인 | I | C | C | A/R | C | - | I |
| 모델 개발 | I | A | R | C | I | - | I |
| API 개발 | I | A | C | I | R | - | C |
| 대시보드 개발 | C | A | I | I | C | R | C |
| A/B 테스트 | A | C | C | I | R | I | C |
| 품질 보증 | I | C | I | I | C | C | A/R |

> R: Responsible (실행), A: Accountable (책임), C: Consulted (자문), I: Informed (통보)

## 5. 협업 방식

### 스프린트 운영
- **스프린트 기간**: 2주
- **스프린트 계획**: 매 스프린트 첫째 날 오전 (2시간)
- **데일리 스탠드업**: 매일 오전 10시 (15분)
- **스프린트 리뷰/회고**: 매 스프린트 마지막 날 오후

### 커뮤니케이션
- **주요 채널**: Slack #recopilot-team
- **문서 관리**: Confluence 또는 Notion
- **이슈 관리**: Jira 또는 Linear
- **코드 관리**: GitHub (PR 기반 코드 리뷰 필수)

### 기술 공유
- **모델 리뷰**: 격주 1회 (ML 팀 → 전체 팀)
- **기술 공유 세션**: 월 1회 (팀원 돌아가며 발표)
