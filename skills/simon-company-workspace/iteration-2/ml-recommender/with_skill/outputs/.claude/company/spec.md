# Spec: 사용자 행동 기반 상품 추천 시스템

**Status**: Approved
**Created**: 2026-03-07

## 1. User Stories & Scenarios

### US-1: 사용자 행동 데이터 수집 (P1)

서비스가 추천을 제공하려면 먼저 사용자의 행동(조회, 클릭, 구매, 장바구니 담기 등)을 수집하고 저장해야 한다. 외부 서비스에서 REST API를 통해 행동 이벤트를 전송한다.

**Why this priority**: 추천의 기반이 되는 데이터가 없으면 어떤 추천도 불가능하다.
**Independent Test**: 행동 이벤트를 API로 전송하고 저장된 데이터를 조회하여 검증한다.

**Acceptance Scenarios**:
1. **Given** 유효한 사용자 ID와 상품 ID가 있을 때, **When** 행동 이벤트(view, click, purchase, cart)를 POST API로 전송하면, **Then** 이벤트가 저장되고 201 응답을 반환한다.
2. **Given** 잘못된 형식의 이벤트 데이터가 전송되었을 때, **When** POST API를 호출하면, **Then** 400 에러와 유효성 검증 메시지를 반환한다.
3. **Given** 동일 사용자의 동일 상품에 대한 중복 이벤트가 짧은 시간 내에 발생했을 때, **When** POST API를 호출하면, **Then** 중복 이벤트도 타임스탬프와 함께 개별적으로 저장된다.

---

### US-2: 상품 메타데이터 관리 (P1)

추천 결과에 상품 정보를 포함하려면 상품 메타데이터(이름, 카테고리, 가격 등)를 등록하고 관리할 수 있어야 한다.

**Why this priority**: 추천 결과에 어떤 상품인지 정보가 없으면 실용성이 없다.
**Independent Test**: 상품 CRUD API를 호출하여 상품 등록/조회/수정/삭제를 검증한다.

**Acceptance Scenarios**:
1. **Given** 관리자가 상품 정보(이름, 카테고리, 가격, 이미지 URL)를 준비했을 때, **When** POST /api/products를 호출하면, **Then** 상품이 등록되고 201 응답과 상품 ID를 반환한다.
2. **Given** 등록된 상품이 존재할 때, **When** GET /api/products/{id}를 호출하면, **Then** 해당 상품의 전체 메타데이터를 반환한다.
3. **Given** 존재하지 않는 상품 ID로 조회할 때, **When** GET /api/products/{id}를 호출하면, **Then** 404 에러를 반환한다.

---

### US-3: Collaborative Filtering 추천 모델 학습 (P1)

수집된 사용자 행동 데이터를 기반으로 Collaborative Filtering 모델을 학습하여 사용자-상품 간 선호도를 예측할 수 있어야 한다.

**Why this priority**: 추천 시스템의 핵심 엔진으로, 이것 없이는 추천 기능 자체가 불가능하다.
**Independent Test**: 학습 데이터를 주입한 후 모델을 학습시키고, 특정 사용자에 대한 예측 점수가 합리적인지 검증한다.

**Acceptance Scenarios**:
1. **Given** 최소 10명의 사용자와 20개의 상품에 대한 행동 데이터가 존재할 때, **When** 모델 학습을 실행하면, **Then** 학습이 완료되고 모델 아티팩트가 저장된다.
2. **Given** 학습된 모델이 존재할 때, **When** 특정 사용자에 대한 추천 점수를 요청하면, **Then** 상품별 예측 선호도 점수를 반환한다.
3. **Given** 행동 데이터가 충분하지 않은 사용자(cold start)일 때, **When** 추천 점수를 요청하면, **Then** 인기도 기반 폴백 추천을 반환한다.

---

### US-4: 추천 결과 REST API 제공 (P1)

외부 서비스에서 특정 사용자에 대한 상품 추천 목록을 REST API로 요청하고 응답받을 수 있어야 한다.

**Why this priority**: 추천 시스템의 최종 산출물이자 외부 서비스와의 유일한 인터페이스이다.
**Independent Test**: API를 호출하여 추천 결과의 형식, 개수, 정렬 순서를 검증한다.

**Acceptance Scenarios**:
1. **Given** 학습된 모델과 사용자 행동 데이터가 존재할 때, **When** GET /api/recommendations/{user_id}?limit=10을 호출하면, **Then** 예측 선호도 순으로 정렬된 최대 10개의 추천 상품 목록(상품 ID, 이름, 카테고리, 추천 점수)을 반환한다.
2. **Given** 추천 결과가 존재할 때, **When** 카테고리 필터(category=electronics)를 추가하여 호출하면, **Then** 해당 카테고리의 추천 상품만 반환한다.
3. **Given** 이미 구매한 상품이 있을 때, **When** 추천 API를 호출하면, **Then** 이미 구매한 상품은 추천 목록에서 제외된다.

---

### US-5: 관리자 대시보드 - 추천 성능 지표 조회 (P1)

관리자가 웹 대시보드에서 추천 시스템의 성능 지표(CTR, 전환율, 커버리지, 다양성 등)를 시각적으로 확인할 수 있어야 한다.

**Why this priority**: 추천 시스템의 효과를 측정하지 않으면 개선할 수 없다. 비즈니스 가치 입증에 필수.
**Independent Test**: 대시보드에 접속하여 성능 지표 차트와 수치가 올바르게 표시되는지 확인한다.

**Acceptance Scenarios**:
1. **Given** 추천 이력과 사용자 반응 데이터가 존재할 때, **When** 관리자가 대시보드에 접속하면, **Then** CTR(클릭률), 전환율, 추천 커버리지, 카탈로그 다양성 지표를 차트로 표시한다.
2. **Given** 대시보드가 표시되었을 때, **When** 날짜 범위 필터를 변경하면, **Then** 해당 기간의 지표로 차트가 갱신된다.
3. **Given** 아직 데이터가 없는 초기 상태일 때, **When** 대시보드에 접속하면, **Then** "데이터 수집 중" 안내 메시지와 함께 빈 차트를 표시한다.

---

### US-6: 관리자 대시보드 - 모델 관리 (P2)

관리자가 대시보드에서 추천 모델의 학습 이력을 확인하고, 수동으로 재학습을 트리거할 수 있어야 한다.

**Why this priority**: 모델 관리 기능은 운영 안정성을 높이지만, 핵심 추천 기능 이후에 구현해도 무방하다.
**Independent Test**: 모델 학습 이력 목록 조회와 수동 학습 트리거를 검증한다.

**Acceptance Scenarios**:
1. **Given** 이전 학습 이력이 존재할 때, **When** 관리자가 모델 관리 페이지에 접속하면, **Then** 학습 이력(날짜, 데이터 건수, 모델 성능 지표, 상태)을 테이블로 표시한다.
2. **Given** 모델 관리 페이지에서 "재학습" 버튼을 클릭했을 때, **When** 확인 다이얼로그에서 확인을 누르면, **Then** 배경에서 모델 재학습이 시작되고 진행 상태를 표시한다.

---

### US-7: 추천 결과 A/B 테스트 지원 (P2)

관리자가 서로 다른 추천 전략(모델 버전, 파라미터)을 비교 실험할 수 있어야 한다.

**Why this priority**: 추천 품질 개선을 위한 실험 기능은 중요하지만 초기 출시 이후에 구현 가능하다.
**Independent Test**: A/B 테스트 그룹 설정 후 각 그룹에 다른 추천 전략이 적용되는지 검증한다.

**Acceptance Scenarios**:
1. **Given** 관리자가 A/B 테스트 설정 페이지에서 실험을 생성할 때, **When** 대조군/실험군 비율과 각 그룹의 추천 전략을 설정하면, **Then** 실험이 생성되고 활성화된다.
2. **Given** 활성화된 A/B 테스트가 있을 때, **When** 추천 API가 호출되면, **Then** 사용자가 할당된 실험 그룹에 따라 다른 추천 전략이 적용된다.
3. **Given** A/B 테스트 결과가 충분히 쌓였을 때, **When** 관리자가 결과 페이지를 조회하면, **Then** 그룹별 성능 지표 비교와 통계적 유의미성을 표시한다.

---

### US-8: 실시간 추천 갱신 (P3)

사용자의 최신 행동을 즉시 반영하여 추천 결과를 실시간으로 갱신할 수 있어야 한다.

**Why this priority**: 실시간 갱신은 추천 품질을 높이지만 배치 기반으로도 충분히 동작하므로 후순위.
**Independent Test**: 사용자가 행동 후 즉시 추천 API를 호출하여 결과 변화를 확인한다.

**Acceptance Scenarios**:
1. **Given** 사용자가 특정 카테고리 상품을 연속 조회했을 때, **When** 즉시 추천 API를 호출하면, **Then** 해당 카테고리 상품의 추천 점수가 상승한 결과를 반환한다.

---

### US-9: 추천 사유 설명 (P3)

추천 결과에 왜 해당 상품이 추천되었는지 설명을 포함할 수 있어야 한다.

**Why this priority**: 추천 설명은 사용자 신뢰도를 높이지만, 핵심 기능은 아니다.
**Independent Test**: 추천 API 응답에 추천 사유 필드가 포함되는지 확인한다.

**Acceptance Scenarios**:
1. **Given** 추천 결과가 존재할 때, **When** GET /api/recommendations/{user_id}?include_reason=true를 호출하면, **Then** 각 추천 상품에 추천 사유(예: "이 상품을 구매한 다른 사용자들이 함께 구매한 상품입니다")가 포함된다.

---

## 2. Functional Requirements

- **FR-001**: 시스템은 사용자 행동 이벤트(view, click, purchase, cart)를 REST API로 수집하고 영구 저장해야 한다
- **FR-002**: 시스템은 상품 메타데이터(이름, 카테고리, 가격, 이미지 URL, 설명)를 CRUD API로 관리할 수 있어야 한다
- **FR-003**: 시스템은 Collaborative Filtering(User-based 및/또는 Item-based) 알고리즘으로 추천 모델을 학습해야 한다
- **FR-004**: 시스템은 학습된 모델을 기반으로 사용자별 개인화된 추천 상품 목록을 REST API로 제공해야 한다
- **FR-005**: 시스템은 Cold Start 사용자(행동 이력이 없거나 적은 사용자)에게 인기도 기반 폴백 추천을 제공해야 한다
- **FR-006**: 시스템은 이미 구매한 상품을 추천 목록에서 제외해야 한다
- **FR-007**: 시스템은 추천 성능 지표(CTR, 전환율, 커버리지, 다양성)를 계산하고 관리자 대시보드에 시각화해야 한다
- **FR-008**: 관리자 대시보드는 날짜 범위 필터로 지표를 조회할 수 있어야 한다
- **FR-009**: 시스템은 모델 학습 이력(날짜, 데이터 건수, 성능 지표, 상태)을 기록하고 조회할 수 있어야 한다
- **FR-010**: 시스템은 API 인증(API Key 기반)으로 비인가 접근을 차단해야 한다
- **FR-011**: 추천 API 응답에는 상품 ID, 이름, 카테고리, 추천 점수가 포함되어야 한다
- **FR-012**: 추천 API는 카테고리 필터링을 지원해야 한다

## 3. Edge Cases

- **EC-001**: Cold Start — 행동 이력이 없는 신규 사용자 → 인기도 기반 폴백 추천 제공
- **EC-002**: 상품 부족 — 등록된 상품이 요청된 추천 개수보다 적을 때 → 가능한 만큼만 반환하고 실제 반환 개수를 응답에 포함
- **EC-003**: 모델 미학습 — 아직 학습된 모델이 없을 때 추천 API 호출 → 인기도 기반 폴백 추천 제공 + 응답 헤더에 폴백 여부 표시
- **EC-004**: 대량 동시 이벤트 — 동일 시점에 수천 건의 행동 이벤트가 유입될 때 → 비동기 처리 또는 큐잉으로 데이터 유실 방지
- **EC-005**: 상품 삭제 — 추천 목록에 포함된 상품이 삭제되었을 때 → 삭제된 상품은 추천 목록에서 자동 제외
- **EC-006**: 중복 행동 이벤트 — 네트워크 재시도로 동일 이벤트가 중복 전송될 때 → 멱등성 키를 지원하여 중복 저장 방지 가능 (선택적)
- **EC-007**: 모델 학습 실패 — 학습 중 오류 발생 → 이전 모델을 유지하고 학습 실패 상태를 기록

## 4. Key Entities

- **User**: 추천 대상 사용자. 외부 시스템의 사용자 ID를 참조. (user_id, 등록일)
- **Product**: 추천 대상 상품. 이름, 카테고리, 가격, 이미지 URL, 설명, 활성 상태. (product_id, name, category, price, image_url, description, is_active)
- **UserAction**: 사용자의 상품에 대한 행동 기록. 행동 유형(view, click, purchase, cart), 타임스탬프. (action_id, user_id, product_id, action_type, timestamp)
- **RecommendationModel**: 학습된 추천 모델 아티팩트. 버전, 학습 일시, 성능 지표, 상태. (model_id, version, trained_at, metrics, status, artifact_path)
- **RecommendationLog**: 추천 API 호출 기록. 어떤 사용자에게 어떤 상품을 추천했는지와 사용자 반응(클릭/구매 여부). (log_id, user_id, recommended_products, clicked_product_id, converted, timestamp)
- **PerformanceMetric**: 추천 성능 지표 집계 데이터. 일별 CTR, 전환율, 커버리지, 다양성. (metric_id, date, ctr, conversion_rate, coverage, diversity)

## 5. Success Criteria (기술 무관, 측정 가능)

- **SC-001**: 추천 API 응답 시간이 95퍼센타일 기준 200ms 이내여야 한다
- **SC-002**: 행동 이벤트 수집 API는 초당 1000건 이상의 이벤트를 처리할 수 있어야 한다
- **SC-003**: 추천 모델 학습이 100만 건의 행동 데이터 기준 30분 이내에 완료되어야 한다
- **SC-004**: 관리자 대시보드의 성능 지표 페이지가 3초 이내에 로드되어야 한다
- **SC-005**: Cold Start 사용자에게도 최소 5개 이상의 추천 결과를 제공해야 한다
- **SC-006**: 추천 커버리지(전체 상품 중 추천에 포함되는 비율)가 30% 이상이어야 한다
- **SC-007**: 시스템 가용성 99.5% 이상이어야 한다
