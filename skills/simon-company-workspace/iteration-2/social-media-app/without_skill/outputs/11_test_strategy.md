# 테스트 전략

## 1. 테스트 피라미드

```
            ┌────────────┐
            │   E2E      │  10% - 핵심 사용자 플로우
            │ (Playwright)│
           ┌┴────────────┴┐
           │  통합 테스트    │  30% - API 엔드포인트, 서비스 간 연동
           │ (Supertest)   │
          ┌┴──────────────┴┐
          │   단위 테스트     │  60% - 서비스 로직, 유틸, 컴포넌트
          │ (Jest/Vitest)   │
          └────────────────┘
```

## 2. 테스트 유형별 전략

### 2.1 단위 테스트

**도구**: Vitest (Jest 호환, Vite 기반 빠른 실행)

**대상 및 커버리지 목표**:

| 대상 | 커버리지 목표 | 예시 |
|------|---------------|------|
| 서비스 레이어 | >= 90% | AuthService, PostService, FeedService |
| 유틸리티 함수 | >= 95% | validators, formatters, helpers |
| React 컴포넌트 | >= 80% | FeedPost, LikeButton, CommentSection |
| 커스텀 훅 | >= 85% | useAuth, useFeed, useLike |

**작성 원칙**:
- TDD 필수: Red -> Green -> Refactor
- 각 테스트는 독립적이어야 함 (격리)
- mock은 외부 의존성(DB, S3, Redis)에만 사용
- 테스트명은 한국어로 작성 ("정상적인 이메일로 회원가입하면 사용자가 생성된다")

**서비스 레이어 테스트 예시 구조**:
```
describe('AuthService', () => {
  describe('signup', () => {
    it('정상적인 입력으로 회원가입하면 사용자와 토큰이 생성된다')
    it('이미 존재하는 이메일이면 ConflictError를 던진다')
    it('이미 존재하는 사용자명이면 ConflictError를 던진다')
    it('비밀번호 정책을 충족하지 않으면 ValidationError를 던진다')
  })
  describe('login', () => {
    it('올바른 이메일과 비밀번호로 로그인하면 토큰이 반환된다')
    it('존재하지 않는 이메일이면 AuthenticationError를 던진다')
    it('잘못된 비밀번호이면 AuthenticationError를 던진다')
  })
})
```

**컴포넌트 테스트 예시 구조**:
```
describe('LikeButton', () => {
  it('좋아요하지 않은 상태에서 빈 하트 아이콘을 표시한다')
  it('좋아요한 상태에서 채워진 하트 아이콘을 표시한다')
  it('클릭하면 좋아요 API를 호출한다')
  it('좋아요 수가 업데이트된다')
  it('API 호출 실패 시 이전 상태로 롤백한다 (낙관적 업데이트)')
})
```

### 2.2 통합 테스트

**도구**: Supertest (API), Prisma (테스트 DB)

**대상**: 모든 API 엔드포인트

**환경**:
- 테스트용 PostgreSQL (Docker)
- 테스트용 Redis (Docker)
- S3는 mock (localstack 또는 mock)
- 각 테스트 스위트 전에 DB 시딩, 후에 클린업

**테스트 케이스 목록**:

| API | 테스트 케이스 수 | 주요 테스트 |
|-----|-----------------|-------------|
| POST /auth/signup | 5 | 성공, 이메일 중복, 사용자명 중복, 유효성 검사 |
| POST /auth/login | 4 | 성공, 잘못된 이메일, 잘못된 비밀번호, 비활성 계정 |
| POST /auth/logout | 2 | 성공, 유효하지 않은 토큰 |
| GET /users/:username | 3 | 성공, 존재하지 않는 사용자, 삭제된 사용자 |
| PATCH /users/me | 5 | 성공, 사용자명 중복, 프로필 이미지, 사용자명 변경 제한 |
| POST /posts | 4 | 성공, 이미지 없음, 파일 크기 초과, 잘못된 형식 |
| GET /posts/:id | 3 | 성공, 존재하지 않는 게시물, 삭제된 게시물 |
| DELETE /posts/:id | 3 | 성공, 권한 없음, 존재하지 않는 게시물 |
| GET /feed | 4 | 성공, 빈 피드, 커서 페이지네이션, 삭제된 게시물 제외 |
| POST /users/:username/follow | 4 | 성공, 자기 팔로우, 이미 팔로우, 존재하지 않는 사용자 |
| POST /posts/:id/like | 3 | 성공, 중복 좋아요, 존재하지 않는 게시물 |
| POST /posts/:id/comments | 3 | 성공, 빈 댓글, 글자 수 초과 |
| GET /notifications | 3 | 성공, 빈 목록, 페이지네이션 |
| GET /search/users | 3 | 성공, 빈 결과, 특수문자 처리 |

### 2.3 E2E 테스트

**도구**: Playwright

**환경**: 스테이징 환경 또는 로컬 Docker Compose

**핵심 사용자 플로우**:

| 플로우 ID | 시나리오 | 우선순위 |
|-----------|----------|----------|
| E2E-001 | 회원가입 -> 로그인 -> 프로필 확인 | P0 |
| E2E-002 | 로그인 -> 게시물 작성 -> 피드에서 확인 | P0 |
| E2E-003 | 로그인 -> 사용자 검색 -> 팔로우 -> 피드에서 게시물 확인 | P0 |
| E2E-004 | 로그인 -> 게시물 좋아요 -> 좋아요 취소 | P0 |
| E2E-005 | 로그인 -> 게시물에 댓글 작성 -> 댓글 삭제 | P0 |
| E2E-006 | 로그인 -> 알림 확인 -> 알림 읽음 처리 | P1 |
| E2E-007 | 로그인 -> 프로필 편집 -> 변경 확인 | P1 |
| E2E-008 | 로그인 -> 게시물 삭제 -> 피드에서 제거 확인 | P1 |
| E2E-009 | 비로그인 상태에서 보호된 페이지 접근 -> 로그인 리다이렉트 | P0 |
| E2E-010 | 피드 무한 스크롤 동작 확인 | P1 |

## 3. 테스트 자동화 및 CI/CD 연동

```
┌──────────────┐     ┌───────────────────────┐     ┌──────────────┐
│  코드 푸시    │────>│   GitHub Actions CI    │────>│  결과 리포트   │
│  (PR 생성)   │     │                       │     │              │
└──────────────┘     │  1. lint (ESLint)     │     │  - 테스트 결과│
                     │  2. type check (tsc)  │     │  - 커버리지   │
                     │  3. 단위 테스트         │     │  - PR 코멘트  │
                     │  4. 통합 테스트         │     │              │
                     │  5. 커버리지 리포트      │     └──────────────┘
                     │  6. E2E (main만)       │
                     └───────────────────────┘
```

### CI 파이프라인 단계:

| 단계 | 실행 조건 | 실패 시 |
|------|-----------|---------|
| Lint | 모든 PR | PR 머지 차단 |
| Type Check | 모든 PR | PR 머지 차단 |
| 단위 테스트 | 모든 PR | PR 머지 차단 |
| 통합 테스트 | 모든 PR | PR 머지 차단 |
| 커버리지 체크 (>= 80%) | 모든 PR | PR 머지 차단 |
| E2E 테스트 | main/develop 머지 시 | 배포 차단 |
| 성능 테스트 | 수동 / 주간 | 알럿 발송 |

## 4. 테스트 데이터 관리

### 시드 데이터 구성:

| 데이터 | 수량 | 용도 |
|--------|------|------|
| 사용자 | 10명 | 다양한 상태 (활성/비활성, 팔로우 관계) |
| 게시물 | 50개 | 피드 테스트, 페이지네이션 |
| 댓글 | 100개 | 댓글 목록 테스트 |
| 좋아요 | 200개 | 좋아요 수 계산, 탐색 피드 |
| 팔로우 | 30개 | 피드 필터링, 팔로워 목록 |
| 알림 | 50개 | 알림 목록, 읽음 처리 |

### 테스트 사용자:

| 사용자명 | 역할 | 특징 |
|----------|------|------|
| test_user_main | 주 테스트 사용자 | 게시물 10개, 팔로잉 5명, 팔로워 3명 |
| test_user_empty | 빈 프로필 사용자 | 게시물 0, 팔로잉 0 |
| test_user_popular | 인기 사용자 | 게시물 20개, 팔로워 100명 |
| test_user_inactive | 비활성 사용자 | is_active = false |
| test_user_deleted | 삭제된 사용자 | deleted_at 설정 |
