# Spec: Picstory - 소셜 미디어 웹앱

**Status**: Approved
**Created**: 2026-03-07

## 1. User Stories & Scenarios

### US-1: 회원가입 및 로그인 (P1)

사용자는 이메일과 비밀번호로 계정을 생성하고, 로그인하여 서비스를 이용할 수 있어야 한다.

**Why this priority**: 모든 기능의 전제 조건. 인증 없이는 사진 업로드, 팔로우, 좋아요 등 핵심 기능 사용 불가.
**Independent Test**: 회원가입 후 로그인하여 인증된 세션 확인.

**Acceptance Scenarios**:
1. **Given** 미가입 사용자가 회원가입 페이지에 접근, **When** 유효한 이메일/비밀번호/사용자명을 입력하고 가입 버튼 클릭, **Then** 계정이 생성되고 로그인 상태로 메인 피드 페이지로 이동
2. **Given** 가입된 사용자가 로그인 페이지에 접근, **When** 올바른 이메일/비밀번호 입력 후 로그인 클릭, **Then** 인증 성공하고 메인 피드 페이지로 이동
3. **Given** 사용자가 잘못된 비밀번호로 로그인 시도, **When** 로그인 버튼 클릭, **Then** "이메일 또는 비밀번호가 올바르지 않습니다" 에러 메시지 표시
4. **Given** 이미 가입된 이메일로 회원가입 시도, **When** 가입 버튼 클릭, **Then** "이미 사용 중인 이메일입니다" 에러 메시지 표시

---

### US-2: 프로필 관리 (P1)

사용자는 자신의 프로필(사용자명, 소개글, 프로필 사진)을 설정하고 수정할 수 있어야 한다.

**Why this priority**: 소셜 미디어의 기본 정체성. 다른 사용자가 팔로우 여부를 판단하는 핵심 정보.
**Independent Test**: 프로필 수정 후 변경된 정보 확인.

**Acceptance Scenarios**:
1. **Given** 로그인한 사용자가 프로필 페이지 접근, **When** 소개글을 수정하고 저장 버튼 클릭, **Then** 소개글이 업데이트되고 성공 메시지 표시
2. **Given** 로그인한 사용자, **When** 프로필 사진 업로드, **Then** 프로필 사진이 변경되고 모든 게시물/댓글에 새 사진이 반영
3. **Given** 다른 사용자의 프로필 페이지 접근, **When** 프로필 페이지 로드, **Then** 해당 사용자의 사용자명, 소개글, 프로필 사진, 게시물 수, 팔로워/팔로잉 수 표시

---

### US-3: 사진 게시물 작성 (P1)

사용자는 사진과 함께 캡션을 작성하여 게시물을 올릴 수 있어야 한다.

**Why this priority**: 소셜 미디어의 핵심 콘텐츠 생산 기능. 피드에 표시할 콘텐츠의 원천.
**Independent Test**: 사진 업로드 후 피드에서 게시물 확인.

**Acceptance Scenarios**:
1. **Given** 로그인한 사용자가 게시물 작성 페이지 접근, **When** 사진을 선택하고 캡션을 입력한 후 게시 버튼 클릭, **Then** 게시물이 생성되고 사용자의 피드와 프로필에 표시
2. **Given** 사용자가 사진 없이 게시 시도, **When** 게시 버튼 클릭, **Then** "사진을 선택해주세요" 에러 메시지 표시
3. **Given** 사용자가 지원하지 않는 파일 형식(예: PDF) 업로드 시도, **When** 파일 선택, **Then** "지원하지 않는 파일 형식입니다. JPG, PNG, GIF만 가능합니다" 메시지 표시
4. **Given** 사용자가 캡션 없이 사진만 게시 시도, **When** 게시 버튼 클릭, **Then** 캡션 없이 사진만 포함된 게시물 생성 성공

---

### US-4: 피드 조회 (P1)

사용자는 팔로우하는 사용자들의 게시물을 최신순으로 정렬된 피드에서 볼 수 있어야 한다.

**Why this priority**: 소셜 미디어의 핵심 소비 기능. 콘텐츠 탐색의 주요 진입점.
**Independent Test**: 팔로우한 사용자의 게시물이 최신순으로 피드에 표시되는지 확인.

**Acceptance Scenarios**:
1. **Given** 로그인한 사용자가 메인 피드 페이지 접근, **When** 페이지 로드, **Then** 팔로우한 사용자들의 게시물이 최신순으로 표시 (자신의 게시물 포함)
2. **Given** 피드에 게시물이 많을 때, **When** 아래로 스크롤, **Then** 다음 페이지의 게시물이 무한 스크롤로 로드
3. **Given** 아무도 팔로우하지 않은 사용자, **When** 피드 페이지 접근, **Then** "아직 팔로우한 사용자가 없습니다. 사용자를 검색하고 팔로우해보세요!" 안내 메시지 표시
4. **Given** 피드에 게시물 표시, **When** 각 게시물 카드 확인, **Then** 작성자 프로필 사진, 사용자명, 사진, 캡션, 좋아요 수, 댓글 수, 작성 시간 표시

---

### US-5: 팔로우/언팔로우 (P1)

사용자는 다른 사용자를 팔로우하거나 언팔로우할 수 있어야 한다.

**Why this priority**: 피드 구성의 기반. 팔로우 관계가 없으면 피드가 비어 있음.
**Independent Test**: 팔로우 후 상대방 프로필에 팔로워 수 증가 및 내 피드에 해당 사용자 게시물 표시.

**Acceptance Scenarios**:
1. **Given** 로그인한 사용자가 다른 사용자의 프로필 방문, **When** 팔로우 버튼 클릭, **Then** 팔로우 관계 생성, 버튼이 "팔로잉"으로 변경, 상대 팔로워 수 +1
2. **Given** 이미 팔로우 중인 사용자의 프로필 방문, **When** 팔로잉 버튼 클릭, **Then** 팔로우 해제 확인 모달 표시 후 확인 시 언팔로우, 버튼이 "팔로우"로 변경
3. **Given** 사용자가 자기 자신의 프로필 방문, **When** 프로필 로드, **Then** 팔로우 버튼이 표시되지 않음

---

### US-6: 좋아요 (P1)

사용자는 게시물에 좋아요를 누르거나 취소할 수 있어야 한다.

**Why this priority**: 소셜 미디어 핵심 인터랙션. 콘텐츠에 대한 피드백 메커니즘.
**Independent Test**: 게시물에 좋아요 후 좋아요 수 증가 확인.

**Acceptance Scenarios**:
1. **Given** 피드에서 게시물을 보는 사용자, **When** 좋아요 버튼(하트) 클릭, **Then** 하트가 채워진 상태로 변경, 좋아요 수 +1
2. **Given** 이미 좋아요한 게시물, **When** 좋아요 버튼 다시 클릭, **Then** 하트가 빈 상태로 변경, 좋아요 수 -1
3. **Given** 게시물의 좋아요 수 표시, **When** 좋아요 수 클릭, **Then** 좋아요한 사용자 목록 표시

---

### US-7: 댓글 (P1)

사용자는 게시물에 댓글을 작성하고 삭제할 수 있어야 한다.

**Why this priority**: 소셜 미디어 핵심 소통 기능. 사용자 간 대화와 커뮤니티 형성.
**Independent Test**: 댓글 작성 후 게시물에 댓글 표시 확인.

**Acceptance Scenarios**:
1. **Given** 게시물 상세 페이지 또는 피드에서 게시물 확인, **When** 댓글 입력란에 텍스트 입력 후 게시 버튼 클릭, **Then** 댓글이 게시물 하단에 추가되고 댓글 수 +1
2. **Given** 자신이 작성한 댓글, **When** 삭제 버튼 클릭, **Then** 삭제 확인 후 댓글 제거, 댓글 수 -1
3. **Given** 빈 텍스트로 댓글 게시 시도, **When** 게시 버튼 클릭, **Then** 게시 버튼 비활성화 상태 유지

---

### US-8: 알림 (P1)

사용자는 자신의 게시물에 대한 좋아요, 댓글, 새 팔로워 알림을 받을 수 있어야 한다.

**Why this priority**: 사용자 재방문과 참여를 유도하는 핵심 기능. 요구사항에 명시됨.
**Independent Test**: 다른 사용자가 좋아요/댓글/팔로우 시 알림 목록에서 확인.

**Acceptance Scenarios**:
1. **Given** 사용자 A가 사용자 B의 게시물에 좋아요, **When** 사용자 B가 알림 페이지 접근, **Then** "사용자 A님이 회원님의 게시물을 좋아합니다" 알림 표시
2. **Given** 사용자 A가 사용자 B의 게시물에 댓글 작성, **When** 사용자 B가 알림 페이지 접근, **Then** "사용자 A님이 댓글을 남겼습니다: [댓글 미리보기]" 알림 표시
3. **Given** 사용자 A가 사용자 B를 팔로우, **When** 사용자 B가 알림 페이지 접근, **Then** "사용자 A님이 회원님을 팔로우합니다" 알림 표시
4. **Given** 읽지 않은 알림이 있는 사용자, **When** 네비게이션 바 확인, **Then** 알림 아이콘에 읽지 않은 알림 수 배지 표시
5. **Given** 알림 목록 확인, **When** 알림 클릭, **Then** 해당 게시물/프로필로 이동하고 알림이 읽음 처리

---

### US-9: 사용자 검색 (P2)

사용자는 다른 사용자를 사용자명으로 검색할 수 있어야 한다.

**Why this priority**: 팔로우 기능의 진입점이지만, 직접 URL 접근이나 추천 등으로 우회 가능.
**Independent Test**: 사용자명 입력 후 검색 결과에서 해당 사용자 확인.

**Acceptance Scenarios**:
1. **Given** 로그인한 사용자가 검색 페이지 접근, **When** 사용자명 일부 입력, **Then** 실시간으로 매칭되는 사용자 목록 표시
2. **Given** 검색 결과에서 사용자 클릭, **When** 사용자 프로필 페이지 로드, **Then** 해당 사용자의 프로필 정보와 게시물 표시
3. **Given** 존재하지 않는 사용자명 검색, **When** 검색 실행, **Then** "검색 결과가 없습니다" 메시지 표시

---

### US-10: 게시물 상세 페이지 (P2)

사용자는 개별 게시물의 상세 페이지에서 사진을 크게 보고 모든 댓글을 확인할 수 있어야 한다.

**Why this priority**: 피드에서 미리보기로 충분히 동작하지만, 전체 댓글 목록과 상세 정보를 위해 필요.
**Independent Test**: 피드에서 게시물 클릭 후 상세 페이지에서 전체 댓글 목록 확인.

**Acceptance Scenarios**:
1. **Given** 피드에서 게시물 확인, **When** 게시물 클릭, **Then** 상세 페이지로 이동하여 사진, 캡션, 전체 댓글 목록, 좋아요 수, 작성 시간 표시
2. **Given** 게시물 상세 페이지, **When** 댓글 작성, **Then** 댓글이 실시간으로 목록에 추가

---

### US-11: 게시물 삭제 (P2)

사용자는 자신의 게시물을 삭제할 수 있어야 한다.

**Why this priority**: 콘텐츠 관리 기능. 핵심은 아니지만 사용자 자기 콘텐츠 제어에 필요.
**Independent Test**: 게시물 삭제 후 피드와 프로필에서 제거 확인.

**Acceptance Scenarios**:
1. **Given** 자신의 게시물 상세 페이지, **When** 삭제 버튼 클릭, **Then** 삭제 확인 모달 표시
2. **Given** 삭제 확인 모달에서 확인 클릭, **When** 삭제 실행, **Then** 게시물이 피드, 프로필, DB에서 제거되고 관련 좋아요/댓글/알림도 삭제
3. **Given** 다른 사용자의 게시물, **When** 게시물 상세 확인, **Then** 삭제 버튼 표시되지 않음

---

### US-12: 탐색 페이지 (P3)

사용자는 팔로우하지 않는 사용자의 인기 게시물을 탐색할 수 있어야 한다.

**Why this priority**: 새 콘텐츠 발견 채널. MVP 없이도 서비스 동작 가능.
**Independent Test**: 탐색 페이지에서 팔로우하지 않은 사용자의 게시물 표시 확인.

**Acceptance Scenarios**:
1. **Given** 로그인한 사용자가 탐색 페이지 접근, **When** 페이지 로드, **Then** 최근 인기 게시물(좋아요 수 기준) 그리드 형태로 표시
2. **Given** 탐색 페이지의 게시물 클릭, **When** 게시물 상세로 이동, **Then** 해당 게시물 상세 정보 표시

---

### US-13: 로그아웃 (P1)

사용자는 로그아웃하여 세션을 종료할 수 있어야 한다.

**Why this priority**: 기본 보안 기능. 공유 기기 사용 시 필수.
**Independent Test**: 로그아웃 후 인증 필요 페이지 접근 시 로그인 페이지로 리다이렉트.

**Acceptance Scenarios**:
1. **Given** 로그인 상태의 사용자, **When** 로그아웃 버튼 클릭, **Then** 세션 종료 후 로그인 페이지로 이동
2. **Given** 로그아웃한 사용자, **When** 피드 페이지 URL 직접 접근, **Then** 로그인 페이지로 리다이렉트

---

## 2. Functional Requirements

- **FR-001**: 시스템은 이메일/비밀번호 기반 회원가입 및 로그인을 지원해야 한다
- **FR-002**: 시스템은 사용자 프로필(사용자명, 소개글, 프로필 사진)을 관리할 수 있어야 한다
- **FR-003**: 시스템은 사진 파일(JPG, PNG, GIF)을 업로드하고 저장할 수 있어야 한다
- **FR-004**: 시스템은 사진과 캡션으로 구성된 게시물을 생성/삭제할 수 있어야 한다
- **FR-005**: 시스템은 팔로우한 사용자의 게시물을 최신순으로 정렬하여 피드를 제공해야 한다
- **FR-006**: 시스템은 사용자 간 팔로우/언팔로우 관계를 관리해야 한다
- **FR-007**: 시스템은 게시물에 대한 좋아요 토글(추가/취소)을 지원해야 한다
- **FR-008**: 시스템은 게시물에 대한 댓글 작성/삭제를 지원해야 한다
- **FR-009**: 시스템은 좋아요, 댓글, 팔로우 이벤트에 대한 알림을 생성하고 표시해야 한다
- **FR-010**: 시스템은 알림의 읽음/안읽음 상태를 관리해야 한다
- **FR-011**: 시스템은 사용자명 기반 사용자 검색을 지원해야 한다
- **FR-012**: 시스템은 무한 스크롤 기반 페이지네이션을 지원해야 한다
- **FR-013**: 시스템은 인증되지 않은 사용자의 접근을 차단해야 한다
- **FR-014**: 업로드 가능한 이미지 크기는 최대 10MB로 제한해야 한다

## 3. Edge Cases

- **EC-001**: 동시에 여러 사용자가 같은 게시물에 좋아요를 누를 때 좋아요 수의 정합성 보장
- **EC-002**: 사용자가 삭제된 게시물에 대한 알림을 클릭할 때의 처리 (404 대신 "삭제된 게시물" 안내)
- **EC-003**: 팔로우한 사용자가 계정을 삭제한 경우 피드에서의 처리
- **EC-004**: 매우 긴 캡션이나 댓글 입력 시 길이 제한 (캡션 2200자, 댓글 1000자)
- **EC-005**: 동일 사용자가 같은 게시물에 중복 좋아요를 보내는 경우 (토글 처리)
- **EC-006**: 네트워크 오류로 이미지 업로드가 중단된 경우의 복구 처리
- **EC-007**: 프로필 사진 없는 사용자의 기본 아바타 표시
- **EC-008**: 자기 자신을 팔로우하려는 시도 차단

## 4. Key Entities

- **User**: 서비스 사용자. 이메일, 사용자명, 소개글, 프로필 사진을 가짐. 다른 사용자를 팔로우하고, 게시물을 작성함.
- **Post**: 사진과 캡션으로 구성된 게시물. 한 명의 작성자(User)가 있으며, 여러 좋아요와 댓글을 가질 수 있음.
- **Follow**: 두 사용자 간의 팔로우 관계. 팔로워(follower)와 팔로잉(following) 방향성 있음.
- **Like**: 사용자가 게시물에 남긴 좋아요. 한 사용자당 하나의 게시물에 하나만 가능.
- **Comment**: 사용자가 게시물에 남긴 텍스트 댓글. 한 게시물에 여러 개 가능.
- **Notification**: 좋아요/댓글/팔로우 이벤트로 생성되는 알림. 수신자, 발신자, 이벤트 유형, 관련 게시물 참조.

## 5. Success Criteria

- **SC-001**: 신규 사용자가 가입 후 3분 이내에 첫 게시물을 작성할 수 있어야 한다
- **SC-002**: 피드 로딩 시 첫 10개 게시물이 2초 이내에 표시되어야 한다
- **SC-003**: 좋아요/댓글 액션 후 1초 이내에 UI 반영이 되어야 한다
- **SC-004**: 알림이 이벤트 발생 후 5초 이내에 생성되어야 한다
- **SC-005**: 100명 동시 접속 환경에서 응답 지연 없이 동작해야 한다
- **SC-006**: 사진 업로드 후 5초 이내에 피드에서 확인 가능해야 한다
