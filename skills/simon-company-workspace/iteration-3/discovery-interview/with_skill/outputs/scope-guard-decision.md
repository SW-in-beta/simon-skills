# Scope Guard 판정

## 판정 기준 분석

| 기준 | simon-company 조건 | 현재 요청 판정 | 비고 |
|------|-------------------|---------------|------|
| 필요 팀 수 | 3개 이상 | **판단 불가 (의뢰 모드 선행 필요)** | 데이터 대시보드 + 고객 경험 개선이라면 PM, CTO, Design, Frontend, Backend, QA, DBA, DevOps (8팀) 가능. 단순 내부 대시보드라면 3-4팀으로도 가능 |
| 인프라 필요 | CI/CD, 배포, 모니터링 필요 | **가능성 높음** | 직원 + 고객 양쪽 서비스라면 배포/모니터링 필요 |
| 디자인 필요 | UI/UX 설계 필요 | **가능성 높음** | "직원들이 데이터를 쉽게 볼 수 있으면" = 대시보드 UI 필요, "고객 경험" = 고객용 UI 필요 |
| 예상 Feature 수 | 5개 이상 | **판단 불가** | 범위 미정 |

## Scope Guard 판정 결과

**결과: simon-company 유지 (잠정)**

### 판정 근거

1. **리다이렉트하지 않는 이유**: 요청에 "직원용 데이터 조회"와 "고객 경험 개선"이라는 두 가지 큰 축이 있어, 최종적으로 다중 팀 협업이 필요한 대규모 프로젝트가 될 가능성이 높다.

2. **잠정 판정인 이유**: 의뢰 모드 인터뷰를 통해 범위가 구체화된 후 Scope Guard를 재평가해야 한다. 인터뷰 결과 "내부 대시보드 하나만 만들면 된다"로 귀결되면, 그 시점에서 simon-bot-pm으로 리다이렉트를 제안할 수 있다.

3. **재평가 트리거**: Discovery Interview Stage 4 (우선순위 & 범위 조정) 완료 시점에서 필요 팀 수와 Feature 수를 기준으로 Scope Guard를 재실행한다.

[Decision] Scope Guard 잠정 유지 -- 요청이 불명확하여 의뢰 모드 인터뷰 후 재평가 필요. 두 사용자 그룹(직원/고객)을 대상으로 하므로 대규모 프로젝트 가능성 높음.
