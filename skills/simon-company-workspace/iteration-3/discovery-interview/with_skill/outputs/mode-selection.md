# 모드 선택 분석

## 사용자 요청 원문

> 우리 회사에서 고객 데이터를 많이 쌓고 있는데, 이걸 좀 더 잘 활용하고 싶어요. 직원들이 데이터를 쉽게 볼 수 있으면 좋겠고... 고객한테도 뭔가 더 좋은 경험을 줄 수 있을 것 같은데, 어떻게 하면 좋을까요? 구체적으로 뭘 만들어야 할지는 잘 모르겠어요.

## 요청 분석

### 불명확성 지표

| 지표 | 판정 | 근거 |
|------|------|------|
| 구체적인 기능 목록 있음 | X | 없음. "좀 더 잘 활용", "쉽게 볼 수 있으면" 등 추상적 표현만 있음 |
| 대상 사용자 명확 | 부분적 | "직원들", "고객" 두 그룹 언급했으나 역할/페르소나가 불명확 |
| 해결하려는 문제 명확 | X | "데이터 활용"이라는 광범위한 주제만 있음, 구체적 페인 포인트 불명 |
| 기대하는 결과물 형태 | X | "구체적으로 뭘 만들어야 할지는 잘 모르겠어요"라고 명시적으로 밝힘 |
| 기술적 요구사항 | X | 언급 없음 |
| 비즈니스 도메인 | 부분적 | "고객 데이터"를 쌓는 회사라는 것만 알 수 있음, 업종 불명 |

### 의뢰 모드 자동 감지 판정

**결과: 의뢰 모드 감지됨**

스킬의 Phase 0-C "의뢰 모드 자동 감지" 조건에 정확히 부합합니다:
- "우리 회사 데이터를 활용해서..." 패턴과 일치
- "이런 문제를 해결하고 싶어" 패턴과 일치
- 사용자 본인이 "구체적으로 뭘 만들어야 할지는 잘 모르겠어요"라고 명시

## 선택된 모드

**의뢰 모드 (Discovery Interview Protocol)** 선택

### 모드 선택 이유
1. 사용자가 아이디어 단계에 있으며 구체적인 기능 목록이 없음
2. "어떻게 하면 좋을까요?"라는 질문 자체가 방향 탐색을 요청하고 있음
3. 대상 사용자(직원 vs 고객)에 따라 완전히 다른 제품이 나올 수 있어, 심층 인터뷰 없이는 스펙 작성이 불가능
4. AI-First Draft로 초안을 작성하더라도 가정이 너무 많아 실질적 가치가 없음

### CEO 메시지 (사용자에게 전달할 내용)

> 요청을 분석해 보니, 아직 구체적인 기능이나 제품 형태가 정해지지 않은 아이디어 단계로 보입니다.
>
> 이런 경우 **의뢰 모드**를 추천드립니다. PM이 구조화된 인터뷰를 통해 요구사항을 함께 발견하고 구체화하는 방식입니다. 실제 소프트웨어 회사가 클라이언트의 프로젝트를 수주할 때와 같은 프로세스로, "데이터 활용"이라는 큰 그림에서 출발하여 실제로 만들어야 할 제품의 모습을 함께 정의해 나가겠습니다.
>
> **의뢰 모드로 진행할까요?**
>
> 참고로 다른 옵션도 있습니다:
> 1. **자동 진행**: 완료 즉시 다음 단계로 (중간 보고만)
> 2. **승인 후 진행**: 각 단계 결과를 보고하고 승인 후 진행
> 3. **의뢰 모드** (추천): PM이 인터뷰를 통해 요구사항을 함께 구체화합니다
