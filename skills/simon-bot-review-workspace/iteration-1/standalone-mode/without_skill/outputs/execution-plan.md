# 실행 계획: `feat/add-auth` 브랜치 코드 리뷰

## 개요

사용자가 `feat/add-auth` 브랜치에서 작업한 JWT 토큰 기반 인증 기능(미들웨어 + 로그인/로그아웃 API)을 리뷰하는 작업이다.

---

## Step 1: 브랜치 전환 및 변경 범위 파악

### 실행할 명령어 (병렬)

```bash
# 1-1. feat/add-auth 브랜치로 체크아웃
git checkout feat/add-auth

# 1-2. main 대비 커밋 히스토리 확인
git log main..feat/add-auth --oneline --no-merges

# 1-3. main 대비 변경된 파일 목록 확인
git diff main...feat/add-auth --stat

# 1-4. main 대비 전체 diff 확인
git diff main...feat/add-auth
```

### 목적

- 브랜치에 포함된 커밋 수와 각 커밋의 의도를 파악한다.
- 변경된 파일 목록과 변경량(추가/삭제 라인 수)을 확인하여 리뷰 범위를 정한다.
- 전체 diff를 통해 실제 코드 변경 내용을 확인한다.

---

## Step 2: 변경된 파일 개별 읽기

### 실행할 도구

`git diff --stat` 결과에서 나온 변경 파일들을 **Read 도구**로 병렬 읽기한다. 예상되는 파일들:

```
# 병렬로 Read 도구 호출 (파일 경로는 Step 1의 diff --stat 결과에 따라 결정)
Read: src/middleware/auth.ts (또는 .js)
Read: src/routes/auth.ts (또는 api/auth)
Read: src/utils/jwt.ts (토큰 유틸리티)
Read: src/types/auth.ts (타입 정의가 있다면)
Read: package.json (새 의존성 확인)
Read: .env.example (환경변수 변경 확인)
Read: 관련 테스트 파일들
```

### 목적

- diff만으로는 전체 맥락을 놓칠 수 있으므로 파일 전체를 읽어 구조를 파악한다.
- 새로 추가된 파일은 전체를, 수정된 파일은 변경 부분의 맥락을 함께 확인한다.

---

## Step 3: 관련 기존 코드 탐색

### 실행할 도구/명령어 (병렬)

```bash
# 3-1. 인증 미들웨어가 어디서 사용되는지 확인
Grep: pattern="auth" 또는 미들웨어 함수명으로 검색

# 3-2. 라우트 등록 방식 확인 (기존 패턴과 일치하는지)
Grep: pattern="router" 또는 "app.use" 등 라우트 등록 패턴

# 3-3. 기존 에러 핸들링 패턴 확인
Grep: pattern="error" 또는 "catch" 등 에러 처리 패턴

# 3-4. 환경 변수 사용 패턴 확인
Grep: pattern="process.env" 또는 "env" 관련 패턴
```

### 목적

- 새 코드가 기존 코드베이스의 패턴과 일관성을 유지하는지 확인한다.
- 미들웨어가 올바르게 적용되었는지, 라우트가 올바르게 등록되었는지 확인한다.

---

## Step 4: 테스트 코드 확인 및 실행

### 실행할 명령어

```bash
# 4-1. 테스트 파일 존재 확인
Glob: pattern="**/*auth*test*" 또는 "**/*auth*spec*"

# 4-2. 테스트 실행 (프로젝트 설정에 따라)
npm test -- --coverage --grep="auth" (또는 해당 테스트 파일 지정)
# 또는
npx jest --coverage src/**/*auth* (Jest인 경우)
# 또는
npx vitest run src/**/*auth* (Vitest인 경우)
```

### 목적

- 인증 관련 테스트가 존재하는지, 충분한 커버리지를 갖는지 확인한다.
- 테스트가 실제로 통과하는지 실행하여 검증한다.
- 테스트 커버리지 80% 이상 기준 충족 여부를 확인한다.

---

## Step 5: 보안 관점 집중 검토

### 수동 검토 항목 (Step 2에서 읽은 코드 기반)

코드를 읽으면서 아래 항목들을 중점적으로 확인한다:

1. **JWT 구현**
   - 시크릿 키가 하드코딩되어 있지 않은지 (환경 변수 사용 여부)
   - 토큰 만료 시간(expiry) 설정이 적절한지
   - 알고리즘 지정이 명시적인지 (`algorithm: 'HS256'` 등, `none` 알고리즘 차단 여부)
   - 토큰 검증 시 `verify` 사용 여부 (`decode`만 사용하면 위험)

2. **인증 미들웨어**
   - Authorization 헤더 파싱이 올바른지 (`Bearer <token>` 형식)
   - 토큰 없거나 유효하지 않을 때 적절한 HTTP 상태 코드 반환 (401/403)
   - 에러 메시지에 민감한 정보가 노출되지 않는지

3. **로그인 API**
   - 비밀번호 평문 비교가 아닌 해시 비교 사용 여부 (bcrypt 등)
   - 로그인 실패 시 "아이디 또는 비밀번호가 틀렸습니다" 같은 모호한 메시지 사용 여부 (사용자 열거 공격 방지)
   - Rate limiting 적용 여부

4. **로그아웃 API**
   - JWT 블랙리스트/토큰 무효화 전략이 있는지
   - 단순히 클라이언트 측 토큰 삭제만 하는 것은 아닌지

5. **일반 보안**
   - SQL Injection / NoSQL Injection 방지
   - 입력 값 검증 (validation)
   - CORS 설정 확인

## Step 6: 코드 품질 검토

### 실행할 명령어

```bash
# 6-1. lint 실행
npm run lint (또는 프로젝트에 설정된 lint 명령어)

# 6-2. 타입 체크 (TypeScript인 경우)
npx tsc --noEmit
```

### 수동 검토 항목

- 함수/변수 네이밍이 명확한지
- 에러 핸들링이 일관적인지
- 불필요한 주석이나 console.log가 남아있지 않은지
- 타입 정의가 적절한지 (`any` 남용 여부)
- 기존 코드 스타일과 일관성 유지 여부


## 병렬 처리 전략

| 단계 | 병렬 가능 여부 | 비고 |
|------|---------------|------|
| Step 1 (1-2, 1-3, 1-4) | 1-1 이후 나머지 병렬 | 체크아웃 완료 후 나머지 병렬 실행 |
| Step 2 | 모든 파일 병렬 읽기 | Step 1의 파일 목록에 의존 |
| Step 3 | 모든 검색 병렬 | Step 1 완료 후 바로 실행 가능 |
| Step 4 (테스트 실행) | background 실행 | Step 5, 6과 병행 |
| Step 5 | Step 2 결과 기반 수동 검토 | 코드 읽기 완료 후 |
| Step 6 (lint, tsc) | 병렬 실행 + background | Step 5와 병행 |
| Step 7 | 모든 단계 완료 후 | 최종 정리 |
