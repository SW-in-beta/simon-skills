# simon-web-search

웹 딥 리서치 스킬. 질문 분해 → 반복 멀티소스 검색 → 소스 교차검증 → simon 스타일 보고서 생성.

## Why this exists

단순 `WebSearch → 요약`은 실제 리서치가 아닙니다. 진짜 딥 리서치는:

> "Search → Reason → Search Again" — 첫 검색 결과로 더 나은 질문을 만들고, 주장(claim) 단위로 교차 검증하며, 불확실한 것은 불확실하다고 명시한다.

이 스킬은 그 사고 흐름을 구현합니다.

## Simon 검증 원칙 (웹 리서치 버전)

코드 리서치의 `file:line` 근거 원칙을 웹 리서치에 적용:

- **`[미확인]`** — 단일 소스에서만 발견되었거나, 1차 소스를 확인하지 못한 주장
- **`[추정]`** — 간접 증거는 있으나 직접 확인하지 못한 내용. 추정 근거를 반드시 명시
- **소스 체인 검증** — 코드의 caller chain 추적처럼, 2차 인용은 1차 소스까지 역추적. "A가 B라고 한다더라"가 아니라 B 원문을 직접 확인
- **`신뢰도: HIGH/MEDIUM/LOW`** — 소스별 신뢰도를 보고서에 명시:
  - HIGH: 공식 문서, peer-reviewed, 1차 소스
  - MEDIUM: 저명 기술 블로그, 확립된 뉴스 미디어, 여러 소스에서 일치
  - LOW: 단일 블로그, 미검증 커뮤니티 포스트, 날짜 오래된 자료

## Instructions

### Phase 0: 의도 파악 (AI-First Draft)

사용자 요청에서 자동 추론 후 한 번에 제시:

```
[Default] 리서치 설정:
- 주제: {주제 요약}
- 깊이: {quick(단일 패스) | standard(2-3라운드) | deep(3-5라운드, 갭 탐지 포함)}
- 관점: {기술 비교 | 최신 동향 | 개념 이해 | 트레이드오프 분석 | ...}
- 변경하려면 알려주세요.
```

**깊이 선택 기준:**
- **quick**: "A가 뭔지 알고 싶어", 사실 확인, 1-2분 내 결론
- **standard**: "A와 B를 비교해줘", 기술 선택, 5-10분 내 분석 보고서
- **deep**: "A 도입 여부를 제대로 판단하고 싶어", 중요 의사결정, 10-20분 종합 보고서

사용자가 교정하면 반영, 교정 없으면 기본값으로 진행.

### Phase 1: 질문 분해 (Query Decomposition)

주 질문을 **서브토픽**으로 분해합니다. 이 단계가 없으면 검색이 피상적으로 끝납니다.

예시:
```
주 질문: "gRPC vs REST 비교해줘"
서브토픽:
  1. 기술적 차이 (프로토콜, 성능, 타입 시스템)
  2. 실제 사용 사례 및 적합 상황
  3. 생태계 성숙도와 도구 지원
  4. 최신 트렌드 (2024-2026)
  5. 마이그레이션/전환 복잡도
```

분해된 서브토픽을 내부적으로 작성합니다 (사용자에게 표시 불필요).

**소스 타입 선택**: 서브토픽 성격에 따라 우선 검색할 소스 타입 결정:
- 기술 개념/도구 → 공식 문서, 기술 블로그, context7
- 최신 동향 → after:2024 필터, GitHub release notes, HN
- 학술/연구 → arXiv, Google Scholar 스타일 검색
- 커뮤니티 의견 → Reddit, HN, dev.to

---

### Phase 2: 멀티패스 병렬 검색

#### 라운드 1: 브로드 검색 (병렬)

서브토픽별로 **동시에** 검색합니다. 컨텍스트 격리 원칙: 페이지 전문(full content) fetch가 필요한 긴 문서는 Agent 서브에이전트로 위임.

```
# 직접 실행 (결과가 짧은 검색)
WebSearch로 각 서브토픽 병렬 검색
- "{주제} {서브토픽} site:docs.xxx OR site:github.com"
- "{주제} {서브토픽} {current_year}"
- "{주제} vs {대안} trade-off"

# Context7 조회 (라이브러리/프레임워크인 경우)
context7로 공식 문서 조회
```

```
# Agent 위임 (긴 페이지 fetch가 필요할 때)
Agent(subagent_type="general-purpose", model="haiku"):
  "다음 URL들을 WebFetch로 읽고, 결과를 /tmp/web-research-{timestamp}.txt에 저장한 뒤
   핵심 내용만 요약해 반환하라.

   URLs: {fetch할 URL 목록}
   저장: /tmp/web-research-$(date +%s).txt

   반환 형식:
   - URL별: 제목, 날짜, 핵심 주장 2-3개, 서브토픽과의 관련성
   - 신뢰도 평가 (소스 타입, 날짜, 저자 권위)
   - raw 파일 경로"
```

#### 라운드 2: 갭 탐지 및 추가 검색

라운드 1 결과를 검토하여 빠진 정보를 찾습니다. 이 단계가 deep research의 핵심입니다:

**갭 탐지 질문들:**
- 서로 모순되는 주장이 있는가? → 양쪽 입장을 더 찾아야 함
- 중요한 서브토픽에서 결과가 빈약한가? → 다른 키워드로 재검색
- 발견된 주장의 1차 소스를 아직 못 확인했는가? → 원본 논문/문서 fetch
- 날짜가 오래됐거나 최신 업데이트가 있는가? → `after:2024` 재검색

갭이 없으면 라운드 2를 건너뜁니다 (quick 깊이는 항상 skip).

#### 라운드 3+ (deep 깊이만): 집중 검증

남은 `[미확인]` 주장들을 추가 검색으로 검증합니다.

---

### Phase 3: 소스 교차검증 (Triangulation)

검색 결과를 분석하여 아래를 식별합니다:

1. **일치하는 주장** — 2개 이상의 독립 소스에서 확인된 사실
2. **단일 소스 주장** — 한 곳에서만 발견 → `[미확인]` 표시
3. **모순된 주장** — 소스마다 다른 내용 → "이견" 섹션에 별도 정리
4. **2차 인용 체인** — "A에 따르면..."으로 인용된 경우 → 원본 A를 찾아 직접 확인

**소스 신뢰도 계층:**
1. 공식 문서, 원저자 블로그, peer-reviewed (HIGH)
2. 저명 기술 블로그 (Martin Fowler, DDIA 저자 등), 확립된 IT 미디어 (MEDIUM)
3. 개인 블로그, 커뮤니티 포스트, 날짜 오래된 자료 (LOW)

---

### Phase 4: 보고서 작성

모든 리서치 결과를 simon 스타일로 정리합니다.

## 보고서 구조

보고서는 질문의 성격에 따라 아래 요소를 적절히 조합합니다.

### 1. 핵심 결론 (Executive Summary)

리서치의 핵심 답변을 **먼저** 제시합니다. 3-5문장. 결론부터, 근거 나중.

### 2. 멘탈 모델 (quick 제외)

이 주제를 이해하는 데 필요한 사고 프레임워크. 2-4개.

simon-study처럼: "전문가는 이것을 이렇게 생각한다"는 관점 전달.

각 멘탈 모델:
- **한 문장 요약** — "X는 Y로 바라봐야 한다"
- **근거 소스** — URL과 신뢰도
- **이 모델이 없으면 생기는 오해**

### 3. 핵심 발견 (Findings)

서브토픽별로 정리. **모든 주장은 소스를 명시합니다.**

형식:
```
#### {서브토픽}
- {주장} — 출처: [{소스명}]({URL}), 신뢰도: HIGH
- {주장} — 출처: [{소스명}]({URL}), 신뢰도: MEDIUM
- {주장} **[미확인]** — 단일 소스. 추가 확인 필요.
- {주장} **[추정]** — {추정 근거}
```

### 4. 트레이드오프 & 비교 (해당 시)

"A vs B" 또는 의사결정이 필요한 경우:

| 기준 | A | B | 비고 |
|------|---|---|------|
| 성능 | ... | ... | 출처: ... |
| 학습 곡선 | ... | ... | |
| 생태계 | ... | ... | |

또는 mermaid 다이어그램으로 시각화:
```mermaid
quadrantChart / flowchart / ...
```

### 5. 이견 & 논쟁 지점 (있을 때)

서로 다른 주장이 충돌하는 경우 별도 섹션으로 정리합니다.

- **쟁점**: {무엇에 대한 이견인가}
- **입장 A**: {주장} — 출처: URL
- **입장 B**: {주장} — 출처: URL
- **상황에 따른 판단 기준**: {어떤 경우에 어느 입장이 맞는가}

### 6. 시사점 & 추천 (해당 시)

사용자의 원래 목적에 대한 직접적 답변:
- 상황별 추천 (조건 명시)
- 추가로 확인해야 할 사항
- 관련 자료 링크

### 7. 소스 목록 (References)

| 소스 | URL | 날짜 | 신뢰도 | 사용 목적 |
|------|-----|------|--------|---------|
| ... | ... | ... | HIGH | 공식 문서 |

## 보고서 작성 원칙

- **결론 먼저**: Executive Summary에서 핵심 답변을 먼저. 근거는 Findings에.
- **소스 필수**: 모든 사실 주장에 URL과 신뢰도 표시. "전문가들에 따르면" 같은 막연한 표현 금지.
- **[미확인]/[추정] 솔직하게**: 단일 소스 또는 간접 증거는 반드시 표시. 불확실한 것을 확실한 것처럼 쓰는 것은 보고서의 신뢰성을 파괴한다.
- **소스 체인 검증**: "A에 따르면 B가 C라고 했다"는 허용되지 않는다. C의 원문을 직접 확인하거나 `[미확인]`으로 표시한다.
- **날짜 명시**: 기술 관련 정보는 날짜가 중요하다. 소스 날짜를 항상 포함.
- **이견 존중**: 전문가들 사이에 의견이 갈리는 주제는 한쪽만 제시하지 않고 이견 섹션에 양쪽을 기록한다.
- **웹 검색 한계 솔직히**: WebSearch/WebFetch 결과가 부실하거나 접근 불가인 경우 솔직히 명시하고, 확인 가능한 범위에서 진행.
- **시각화 적극 활용**: 비교/트레이드오프는 표나 mermaid 다이어그램으로.
- **한국어**: 한국어로 작성하되, 기술 용어는 영어 병기.

## 깊이별 실행 요약

| 단계 | quick | standard | deep |
|------|-------|----------|------|
| 질문 분해 | 간략 (2-3 서브토픽) | 상세 (4-6) | 상세 (5-8) |
| 검색 라운드 | 1 | 2 | 3-5 |
| 소스 교차검증 | 주요 주장만 | 전체 | 전체 + 1차 소스 추적 |
| Agent subagent | 선택 | 권장 | 필수 |
| 보고서 구조 | 결론 + 발견 | 전체 | 전체 + 이견 상세 |
| 예상 소요 | 1-2분 | 3-5분 | 7-15분 |
