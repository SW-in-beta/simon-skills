# Docs-First Protocol

라이브러리·DB·프레임워크·외부 서비스 등을 사용할 때, 학습 데이터 기반 기억에 의존하지 않고 공식 문서를 먼저 조회한다. LLM의 학습 데이터는 버전·API 시그니처·설정 방법에 대해 오래되었거나 부정확할 수 있기 때문이다.

## 적용 기준

**"이 정보가 틀렸을 때 사용자에게 피해가 가는가?"**를 기준으로 판단한다.

| 반드시 확인 | 확인 불필요 |
|------------|-----------|
| 라이브러리/프레임워크의 설정·마이그레이션·플러그인 | 언어 기본 문법, 표준 라이브러리 기본 기능 |
| 외부 서비스 대시보드 UI 경로·설정 절차 | 이미 코드베이스에 동일 패턴이 존재하는 API 사용 |
| DB 고유 문법·기능 (RLS, Extension, 특수 쿼리) | 단순 CRUD 쿼리 (SELECT, INSERT) |
| 처음 사용하는 API·메서드 시그니처·옵션 | 프로젝트 내에서 반복 사용 중인 패턴 |
| CLI 도구의 플래그·옵션 | — |

## 도구 우선순위

0. **프로젝트 내 CLI/스크립트** — `tool --help`, `tool -h`, 스크립트 소스 코드 읽기. 프로젝트에 CLI 도구가 있으면(Makefile targets, package.json scripts, go tool 등) 먼저 `make help`, `npm run --list` 등으로 사용 가능한 명령을 파악하고, 각 명령의 `--help`로 옵션과 동작을 학습한다.
1. **context7** — 라이브러리/프레임워크 공식 문서 (1순위, 가장 빠름). `resolve-library-id`로 라이브러리 식별 후 `query-docs`로 관련 문서를 조회한다.
2. **WebFetch** — 특정 공식 문서 URL을 아는 경우. 공식 문서 페이지를 직접 가져온다.
3. **WebSearch** — 서비스 존속 확인, 최신 변경사항 파악, 문서 URL을 모르는 경우. 검색 결과에서 공식 문서 링크를 찾아 WebFetch로 상세 조회한다.

## 조회 불가 시 대응

기억 기반으로 추측하여 안내하지 않는다. 다음과 같이 대응한다:

- **사용자에게 안내**: "공식 문서를 확인하지 못했습니다. 직접 해당 서비스의 문서/대시보드에서 확인해주세요"
- **구현 중이라면**: 해당 부분을 TODO 주석으로 표시하고, 문서 확인이 필요한 이유를 명시한다
- **절대 하지 않는 것**: 기억에 기반한 API 시그니처, 설정값, UI 경로를 사실처럼 제시

## 적용 시점

| Phase | Docs-First 활용 |
|-------|-----------------|
| **설계** | 기술 스택 선택 시 공식 문서로 기능·제약·호환성을 확인한다 |
| **구현** | 처음 사용하는 API/설정에 대해 공식 문서를 조회하고, 조회한 문서 소스를 기록한다 |
| **배포** | 외부 서비스 설정 안내 시 공식 문서를 조회한다. 서비스명 변경·UI 경로 변경을 WebSearch로 확인한다 |
| **재시도 (grind)** | 학습 데이터 기반 기억으로 구현했다가 실패한 경우, 재시도 전에 반드시 공식 문서를 조회하여 정확한 API/설정을 확인한다 |
