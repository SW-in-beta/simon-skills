# simon-study

문제 기반 심층 학습 스킬. 구체적인 문제에서 출발하여, 개념 이해 → 멘탈 모델 추출 → 설계 트레이드오프 → 개선 방향을 **하나의 보고서**로 정리합니다.

## Why this exists

학습은 보통 추상적 개념이나 코드 경로에서 시작하지 않습니다. 실제로는 **구체적인 문제**에서 시작합니다:

> "Quantile이 0%일 때 문제가 있다. Quantile이 뭔지부터, 코드에서 어떻게 동작하는지, 왜 0%가 문제인지, 어떻게 고칠지 알고 싶다."

이 스킬은 그 사고 흐름을 따라 개념과 코드를 엮어 보고서를 만듭니다.

## Instructions

### Phase 1: 문제 정의

사용자의 질문/문제를 파악합니다.

**구체적인 경우** (예: "Quantile 0% 문제 파악하고 싶어"):
- 바로 다음 단계로 진행

**모호한 경우** (예: "이 코드 좀 공부하고 싶어"):
- 2-3개의 구체적 방향을 제시하여 선택받음

문제가 정의되면, 학습에 필요한 **질문 체인**을 내부적으로 구성합니다:

```
예시 — "Quantile 0% 문제 개선"
1. Quantile이란 무엇인가? (개념)
2. 우리 코드에서 Quantile을 어떻게 사용하는가? (코드)
3. 0%일 때 어떤 동작이 발생하는가? (코드 + 개념)
4. 왜 그것이 문제인가? (설계 트레이드오프)
5. 어떤 개선 방법이 있는가? (개념 + 코드)
```

### Phase 2: 리서치 — 개념과 코드를 동시에

**병렬로** 수집합니다:

**개념 리서치:**
- context7 문서 조회 — 관련 라이브러리/프레임워크 공식 문서
- 웹 검색 — 핵심 원리, 베스트 프랙티스, 알려진 함정

**코드 리서치:**
- 진입점 파악 — 이 로직이 어디서 시작되는가
- 데이터 흐름 추적 — 관련 데이터가 어떻게 변환되는가
- 의존성과 연관 코드 — 이 로직이 영향을 미치는 범위
- 테스트 코드 — 어떤 케이스를 검증하고 있는가

**사내 자료 리서치 (설계 의도/맥락이 불명확할 때):**

코드만으로 "왜 이렇게 만들었는지" 파악이 어려울 때, Confluence와 Slack에서 관련 자료를 검색합니다.

Confluence — RFC, 설계 문서, 기술 결정 기록:
```bash
~/.claude/skills/confluence/scripts/search.sh -q "검색어" -s DEM -f
~/.claude/skills/confluence/scripts/search.sh -q "검색어" -s DEM -l rfc -f
```

Slack — 당시 논의, 의사결정 맥락, 관련 대화:
```bash
~/.claude/skills/slack/scripts/search.sh -q "검색어" -c dev-demand
~/.claude/skills/slack/scripts/fetch_thread.sh "https://buzzvil.slack.com/archives/..."
```

검색 기준:
- 코드에서 발견된 핵심 키워드 (기능명, 모듈명, 패턴명)
- 관련 RFC나 설계 문서가 있을 법한 주제
- git blame으로 발견된 관련 커밋 메시지의 키워드

사내 자료에서 찾은 내용은 보고서에 자연스럽게 통합합니다. **반드시 출처 링크를 포함** — Confluence 페이지 URL 또는 Slack 메시지 permalink을 달아서 사용자가 원문을 직접 확인할 수 있게 합니다.

리서치 결과가 부실하면 솔직히 말하고 확인 가능한 범위에서 진행합니다.

### Phase 3: 보고서 출력

질문 체인을 따라, 개념과 코드를 엮은 **보고서**를 출력합니다. 보고서는 사용자의 문제에 맞게 구성하되, 아래 요소들을 포함합니다.

---

#### 1. 핵심 멘탈 모델

이 문제를 이해하는 데 필요한 사고 프레임워크. 3-5개.

codebase-mental-model처럼, 단순 개념 나열이 아니라 **"전문가는 이것을 이렇게 생각한다"**는 관점을 전달합니다.

각 멘탈 모델은:
- **한 문장 요약** — "X는 Y로 바라봐야 한다"
- **코드에서의 증거** — 이 멘탈 모델이 드러나는 구체적 코드 위치 (file:line). 코드에 해당 개념이 구현되어 있을 때만 포함.
- **사내 자료 근거** — Confluence RFC/설계 문서, Slack 논의에서 관련 내용을 발견한 경우, 핵심 내용을 요약하고 **원문 링크(URL)**를 반드시 포함. 코드 증거와 함께 "왜 이렇게 만들었는지"를 입체적으로 설명.
- **이 모델이 없으면 생기는 문제** — 어떤 실수를 하게 되는지
- **흔한 오해** — 이 개념에 대해 자주 틀리는 점

#### 2. 플로우차트와 다이어그램

개념과 코드의 동작을 시각적으로 표현합니다. mermaid 문법을 사용합니다.

언제 어떤 다이어그램을 쓸지:
- **데이터 흐름/처리 과정** → flowchart (가장 자주 사용)
- **시간 순서가 중요한 상호작용** → sequenceDiagram
- **상태 변화** → stateDiagram-v2
- **의사결정 분기** → flowchart with decision nodes

다이어그램은 보고서 중간중간에 자연스럽게 배치합니다. 멘탈 모델을 설명한 직후, 또는 데이터 흐름을 설명할 때 함께 제시하면 효과적입니다. 별도 섹션으로 모아두지 않습니다.

#### 3. 설계 트레이드오프

코드에서 발견되는 의도적인 선택과 그 대가. 2-3개.

각 트레이드오프는:
- **무엇을 선택했는가** — 현재 접근 방식 (file:line)
- **무엇을 포기했는가** — 대안과 그것을 택하지 않은 이유
- **언제 이 선택이 문제가 되는가** — 한계 시나리오

사용자의 원래 문제가 트레이드오프의 한계 시나리오에 해당하는 경우, 이를 명시적으로 연결합니다.

#### 4. 코드 심층 분석 (조건부)

배울 만한 기술 개념이 코드에 구현되어 있을 때만 이 섹션을 포함합니다.

포함 기준: 코드에서 발견된 패턴이 일반적인 기술 개념(알고리즘, 디자인 패턴, 동시성 모델 등)을 구현하고 있어서, 코드를 통해 개념을 더 깊이 이해할 수 있는 경우.

포함하지 않는 경우: 단순한 CRUD, 설정값 전달, 보일러플레이트처럼 코드 자체에 학습할 기술적 깊이가 없는 경우.

포함할 때의 구성:
- **코드에서의 구현 방식** — 구체적 위치와 패턴 (file:line)
- **개념과의 갭** — 교과서적 구현과 다른 점이 있다면 왜 그런지
- **코드 작성자의 의도 추론** — 이 선택이 합리적인 이유 (추론임을 명시)

#### 5. 문제 진단과 개선 방향

사용자의 원래 문제에 대한 직접적 답변.

- **문제의 근본 원인** — 개념적 이유 + 코드적 이유
- **개선 방향 2-3가지** — 각각의 장단점, 영향 범위, 복잡도
- **참고 자료** — 관련 공식 문서, 블로그, 논문 등

---

### 보고서 작성 원칙

- **문제 중심**: 모든 내용은 사용자의 원래 문제와 연결. 교과서 복붙 금지
- **코드 증거**: 코드 인용은 반드시 file:line 형식. 코드에서 확인할 수 없는 설계 의도는 추론임을 명시
- **사내 자료 출처**: Confluence 페이지나 Slack 메시지를 인용했다면 반드시 원문 URL 링크를 포함. 링크 없는 사내 자료 인용은 금지
- **시각화 적극 활용**: 복잡한 흐름이나 관계는 mermaid 다이어그램으로. 텍스트만으로 설명이 어려울 때 즉시 다이어그램 삽입
- **정직함**: 웹 검색 결과가 부실하면 솔직히 말하고, 확인 가능한 범위에서 진행
- **한국어**: 한국어로 작성하되, 기술 용어는 영어 병기
