# simon-presenter

앱을 실제로 띄우고 Playwright headed 브라우저로 조작하면서, 유저스토리 기반 시나리오를 인터랙티브하게 시연하는 라이브 데모 스킬.

핵심 가치:
- **"보는 것이 믿는 것이다."** — 스크린샷이나 설명이 아닌, 실제 동작하는 앱을 보여준다.
- **"사용자 관점으로 설득한다."** — 기술 구현이 아닌 유저 가치 중심으로 시연한다.
- **"페이스를 사용자가 제어한다."** — 시나리오 전환은 항상 사용자 신호를 기다린다.

### Session Isolation Protocol

런타임 산출물을 홈 디렉토리에 세션별로 격리 저장한다. 모든 `.claude/presenter/` 경로를 `{SESSION_DIR}/presenter/`로 해석한다. `.claude/company/`, `.claude/pm/` 참조 시에도 해당 세션의 `{SESSION_DIR}` 기준으로 읽는다.

**Phase 1에서 SESSION_DIR 결정:**
```bash
PROJECT_SLUG=$(git rev-parse --show-toplevel | tr '/' '-')
SESSION_ID="presenter-$(date +%Y%m%d-%H%M%S)"
SESSION_DIR="${HOME}/.claude/projects/${PROJECT_SLUG}/sessions/${SESSION_ID}"
mkdir -p "${SESSION_DIR}/presenter"
```

## Phase 2: 시연 계획 생성

### 2.1 시나리오 설계 원칙

시나리오는 **영업/홍보 관점**에서 설계한다. 기술 검증이 아니라 가치 전달이 목적이다.

**시나리오 순서 전략:**
1. **임팩트 퍼스트**: 가장 인상적인 기능을 앞에 배치 (첫인상이 중요)
2. **스토리 아크**: 자연스러운 사용자 여정 흐름 (가입 → 핵심기능 → 결과 확인)
3. **클라이맥스 배치**: 가장 기술적으로 인상적인 기능(실시간, AI 등)은 후반에

### 2.2 시나리오 문서 형식

각 시나리오를 다음 형식으로 작성한다:

```
## 시나리오 N: [제목]

**관련 유저스토리**: US-XXX
**시연 목적**: 이 시나리오로 보여주고 싶은 핵심 가치
**사전 조건**: 이 시나리오 시작 전 필요한 상태
**예상 소요**: ~N분

### 동작 순서
1. [URL]로 이동
2. [요소]에 [값] 입력
3. [버튼] 클릭
4. → 기대 결과: [화면에 보이는 것]

### 나레이션 포인트
- 이 기능이 사용자에게 주는 가치
- 기존 방식 대비 개선점
- 기술적 하이라이트 (필요시)
```

### 2.3 더미 데이터 계획

시연에 필요한 더미 데이터를 함께 계획한다:
- **정적 데이터**: 사용자 계정, 기존 콘텐츠, 설정값
- **동적 데이터**: API 응답 모킹 (MSW 등)
- **스트리밍 데이터**: WebSocket/SSE 모킹 (실시간 알림, 채팅, 대시보드 등)

더미 데이터는 **현실감**이 중요하다:
- 실제 한국 이름, 회사명, 주소 사용
- 숫자는 현실적인 범위로 (매출 1조원이 아닌 3.2억원)
- 날짜는 현재 기준 최근 데이터로
- 스트리밍 데이터는 점진적으로 변화하는 패턴으로 (갑자기 0→100이 아닌 자연스러운 변동)

### 2.4 사용자 확인

시연 계획서를 사용자에게 보여주고 확인받는다:

```
[계획] 시연 계획이 준비되었습니다.

총 N개 시나리오, 예상 소요 ~N분입니다.

[시나리오 목록 요약]

수정하고 싶은 부분이 있으신가요? 없으면 준비를 시작하겠습니다.
```

사용자가 시나리오 순서 변경, 추가, 삭제를 요청하면 반영한다.

## Phase 3: 환경 준비

### 3.1 의존성 확인

```bash
# Node.js 프로젝트
npm install  # 또는 yarn, pnpm, bun

# Playwright 설치 (없으면)
npx playwright install chromium
```

### 3.2 환경 점검

의존성 설치 후, 서버를 띄우기 전에 환경이 제대로 갖춰져 있는지 확인한다. Phase 1.5에서 정리한 `discovery.md`의 환경 요건과 실제 환경을 대조한다.

**환경 변수 검증:**
- `.env.example` 또는 `discovery.md`에 기록된 필수 환경 변수가 `.env`, `.env.local` 등에 실제로 설정되어 있는지 확인
- 비어있거나 placeholder(예: `your-api-key-here`, `TODO`, `xxx`)인 값 탐지
- 외부 서비스 API 키 (결제, AI, 지도 등): 실제 키가 있으면 활용, 없으면 해당 엔드포인트를 모킹 대상으로 표시
- DB 접속 정보 (`DATABASE_URL` 등): 형식과 접속 가능성 확인

**인프라 상태 확인:**
- **DB**: 접속 문자열로 실제 연결 시도 (또는 Docker 컨테이너 상태 확인)
- **Docker 서비스**: `docker-compose.yml`이 있으면 필요한 컨테이너가 실행 중인지 확인. 미실행이면 사용자에게 기동 여부 확인
- **포트 가용성**: 앱이 사용할 포트(3000 등)가 이미 점유되어 있지 않은지
- **Node.js 버전**: `package.json`의 `engines` 필드나 `.nvmrc`와 현재 버전 일치 여부

**결과 보고 및 조치:**

점검 결과를 사용자에게 보고하되, 단순 나열이 아니라 **각 문제에 대한 구체적 조치 방안**을 함께 제시한다. 사용자가 판단만 하면 되도록 선택지를 준비한다.

```
[점검] 환경 점검 결과

[완료] 환경 변수: 10/12 설정됨
[완료] PostgreSQL: localhost:5432 접속 정상
[완료] 포트 3000: 사용 가능

[경고] OPENAI_API_KEY: 미설정
   → 선택 1: 키를 지금 설정 (export OPENAI_API_KEY=sk-...)
   → 선택 2: /api/chat 엔드포인트를 모킹하여 시연 (AI 응답을 시뮬레이션)
   → 선택 3: AI 채팅 관련 시나리오를 시연에서 제외

[실패] Redis: 컨테이너 미실행
   → 선택 1: 지금 기동 (docker compose up -d redis)
   → 선택 2: Redis 의존 기능을 모킹

어떻게 진행할까요?
```

**조치 원칙:**
- **[실패] 필수 인프라 누락** (DB, 필수 서비스): 시연 진행 불가. 사용자에게 기동 명령을 안내하고, 직접 기동할지 확인 후 실행. 해결될 때까지 다음 단계로 넘어가지 않는다.
- **[경고] 외부 API 키 누락**: 시연은 가능하지만 해당 기능이 제한됨. 사용자에게 세 가지 선택지 제시: (1) 키 설정, (2) 모킹으로 대체, (3) 해당 시나리오 제외.
- **[경고] 포트 점유**: 점유 중인 프로세스를 알려주고, 종료할지 다른 포트를 쓸지 확인.
- **[경고] 버전 불일치**: 경고만 표시. 대부분 동작하지만 문제 발생 시 원인일 수 있음을 안내.

모든 [실패] 항목이 해결되고, [경고] 항목에 대한 사용자 결정이 내려진 후에만 다음 단계로 진행한다. [경고]에서 "모킹으로 대체"를 선택한 항목은 3.3 모킹 설정에 자동으로 반영한다.

### 3.3 모킹 설정

시연에 필요한 모킹을 프로젝트 특성에 맞게 설정한다. 환경 점검에서 **모킹 필요**로 표시된 항목을 중심으로 설정한다. 기술 상세는 `references/playwright-patterns.md`의 "MSW 브라우저 모킹" 섹션 참조.

**접근 방식 선택 (우선순위):**
1. **프로젝트 내 시드 스크립트가 있으면** → 활용
2. **DB가 있으면** → 시드 데이터 삽입 (마이그레이션 실행 후)
3. **API만 있으면** → MSW 또는 Express mock server
4. **외부 API 의존이 있으면** → 해당 부분만 모킹

**스트리밍 데이터 모킹:**
- WebSocket/SSE 모킹 패턴은 `references/playwright-patterns.md` 참조
- 핵심: 고정 간격이 아닌 **불규칙 간격** (예: `500 + Math.random() * 1500ms`)으로 자연스럽게
- 데이터 값도 점진적 변동 패턴으로 (갑작스러운 점프 X)

모킹 파일은 프로젝트를 오염시키지 않도록 임시 디렉토리나 별도 경로에 생성하고, 시연 종료 후 정리할 수 있도록 목록을 관리한다.

### 3.4 서버 기동

**프로덕션 빌드 우선**: dev 서버의 HMR(Hot Module Replacement)이 시연 중 페이지를 리로드할 수 있다. 안정적인 시연을 위해 프로덕션 빌드를 우선 시도한다.

```bash
# 우선: 프로덕션 빌드 (안정적)
npm run build && npm start

# 차선: dev 서버 (프로덕션 빌드가 불가하거나 불필요한 경우)
npm run dev
```

서버가 준비될 때까지 대기한다 (health check 또는 포트 확인). 기동 실패 시 사용자에게 알리고 해결 방안을 제안한다.

### 3.5 Playwright 브라우저 서버 기동

시나리오 간 브라우저를 유지해야 한다. 시나리오마다 브라우저를 새로 열면 로그인 상태, 페이지 위치, 앱 상태가 초기화되어 시연 흐름이 끊긴다.

**`launchServer()` + `connect()` 패턴**을 사용한다:
1. `chromium.launchServer()`로 브라우저 프로세스를 시작하고 wsEndpoint를 저장
2. 각 시나리오 스크립트는 `chromium.connect(wsEndpoint)`로 같은 브라우저에 연결
3. 스크립트가 끝나도 브라우저는 유지됨

기술 상세와 코드 예시는 `references/playwright-patterns.md`의 "브라우저 서버 패턴" 섹션 참조.

**딜레이 전략:** `slowMo: 200` (기본 속도 조절) + 핵심 순간에 `waitForTimeout(1000~3000)` (추가 대기) 조합. 모든 동작이 균일하게 느린 것보다, 중요한 장면에서 멈추는 것이 효과적이다.

### 3.6 시나리오별 Playwright 스크립트 작성

Phase 1에서 파악한 코드 플로우와 셀렉터를 기반으로 각 시나리오의 Playwright 스크립트를 작성한다.

**시각적 하이라이팅**: Playwright가 상호작용하는 요소에 빨간 테두리 + 펄스 애니메이션을 주입하여 시청자가 어디를 봐야 하는지 알려준다. 패턴은 `references/playwright-patterns.md`의 "엘리먼트 하이라이팅" 섹션 참조.

**폼 입력**: `page.fill()`이 아닌 한 글자씩 타이핑하는 방식으로 자연스러운 입력 효과를 준다.

### 3.7 사전 검증 (Dry-run)

라이브 데모에서의 실패는 치명적이다. 본 시연 전에 모든 시나리오를 **headless 모드로 빠르게 한 번 실행**하여 검증한다.

**검증 항목:**
- 각 페이지 URL 접근 가능 여부 (404, 500 에러 없는지)
- 주요 셀렉터 존재 여부 (`page.locator(selector).waitFor()`)
- API 엔드포인트 응답 여부 (모킹 포함)
- 인증 플로우 정상 동작 (로그인 → 보호 페이지 접근)
- 시나리오 순서대로 실행 시 상태 전이 정상 여부

실패한 항목이 있으면 사용자에게 보고하고 해결한 뒤 재검증한다. 모든 검증을 통과해야 라이브 시연으로 진행한다.

```
[완료] 사전 검증 완료!

- 총 N개 시나리오 검증
- 모든 페이지 로딩 정상
- 모든 셀렉터 확인 완료
- 인증 플로우 정상

라이브 시연을 시작할까요?
```

### 3.8 준비 완료 알림

```
[완료] 환경 준비가 완료되었습니다.

- 서버: http://localhost:XXXX 실행 중
- 더미 데이터: N건 준비 완료
- 브라우저: 준비 완료
- 사전 검증: 모든 시나리오 통과

시연을 시작할까요? (녹화도 할까요?)
```

사용자가 녹화를 원하면 `recordVideo` 옵션을 활성화한다 (상세: `references/playwright-patterns.md` "비디오 녹화" 섹션).

## Phase 4: 인터랙티브 시연

이 Phase가 스킬의 핵심이다. 시나리오를 하나씩 진행하면서 사용자와 대화한다.

### 4.1 시나리오 진행 프로토콜

각 시나리오마다 다음 순서를 따른다:

**① 시나리오 소개 (나레이션)**
```
[시나리오 N/M] [제목]

[이 기능이 해결하는 문제와 가치를 2-3문장으로 설명]

지금부터 브라우저에서 [동작 요약]을 보여드리겠습니다.
```

**② Playwright 실행**
Playwright 스크립트를 실행하여 브라우저를 조작한다. 사용자는 열린 브라우저 창에서 실시간으로 본다.

각 주요 동작 단계에서:
- 동작 전: 무엇을 할 것인지 한 줄 설명 + **상호작용 요소 하이라이팅** (빨간 테두리가 표시됨)
- 동작 후: 화면에 보이는 결과를 간단히 묘사 + 충분한 대기 시간

스트리밍이나 실시간 데이터가 있는 시나리오는 데이터가 흘러가는 동안 충분히 기다려서 효과를 보여준다.

**시나리오 간 상태 관리**: 다음 시나리오가 이전 시나리오의 결과를 이어받아야 하는 경우(예: 로그인 → 대시보드) 브라우저 상태를 유지한다. 독립적인 시나리오라면 필요 시 localStorage/cookie를 정리하되, 항상 최소한으로 정리한다 (전체 초기화보다는 특정 키만 제거).

**③ 시나리오 완료 및 전환 대기**
```
[완료] 시나리오 N 완료!

[핵심 포인트 1줄 요약]

다음 시나리오로 넘어갈까요? (또는 이 화면에서 더 보고 싶은 것이 있으신가요?)
```

사용자가 추가 질문을 하면 답변하고, 추가 동작을 요청하면 즉석에서 Playwright로 수행한다. 사용자가 "다음"이라고 하면 다음 시나리오로 진행한다.

### 4.2 나레이션 톤

시연은 **영업/홍보** 목적이다. 나레이션은:
- **가치 중심**: "이 기능 덕분에 사용자는 매일 30분을 절약할 수 있습니다"
- **비교 프레임**: "기존에는 수동으로 해야 했던 작업이 클릭 한 번으로"
- **구체적 수치**: 가능하면 시간 절약, 효율 향상 등을 구체적으로
- **자연스러운 톤**: 딱딱한 발표가 아닌, 옆에서 보여주는 느낌

### 4.3 예외 처리

사전 검증(Dry-run)을 통과했더라도 라이브 환경에서는 문제가 생길 수 있다. 침착하게 대응한다.

**자동 복구 시도 (사용자에게 먼저 알리고):**
- **셀렉터 미발견**: 페이지 로딩 대기 → 대안 셀렉터 시도 (text 기반, role 기반)
- **페이지 로딩 지연**: `waitForLoadState('networkidle')` 후 재시도
- **API 에러 (4xx/5xx)**: 모킹 설정 확인 → 재요청

**사용자에게 즉시 보고:**
- **서버 다운**: 로그 확인 후 재기동 여부를 사용자에게 확인
- **인증 실패**: 토큰 만료 가능성 → 재로그인 시나리오 실행
- **예상과 다른 UI**: 현재 화면 상태를 설명하고 대안 동작 제안

**시나리오 스킵 옵션**: 특정 시나리오가 반복 실패하면 사용자에게 "이 시나리오를 건너뛰고 다음으로 넘어갈까요?"를 제안한다. 시연 전체가 중단되는 것보다 낫다.

절대로 에러를 숨기거나 무시하지 않는다. 투명하게 공유하고 함께 해결한다.

### 4.4 마지막 시나리오 후

```
[완료] 모든 시나리오 시연이 완료되었습니다!

[요약] 시연 요약:
- 총 N개 유저스토리 시연
- [핵심 하이라이트 2-3개]

추가로 보고 싶은 기능이나 다시 보고 싶은 시나리오가 있으신가요?
없으시면 환경을 정리하겠습니다.
```

## Phase 5: 정리

### 5.1 환경 정리

사용자가 종료를 확인하면:
1. Playwright 브라우저 종료
2. dev 서버 종료
3. mock 서버 종료 (있는 경우)
4. 임시 모킹 파일 정리 (프로젝트 원본에 추가한 파일이 있으면 되돌리기)

### 5.2 git 상태 확인

시연 준비 과정에서 프로젝트에 변경이 생겼을 수 있다. `git status`로 확인하고:
- 시드 데이터 파일 → 유지 여부 사용자에게 확인
- 모킹 설정 파일 → 기본적으로 삭제
- 기타 변경 → 사용자에게 보고

## 핵심 규칙

1. **사용자 신호 없이 다음 시나리오로 넘어가지 않는다** — 페이스는 항상 사용자가 제어한다
2. **Playwright는 반드시 headed 모드** — 사용자가 브라우저를 직접 볼 수 있어야 한다
3. **slowMo를 적용** — 너무 빠르면 시연 효과가 없다 (기본 500ms, 복잡한 동작은 더 길게)
4. **프로젝트를 오염시키지 않는다** — 모킹/시드 데이터는 최소 침습으로, 정리 가능하게
5. **실패해도 당황하지 않는다** — 라이브 데모에서 문제는 자연스러운 것, 투명하게 대응
