# Expert Comment Review Protocol

사용자가 PR에 남긴 코멘트를 전문가 Agent가 검증하는 프로토콜.

## 전문가 선정 기준

코멘트의 내용을 분석하여 가장 관련성 높은 전문가 1-2명을 선정한다.

| 도메인 | 전문가 | 트리거 키워드/패턴 |
|--------|--------|-------------------|
| 아키텍처/설계 | Architecture Expert | 구조, 패턴, 의존성, 결합도, 분리, 레이어, DDD |
| 성능 | Performance Expert | 느림, N+1, 쿼리 최적화, 캐시, 인덱스, 병목, 메모리 |
| 보안 | Security Expert | 인증, 권한, 주입, XSS, CSRF, 토큰, 암호화 |
| 테스트 | Testing Expert | 테스트, 커버리지, 모킹, 엣지케이스, 검증 |
| 비즈니스 로직 | Domain Expert | 도메인 규칙, 비즈니스 요구사항, 스펙, 정책 |
| 코드 품질 | Code Quality Expert | 네이밍, 가독성, 중복, 컨벤션, 리팩토링 |
| 데이터/DB | Data Expert | 스키마, 마이그레이션, 쿼리, 트랜잭션, 정합성 |
| 에러 처리 | Reliability Expert | 예외, 에러 핸들링, 재시도, 복구, 로깅 |

**다중 도메인**: 코멘트가 여러 도메인에 걸치면 관련 전문가를 모두 호출하여 병렬로 검증한다.

**도메인 불명확**: 분류가 어려우면 Architecture Expert + Domain Expert를 기본으로 호출한다.

## Agent 호출 방법

각 전문가를 `Agent(subagent_type="general-purpose")`로 spawn한다.

**Agent에 전달할 컨텍스트:**

```
당신은 {전문가 역할}입니다.

## 검증 대상
- **사용자 코멘트**: {코멘트 원문}
- **코멘트 위치**: {파일 경로}:{라인 번호}
- **PR 번호**: #{pr_number}
- **리뷰 사이클**: R{N} (현재 {N}번째 리뷰 라운드)

## 작업
1. 코멘트가 언급하는 코드를 Read로 직접 확인하세요
2. 코멘트의 지적이 기술적으로 타당한지 검증하세요:
   - 실제 코드 동작과 코멘트의 주장이 일치하는지
   - 코드베이스의 기존 패턴/컨벤션과 비교
   - 잠재적 부작용이나 엣지케이스 확인
3. 기술적 주장에 대해 공식 문서로 fact-check하세요:
   - 코멘트 또는 자신의 분석에 "이 버전에서 지원/미지원", "deprecated", "anti-pattern", "권장 방식" 등의 주장이 포함되면, 반드시 공식 문서를 확인한 뒤에 verdict를 내리세요
   - context7 MCP(`resolve-library-id` → `query-docs`)로 라이브러리 문서를 조회하세요. context7에서 찾지 못하면 WebSearch + WebFetch로 공식 문서를 직접 확인하세요
   - 문서를 확인하지 못했다면, verdict에 "문서 미확인" 상태를 명시하고 확인되지 않은 주장을 사실처럼 단정하지 마세요
   - LLM 학습 데이터는 최신이 아닐 수 있다는 점을 항상 인지하세요 — 이전 세션에서 "Django 2.2에서 TextChoices 미지원"이라고 답변했으나 실제로는 지원하는 사례가 있었습니다
4. verdict를 내려주세요:
   - AGREE: 코멘트가 타당하며 수정이 필요
   - PARTIAL: 코멘트의 일부만 타당하거나, 다른 방식의 수정이 더 적절
   - COUNTER: 코멘트의 지적이 부정확하거나, 현재 구현이 더 적절한 근거가 있음

## 출력 형식
verdict: {AGREE | PARTIAL | COUNTER}
reasoning: {기술적 근거 — 코드 참조 포함}
action: {AGREE일 때: 구체적 수정 방안 / PARTIAL일 때: 수정된 방안 + 이유 / COUNTER일 때: 반론 근거}
code_references: {근거가 되는 코드 위치 목록}
doc_references: {공식 문서에서 확인한 내용 — URL + 핵심 발췌. 문서 확인을 하지 못한 경우 "미확인: {이유}" 명시}
```

## Verdict 파일 저장

전문가 Agent의 verdict를 받은 후, 코드 수정이나 대댓글 작성 **전에** verdict를 파일로 저장한다. 이 파일이 존재해야 코드 수정을 진행할 수 있다.

```bash
mkdir -p {SESSION_DIR}/memory/expert-verdicts
# 파일명: 코멘트 ID (GitHub API의 id 필드)
cat > {SESSION_DIR}/memory/expert-verdicts/{comment_id}.md << 'EOF'
- **comment_id**: {comment_id}
- **expert_role**: {전문가 역할}
- **verdict**: {AGREE | PARTIAL | COUNTER}
- **reasoning**: {기술적 근거 요약}
- **action**: {수정 방안 또는 반론}
- **doc_references**: {확인한 공식 문서 URL + 핵심 내용 요약. 미확인 시 "미확인: {이유}"}
- **review_cycle**: R{N}
- **timestamp**: {ISO 8601}
EOF
```

## Verdict 처리

### AGREE (동의)

1. 전문가가 제안한 수정 방안대로 코드 수정
2. 커밋 → 푸시
3. PR 대댓글 작성:

```markdown
**[R{N}]** 수정 완료했습니다.

**전문가 검증**: {전문가 역할}
**변경 전**: {기존 코드 발췌 또는 동작 설명}
**변경 후**: {수정된 코드 발췌 또는 동작 설명}
**근거**: {왜 이 수정이 맞는지 — 코드 참조}
**공식 문서**: {확인한 공식 문서 URL + 핵심 내용. 해당 없으면 생략}
```

### PARTIAL (부분 동의)

1. 전문가의 수정된 방안대로 코드 수정
2. 커밋 → 푸시
3. PR 대댓글 작성:

```markdown
**[R{N}]** 수정 반영했습니다 (일부 다르게 적용).

**전문가 검증**: {전문가 역할}
**동의 부분**: {코멘트에서 타당한 부분}
**변경 전**: {기존 코드 발췌 또는 동작 설명}
**변경 후**: {수정된 코드 발췌 또는 동작 설명}
**다르게 적용한 이유**: {왜 원래 요청과 다르게 했는지}
**근거**: {기술적 이유 — 코드 참조}
**공식 문서**: {확인한 공식 문서 URL + 핵심 내용. 해당 없으면 생략}
```

### Self-Agreement Bias 견제 (COUNTER verdict 전 필수 자문)

> **참조**: 이 견제는 Step 4(피드백 루프)의 전문가 검증에 적용된다.
> Step 2의 Pass 1(최초 리뷰)에는 별도의 Adversarial Self-Check가 있다
> (review-strategy.md Pass 1, 6번 참조). 두 메커니즘의 목적은 동일하다 —
> AI가 자기(또는 동종 AI)가 작성한 코드에 대해 정당화 편향을 갖는 것을 방지한다.

이 코드는 다른 AI 에이전트가 작성한 것이다. 같은 모델이 작성한 코드를 검증할 때, 코드의 접근법에 자연스럽게 동의하는 경향(self-agreement bias)이 존재한다 — 구현자의 추론 패턴과 검증자의 추론 패턴이 동일하기 때문이다.

COUNTER verdict를 내리기 전에 다음을 자문한다:
1. 사용자의 지적을 반박하는 근거가 **코드/문서에서 직접 확인**된 것인지, "이 접근법이 합리적으로 보인다"는 직관에 기반한 것인지?
2. 만약 이 코드가 **주니어 개발자가 작성**한 것이라면, 동일한 verdict를 내릴 것인지?
3. 사용자의 지적이 타당할 경우의 영향(잠재적 버그/성능 저하)이, 현재 구현을 유지할 경우의 이점보다 큰지?

직관이 아닌 검증된 근거(코드 동작, 공식 문서, 테스트 결과)만으로 verdict를 내린다.

### COUNTER (반박)

1. 코드 변경 없음
2. PR 대댓글 작성:

```markdown
**[R{N}]** 검토 결과, 현재 구현을 유지하는 것이 적절하다고 판단합니다.

**전문가 검증**: {전문가 역할}
**검토 내용**: {코멘트의 지적 사항 요약}
**현재 구현 유지 근거**:
- {근거 1 — 코드 참조}
- {근거 2 — 코드 참조}
**공식 문서**: {확인한 공식 문서 URL + 핵심 내용. 해당 없으면 생략}
**고려한 대안**: {코멘트대로 수정했을 때의 잠재적 문제}
```

## 다수 전문가 의견 충돌 시

2명 이상의 전문가가 호출되어 verdict가 갈리는 경우:

1. 각 전문가의 verdict와 reasoning을 종합
2. **다수결이 아닌 근거의 질**로 판단 — 코드 참조가 더 구체적이고 재현 가능한 쪽을 우선
3. 판단이 어려우면 양쪽 근거를 모두 대댓글에 포함하고, 사용자에게 최종 판단을 요청:

```markdown
전문가 의견이 갈립니다. 양쪽 근거를 공유합니다.

**{전문가 A} (AGREE)**:
{reasoning}

**{전문가 B} (COUNTER)**:
{reasoning}

어떤 방향으로 진행할까요?
```

## 복수 코멘트 병렬 처리

사용자가 여러 코멘트를 동시에 남긴 경우, 각 코멘트의 전문가 호출을 **병렬**로 수행한다. 단, 코멘트 간 의존성이 있는 경우 (같은 파일의 연관된 코드) 순차 처리한다.
