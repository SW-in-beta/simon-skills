## simon-bot-pm's Response

이 요청은 **simon-bot-pm 스킬의 범위에 해당하지 않습니다.** 이유를 설명드리겠습니다.

### Scope Guard 판단

simon-bot-pm은 **여러 기능을 조율하는 전체 프로젝트 관리**를 위한 PM 스킬입니다. 아래 Scope Guard 기준표에 따라 판단했습니다:

| 기준 | 이 요청의 해당 사항 | 판정 |
|------|---------------------|------|
| **기능(Feature) 수** | 1개 (마크다운 파일 읽기 + 목차 생성) | simon-bot으로 리다이렉트 |
| **예상 변경 파일 수** | 5개 미만 (index.js, package.json, README.md, 테스트 파일 정도) | simon-bot으로 리다이렉트 |
| **기능 간 의존성** | 없음 (단일 기능) | simon-bot으로 리다이렉트 |
| **PRD/작업 분해 필요** | 불필요 (범위가 이미 명확) | simon-bot으로 리다이렉트 |

4개 기준 **모두** "simon-bot으로 리다이렉트"에 해당합니다.

### 왜 simon-bot-pm이 과한가

- **단일 기능 요청입니다.** "마크다운 파일을 읽어서 목차를 자동 생성"하는 하나의 명확한 기능만 있습니다. simon-bot-pm은 3개 이상의 기능을 조율하고, PRD를 작성하고, 작업을 분해해서 여러 에이전트에게 분배하는 프로젝트 관리 스킬입니다.
- **PRD 인터뷰가 불필요합니다.** 이미 무엇을 만들지 (CLI 도구), 무엇을 하는지 (마크다운 목차 생성), 어떤 기술로 (Node.js) 명확히 정의되어 있습니다. 3~5라운드의 인터뷰와 PRD 문서 작성은 이 규모에서는 과도한 오버헤드입니다.
- **작업 분해가 필요 없습니다.** Feature 분해, 의존성 그래프, 실행 그룹 같은 PM 워크플로는 여러 기능이 얽혀 있을 때 가치가 있습니다. 이 요청은 하나의 simon-bot이 처음부터 끝까지 처리할 수 있는 크기입니다.

### 추천

**`simon-bot`을 사용하시는 것을 추천합니다.** 동일한 요청을 simon-bot에게 전달하면 바로 구현에 들어갈 수 있습니다. 예를 들어:

> `/simon-bot 간단한 CLI 도구 하나 만들어줘. 마크다운 파일을 읽어서 목차를 자동 생성하는 거야. Node.js로.`

simon-bot이 TDD 기반으로 테스트 작성 -> 구현 -> 코드 리뷰 -> 검증까지 한 번에 처리해줄 것입니다.

물론, 그래도 PM 워크플로(PRD 작성, 체계적 작업 분해 등)를 원하시면 진행할 수 있습니다. 어떻게 하시겠습니까?
