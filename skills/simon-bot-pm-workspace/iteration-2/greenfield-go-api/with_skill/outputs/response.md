```markdown
# PRD: 사내 커뮤니티 REST API 서버

## 1. Overview
- **비전**: 사내 구성원 간 소통과 정보 공유를 위한 커뮤니티 플랫폼 API 서버
- **배경과 동기**: 사내 커뮤니티 활성화를 위해 게시판, 사용자 관리, 알림 기능을 갖춘 백엔드 API 서버 구축
- **대상 사용자**: 사내 구성원 (일반 사용자 + 관리자)

## 2. Goals & Non-Goals

### Goals
- JWT 기반 인증/인가 시스템 구축
- 카테고리별 다중 게시판 시스템 구현 (게시글, 댓글, 좋아요)
- 인앱 알림 시스템 구현 (API 폴링 방식)
- 관리자 기능 (사용자 차단, 공지 고정, 게시글 삭제)
- RESTful API 설계 (/api/v1/ 형태)
- Docker 컨테이너화
- Swagger/OpenAPI 문서 자동 생성

### Non-Goals
- 파일/이미지 첨부 (향후 추가 예정)
- 실시간 알림 (WebSocket/SSE) (향후 추가 예정)
- 이메일 알림
- 프론트엔드 UI
- OAuth2 소셜 로그인

## 3. User Stories

### 일반 사용자
- 사용자는 회원가입/로그인하여 서비스를 이용할 수 있다
- 사용자는 자신의 프로필을 조회/수정할 수 있다
- 사용자는 카테고리별 게시판에서 게시글을 작성/조회/수정/삭제할 수 있다
- 사용자는 게시글에 댓글을 작성/수정/삭제할 수 있다
- 사용자는 게시글에 좋아요를 누르거나 취소할 수 있다
- 사용자는 알림 목록을 조회하고 읽음 처리할 수 있다

### 관리자
- 관리자는 전체 사용자 목록을 조회할 수 있다
- 관리자는 특정 사용자를 차단/해제할 수 있다
- 관리자는 게시글을 공지로 고정/해제할 수 있다
- 관리자는 부적절한 게시글/댓글을 삭제할 수 있다

## 4. Feature Inventory

### Must-have
| ID | 기능 | 설명 | 우선순위 |
|----|------|------|---------|
| F1 | 프로젝트 기반 설정 | DB 연결, 라우터, 미들웨어, 프로젝트 구조 | P0 |
| F2 | 사용자 인증 | 회원가입, 로그인, 로그아웃, JWT 토큰 관리 | P0 |
| F3 | 사용자 프로필 | 프로필 조회/수정, 관리자 사용자 목록/차단 | P1 |
| F4 | 게시판 카테고리 | 카테고리 CRUD (관리자), 카테고리 목록 조회 | P1 |
| F5 | 게시글 | 게시글 CRUD, 페이지네이션, 공지 고정 | P1 |
| F6 | 댓글 | 댓글 CRUD, 게시글별 댓글 목록 | P1 |
| F7 | 좋아요 | 게시글 좋아요/취소, 좋아요 수 집계 | P2 |

### Nice-to-have
| ID | 기능 | 설명 |
|----|------|------|

## 5. Technical Requirements

### 기술 스택
- **언어**: Go
- **웹 프레임워크**: Gin
- **데이터베이스**: PostgreSQL
- **인증**: JWT (Access Token + Refresh Token)
- **ORM/Query**: sqlx 또는 GORM (구현 시 결정)
- **API 문서**: swaggo/swag (Swagger)

### 아키텍처 방향
- 레이어드 아키텍처: Handler → Service → Repository
- 패키지 구조: 도메인별 분리 (user, post, comment, notification)
- 의존성 주입 패턴 사용
- 통일된 에러 응답 포맷
- 환경 변수 기반 설정 관리

### API 설계
- 엔드포인트: `/api/v1/...`
- 페이지네이션: offset-based (`?page=1&size=20`)
- 에러 응답: `{ "error": { "code": "...", "message": "..." } }`

## 6. Constraints & Assumptions

### 기술적 제약
- Go 1.21 이상
- PostgreSQL 14 이상

### 가정 사항
- 단일 서버 배포 (수평 확장은 향후 고려)
- 사내 서비스이므로 대규모 트래픽은 고려하지 않음
- 파일 첨부, 실시간 알림은 v2에서 추가

## 7. Success Criteria
- [ ] 모든 Must-have 기능 구현 완료
- [ ] 전체 빌드 성공 (go build ./...)
- [ ] 핵심 비즈니스 로직 단위 테스트 통과
- [ ] API 엔드포인트 정상 동작 확인
- [ ] Docker 빌드 및 실행 가능
- [ ] Swagger 문서 접근 가능
```

### Key Design Decisions in the Simulation

1. **Greenfield 자동 판별**: "만들고 싶어"라는 명확한 신규 프로젝트 시그널 감지. 별도 질문 없이 확인만 요청.

2. **인터뷰 효율성**: 사용자가 첫 메시지에서 이미 밝힌 정보(Go, Gin, PostgreSQL, 3개 기능)를 재질문하지 않고, 해당 정보를 기반으로 더 구체적인 질문(인증 방식, 게시판 구조, 알림 트리거 등)으로 진행.

3. **PRD의 Non-Goals 명확화**: 인터뷰에서 "나중에"라고 한 파일 첨부, 실시간 알림을 Non-Goals로 분류하여 스코프를 명확히 함.

4. **Feature Inventory 세분화**: 사용자가 말한 3개 기능(사용자 관리, 게시판, 알림)을 실제 구현 가능한 8개 Feature로 세분화 (인프라 설정 포함).

5. **Phase 2 이후 미진행**: 시뮬레이션 범위가 Phase 0 ~ Phase 1이므로 PRD 리뷰에서 중단. 실제로는 사용자 승인 후 Phase 2(Task Breakdown)로 진행.
