## Phase 1: Spec-Driven Design - Vision Interview (1-A)

말씀해주신 내용을 정리하면:
- **기술 스택**: Go + Gin + PostgreSQL (선호사항으로 기록)
- **핵심 기능 3가지**: 사용자 관리, 게시판, 알림

좋은 출발점입니다. 프로젝트를 구체화하기 위해 몇 가지 여쭤볼게요.

---

### Round 1: 비전 & 대상

1. **이 REST API 서버의 주요 사용 목적이 무엇인가요?**
   - 예: 사내 커뮤니티 서비스, 외부 고객용 포럼, 학습/포트폴리오 프로젝트 등
   - 이 서비스를 사용하는 클라이언트는 무엇인가요? (웹 프론트엔드, 모바일 앱, 다른 백엔드 서비스 등)

2. **예상 사용자 규모와 트래픽은 어느 정도인가요?**
   - 예: 소규모(100명 이하), 중규모(1K~10K), 대규모(10K+)
   - 동시 접속자 수 예상이 있으시면 알려주세요.

3. **배포 환경은 어떻게 생각하고 계시나요?**
   - 예: Docker, AWS, GCP, 자체 서버, 아직 미정 등
   - MVP를 언제까지 완성하고 싶으신 시간 제약이 있나요?

---

### Round 2: 핵심 기능 상세 (사용자 관리)

4. **사용자 관리에서 필요한 기능을 구체적으로 알려주세요.**
   - 회원가입/로그인 방식: 이메일+비밀번호? OAuth(Google, GitHub 등)? 둘 다?
   - 사용자 역할(Role)이 필요한가요? (예: 일반 사용자, 관리자, 중간 관리자 등)
   - 프로필 관리: 어떤 정보를 관리해야 하나요? (닉네임, 프로필 이미지, 자기소개 등)

5. **인증 방식에 대한 선호가 있나요?**
   - JWT 토큰 기반, 세션 기반, 또는 특별한 요구사항이 있는지

---

### Round 3: 핵심 기능 상세 (게시판 & 알림)

6. **게시판 기능의 상세 요구사항을 알려주세요.**
   - 게시판 종류: 단일 게시판? 카테고리별 다중 게시판?
   - 게시글 기능: 제목+본문만? 이미지 첨부? 태그? 마크다운 지원?
   - 댓글: 단일 댓글? 대댓글(계층형)? 댓글에 좋아요?
   - 검색: 제목 검색? 본문 검색? 태그 검색?
   - 페이지네이션: 커서 기반? 오프셋 기반?

7. **알림 기능에 대해 구체적으로 알려주세요.**
   - 어떤 이벤트에 알림을 보내나요? (내 글에 댓글, 좋아요, 새 공지 등)
   - 알림 채널: 인앱 알림만? 이메일? 푸시 알림?
   - 실시간 알림이 필요한가요? (WebSocket 등) 아니면 폴링 방식도 괜찮은가요?

---

### Round 4: 우선순위 & 범위 확인

8. **Must-have / Nice-to-have / Out of scope를 구분해주세요.**
   - 아래는 제가 예상하는 분류인데, 맞는지 확인해주시고 수정할 부분이 있으면 말씀해주세요:

   | 분류 | 기능 |
   |------|------|
   | **Must-have (P1)** | 회원가입/로그인, 게시글 CRUD, 댓글, 인앱 알림 |
   | **Nice-to-have (P2)** | 좋아요, 검색, 프로필 이미지, 이메일 알림 |
   | **Out of scope** | 실시간 채팅, 파일 업로드(대용량), 결제 |

9. **추가로 고려해야 할 비기능 요구사항이 있나요?**
   - 예: 로깅/모니터링, Rate Limiting, API 문서화(Swagger), 국제화(i18n) 등

---

위 질문들에 답변해주시면, 답변을 바탕으로 **Feature Specification (WHAT)** 문서를 작성하겠습니다. 기술 스택은 Go + Gin + PostgreSQL 선호사항을 기록해두었고, Spec 작성 후 전문가 패널(CTO Agent + Dev Lead Agent)이 해당 스택의 적합성을 검증하고 필요한 보완 사항을 추천할 예정입니다.

답변은 한번에 다 주셔도 되고, 편하게 나눠서 주셔도 됩니다.
