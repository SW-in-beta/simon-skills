좋은 요청입니다. 세션 기반에서 JWT + OAuth2로의 전환은 꽤 큰 작업이고, 특히 5개 서비스에 걸쳐 기존 API 호환성까지 유지해야 한다면 신중한 접근이 필요합니다.

바로 작업에 들어가기 전에, 몇 가지 확인이 필요합니다.

---

## 먼저 파악해야 할 것들

작업 범위를 정확히 잡기 위해 아래 질문들에 대한 답이 필요합니다:

### 1. 현재 상태 파악

- **현재 세션 관리 방식**: 서버 사이드 세션 스토어(Redis, DB 등)를 쓰고 있는지, 쿠키 기반인지?
- **인증 관련 코드 위치**: 모노레포 내에서 인증 로직이 어디에 있는지? (공통 라이브러리, 각 서비스 자체, API Gateway 등)
- **5개 서비스 목록**: 어떤 서비스들이 관련되어 있는지, 각각의 언어/프레임워크는?
- **현재 사용자 모델**: 유저 정보가 어떤 DB에, 어떤 스키마로 저장되어 있는지?

### 2. 목표 상태 확인

- **JWT 용도**: 서비스 간 통신(internal)용인지, 클라이언트-서버(external)용인지, 둘 다인지?
- **OAuth2 범위**: 자체 OAuth2 Provider를 구축하는 건지, 외부 IdP(Google, Kakao 등) 연동인지, 둘 다인지?
- **OAuth2 Grant Types**: Authorization Code, Client Credentials, PKCE 중 어떤 것이 필요한지?
- **토큰 저장 전략**: Access Token + Refresh Token 패턴을 쓸 건지?
- **API 호환성 기간**: 기존 세션 방식을 얼마나 오래 병행 운영할 건지?

### 3. 인프라 및 제약사항

- **API Gateway 유무**: Kong, Nginx, AWS API Gateway 등을 사용 중인지?
- **서비스 간 통신 방식**: gRPC, REST, 메시지 큐 등?
- **배포 전략**: Blue-Green, Canary 등 점진적 배포가 가능한 환경인지?
- **보안 요구사항**: 토큰 만료 시간, 키 로테이션 주기 등에 대한 정책이 있는지?

## 일반적인 접근 방향 (참고용)

위 질문에 대한 답을 확인한 후 구체적인 계획을 세우겠지만, 이런 규모의 인증 리팩토링에서 일반적으로 고려하는 큰 방향은 다음과 같습니다:

### Phase 1: 분석 및 설계
- 현재 인증 흐름 전체 매핑 (5개 서비스 각각)
- 세션 의존성 파악 (어디서 세션을 읽고/쓰는지)
- JWT + OAuth2 아키텍처 설계
- 마이그레이션 전략 수립 (Dual-write/Dual-read 패턴 등)

### Phase 2: 코어 인프라 구축
- JWT 발급/검증 공통 라이브러리
- OAuth2 Authorization Server (또는 외부 IdP 연동 레이어)
- 토큰 관리 (발급, 갱신, 폐기)

### Phase 3: 점진적 마이그레이션
- 기존 세션과 JWT 병행 운영 레이어 (Adapter 패턴)
- 서비스별 순차 전환
- 기존 API 엔드포인트에 대한 하위 호환 유지

### Phase 4: 정리
- 세션 기반 코드 제거
- 모니터링 및 알림 설정
- 문서화
