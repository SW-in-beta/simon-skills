인증 시스템 전면 리팩토링은 규모가 크고 영향 범위가 넓은 작업입니다. 체계적으로 접근하기 위한 제안을 드리겠습니다.

---

## 현재 상황 파악이 먼저 필요합니다

본격적인 설계에 앞서 아래 사항들을 확인해야 합니다.

1. **기존 세션 기반 인증의 구현 방식** -- 세션 스토어(Redis, DB 등), 세션 ID 전달 방식(Cookie, Header), 세션 만료/갱신 정책
2. **관련 5개 서비스의 인증 의존 구조** -- 각 서비스가 세션을 어떻게 검증하는지(직접 세션 스토어 조회, 중앙 인증 서비스 호출, 미들웨어 등)
3. **현재 API 클라이언트 현황** -- 모바일 앱, 웹 프론트엔드, 서드파티, 내부 서비스 간 호출 등 어떤 클라이언트가 있는지
4. **인증이 필요한 API 엔드포인트 규모** -- 마이그레이션 대상의 전체 규모 파악

이 정보 없이 설계를 시작하면 중간에 큰 방향 수정이 생길 수 있습니다.

## 권장 접근 방향

### 1단계: 병행 운영 구조 설계 (Dual-Auth)

기존 API 호환성 유지가 요구사항이므로, 세션과 JWT를 동시에 지원하는 이중 인증 레이어를 먼저 구축합니다.

- 인증 미들웨어가 Authorization 헤더(Bearer JWT)를 우선 확인하고, 없으면 기존 세션 쿠키로 폴백
- 새 토큰 발급 엔드포인트 추가 (`POST /auth/token`, `POST /auth/token/refresh`)
- 기존 로그인 엔드포인트는 세션과 JWT를 동시에 발급하도록 수정

### 2단계: JWT 설계

- **Access Token**: 짧은 만료(15~30분), 서비스 간 검증에 비밀키 공유 또는 공개키 배포
- **Refresh Token**: 긴 만료(7~30일), DB 또는 Redis에 저장하여 폐기(revoke) 가능하게
- **토큰 페이로드**: user_id, roles/permissions, 서비스별 필요 클레임 최소화
- **키 관리**: RSA 또는 ECDSA 비대칭 키 사용 권장 -- 각 서비스가 공개키만으로 독립 검증 가능

### 3단계: OAuth2 도입

- Authorization Code Flow (웹), PKCE (모바일/SPA) 지원
- 내부 서비스 간 통신은 Client Credentials Flow
- OAuth2 서버를 별도 서비스로 분리할지, 기존 인증 서비스에 통합할지 결정 필요
- 스코프(scope) 체계 설계 -- 서비스별 접근 권한 분리

### 4단계: 서비스별 순차 마이그레이션

5개 서비스를 한 번에 전환하면 위험합니다. 우선순위를 정해 하나씩 전환합니다.

- 영향도가 낮은 내부 서비스부터 시작
- 각 서비스 전환 시: 미들웨어 교체 -> 테스트 -> 모니터링 -> 다음 서비스
- 전환 완료된 서비스도 폴백 기간 동안 세션 인증 유지

### 5단계: 세션 인증 제거

- 모든 서비스와 클라이언트가 JWT로 전환 완료된 후
- 세션 관련 코드/스토어 제거
- 폴백 미들웨어 제거

## 주요 리스크와 대응

| 리스크 | 대응 |
|--------|------|
| 마이그레이션 중 인증 장애 | Dual-Auth로 세션 폴백 항상 유지 |
| JWT 탈취 | 짧은 만료 + Refresh Token 회전(rotation) + 블랙리스트 |
| 서비스 간 키 동기화 | 비대칭 키 사용, 공개키는 JWKS 엔드포인트로 배포 |
| API 호환성 깨짐 | 기존 엔드포인트 시그니처 유지, 인증 방식만 내부적으로 변경 |
| 롤백 필요 시 | Feature flag로 인증 방식 전환 가능하게 구성 |

## 진행하기 위해 필요한 것

이 작업을 실제로 진행하려면:

1. **코드베이스 탐색** -- 현재 인증 관련 코드의 위치, 구조, 의존성 파악
2. **서비스 목록과 역할** -- 5개 서비스가 각각 어떤 역할이고, 인증을 어떻게 사용하는지
3. **우선순위 결정** -- 어떤 서비스/기능부터 전환할지
4. **기술 스택 확인** -- Go 서비스인지, Python인지, 혼합인지에 따라 JWT 라이브러리와 미들웨어 선택이 달라짐
