# Discovery Interview Protocol (의뢰 모드)

> **적용 조건**: 의뢰 모드일 때.
> 사용자가 아이디어만 있고 구체적인 기능 목록이 없을 때, PM이 체계적인 인터뷰로 요구사항을 함께 발견하고 구체화한다.

의뢰 모드는 실제 소프트웨어 회사가 외부 클라이언트의 프로젝트를 수주할 때와 같은 프로세스다. 클라이언트(사용자)는 "이런 문제를 해결하고 싶다" 수준의 아이디어만 있을 수 있고, PM은 그 아이디어를 구현 가능한 스펙으로 점진적으로 구체화해야 한다.

---

## 인터뷰 5단계

### Stage 1: 비전 & 맥락 (Vision & Context)
사용자의 큰 그림을 이해한다. 기능이 아니라 **문제와 목적**에 집중한다.

```
PM 질문 예시:
- "이 프로젝트를 시작하게 된 배경이 뭔가요? 어떤 문제를 해결하고 싶으신가요?"
- "이 서비스의 핵심 사용자는 누구인가요? 그들이 현재 어떻게 이 문제를 해결하고 있나요?"
- "성공했을 때의 모습을 한 문장으로 표현하면?"
- "참고하는 서비스나 경쟁자가 있나요? 그것과 어떻게 다르게 만들고 싶나요?"
```

한 번에 **2개 이하**의 질문만 한다. 사용자의 답변을 기반으로 **비전 요약**을 작성하고 확인받는다.

### Stage 2: 사용자 여정 탐색 (User Journey Discovery)
핵심 사용자의 관점에서 서비스 이용 흐름을 함께 그린다.

```
PM 질문 예시:
- "사용자가 처음 이 서비스에 접속하면 무엇을 하게 되나요?"
- "그 다음 단계는요? 핵심적으로 사용자가 반복하는 액션이 있나요?"
- "사용자가 이 서비스에서 가장 많은 시간을 보내는 화면은 어떤 화면일까요?"
- "관리자나 운영자 등 다른 유형의 사용자가 있나요?"
```

답변을 기반으로 **사용자 여정 초안**을 시각적으로 제시한다:
```
사용자 여정:
[가입] → [첫 설정] → [핵심 기능 사용] → [결과 확인] → [반복 사용]
         ↑ 여기서 사용자가 이탈할 수 있는 지점
```

### Stage 3: 기능별 심층 인터뷰 (Feature Deep-Dive)
사용자 여정의 각 단계를 하나씩 깊게 파고든다.

```
PM 질문 예시:
- "[기능 X]에서 사용자가 할 수 있어야 하는 것은 정확히 뭔가요?"
- "이걸 사용할 때 불편하거나 실수할 수 있는 상황이 있을까요?"
- "이 기능에서 가장 중요한 것과 있으면 좋지만 없어도 되는 것을 나눠본다면?"
- "이 기능이 잘 작동하는지 어떻게 확인할 수 있을까요?"
```

각 기능 인터뷰 후, 해당 기능의 **User Story 초안**을 즉석에서 보여주고 확인받는다:
```
확인: 이 이해가 맞나요?

US-001: [기능 이름]
As a [사용자], I want to [행위], so that [목적].
- Given [전제 조건]
- When [행동]
- Then [기대 결과]
- Edge case: [예외 상황] → [처리 방식]

맞다면 다음 기능으로 넘어가겠습니다. 수정할 부분이 있으면 알려주세요.
```

### Stage 4: 우선순위 & 범위 조정 (Prioritization & Scoping)
모든 기능이 나왔으면 함께 우선순위를 정한다.

```
PM:
"지금까지 나온 기능을 정리하면 이렇습니다:

[기능 목록 + 각 기능의 한줄 설명]

이 중에서:
- 반드시 있어야 하는 것 (P1 - MVP)은?
- 있으면 좋은 것 (P2)은?
- 나중에 추가해도 되는 것 (P3)은?

또한, 빠진 기능이 있을까요?"
```

### Stage 5: 제약 조건 & 품질 기준 (Constraints & Quality)
비기능 요구사항과 프로젝트 제약을 확인한다.

```
PM 질문 예시:
- "사용자 규모는 어느 정도를 예상하시나요? (수십 명? 수천 명? 수만 명?)"
- "성능 요구사항이 있나요? (예: 페이지 로딩 2초 이내)"
- "보안이 특히 중요한 부분이 있나요? (예: 결제, 개인정보)"
- "배포 환경에 대한 선호가 있나요? (클라우드, 온프레미스 등)"
- "일정이나 예산 제약이 있나요?"
```

---

## 완료 조건

- 모든 P1 기능에 대해 User Story + Acceptance Criteria가 확인됨
- 사용자가 "이 범위로 진행해도 됩니다"라고 명시적으로 승인
- 이후 1-B(Feature Specification)로 진행하여 정식 spec.md를 작성

## 인터뷰 원칙

- 한 번에 질문은 **최대 2개**. 사용자가 부담 없이 답변할 수 있어야 한다.
- 사용자의 답변을 **즉시 구조화하여 되돌려준다** (User Story, 여정 맵 등). 사용자가 자신의 말이 어떻게 해석되는지 바로 확인할 수 있어야 한다.
- 사용자가 모르겠다고 하면 **선택지를 제시**한다. "A 방식과 B 방식이 있는데, A는 ~이고 B는 ~입니다. 어느 쪽이 더 가까우세요?"
- **가정을 숨기지 않는다**. PM이 합리적인 가정을 했다면 명시하고 확인받는다. "~라고 가정했는데, 맞나요?"
- 인터뷰 전체를 통해 **누적 진행률**을 보여준다: "현재 6/10 기능의 요구사항이 구체화되었습니다."
